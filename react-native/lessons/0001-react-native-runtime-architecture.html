📱 Lesson 0001 — React Native 執行時架構心智模型

約 15 分鐘 · 難度：基礎 · 建立於 2026-06-12

為什麼這很重要

你已經會用 React 寫網頁。瀏覽器給你 DOM，React 操作 DOM。但手機上沒有 DOM。 React Native 的核心問題就是：JavaScript 如何驅動真正的 iOS/Android 原生元件？

如果你不理解這個問題的答案，當 bug 發生在 JS 和 native 的邊界時，你會完全無從下手。這堂課建立的 心智模型，是後續所有學習的地基。

🌐 先回想：React Web 的模型

┌─────────────────────────────────┐
│          Browser (單一 Process)   │
│                                 │
│  ┌──────────┐    ┌───────────┐  │
│  │ JS Engine │───▶│  DOM API  │  │
│  │ (V8)      │    │           │  │
│  └──────────┘    └───────────┘  │
│       │                │        │
│       │          ┌─────▼──────┐ │
│       │          │  Render    │ │
│       └──────────▶  Pipeline  │ │
│                  └────────────┘ │
└─────────────────────────────────┘

關鍵特點：JS 和 DOM 在同一個 process 裡。React 呼叫 document.createElement， 瀏覽器同步建立 DOM node。JS 可以直接讀取佈局資訊（getBoundingClientRect）， 一切都是同步的、在同一個 thread。

📲 React Native：三個世界

React Native 把這個模型拆成了三個獨立的執行環境：

┌──────────────────┐       ┌──────────────────┐
│  🧵 JS Thread    │       │  🧵 Native Thread │
│  (Hermes / JSC)  │       │  (Main / UI)      │
│                  │       │                  │
│  React           │       │  UIView          │
│  Components      │       │  ViewGroup       │
│  State / Props   │       │  Layout          │
│  Business Logic  │       │  Gestures        │
│                  │       │                  │
└────────┬─────────┘       └────────┬─────────┘
         │                          │
         │    ┌─────────────────┐   │
         └────│  🌉 JSI / Bridge  │───┘
              │  (C++ Layer)     │
              └─────────────────┘
                      │
              ┌───────▼────────┐
              │ 🧵 Shadow Tree │
              │ (Yoga Layout)  │
              └────────────────┘

核心差異：

  • JS Thread：跑你的 React 程式碼、狀態管理、API 呼叫邏輯。使用 Hermes（Meta 為 mobile 優化的 JS engine）而非 V8。
  • Native Main Thread（UI Thread）：跑真正的原生 UI — iOS 的 UIView、Android 的 ViewGroup。手勢、動畫、畫面繪製都在這裡。
  • JSI / Bridge（C++ Layer）：JS 和 Native 之間的翻譯層。這是 RN 最重要的基础设施。
  • Shadow Tree：在背景 thread 計算 layout（用 Yoga 排版引擎），結果再送到 UI thread。
💡 關鍵心智轉換：
Web: JS → DOM API → 畫面（全在同一 process）
RN: JS → JSI/C++橋樑 → 原生元件（跨 thread、跨語言）

🏗️ 舊架構 vs 新架構（New Architecture）

這是你在閱讀 RN 文章時最容易困惑的地方。從 2024 年 10 月（RN 0.76）起，新架構已經是預設。但很多教學仍描述舊架構。

舊架構：Async Bridge

JS Thread                  Bridge                  Native Thread
─────────                  ──────                  ─────────────
onPress()                    │                        │
  ──JSON serialize──▶       │    ──JSON parse──▶     │
  │                          │    (排隊、非同步)        │  createView()
  ◀──JSON serialize──       │    ◀──JSON parse──     │
  setState()                 │                        │

問題：

  • 所有通訊都要 JSON 序列化/反序列化 — 大量資料時很慢
  • 非同步 — JS 不能同步讀取 native 的 layout 資訊
  • 所有 native module 在啟動時就全部載入 — 拖慢啟動速度

新架構：JSI + Fabric + TurboModules

JS (Hermes) ◀──── C++ JSI Layer ────▶ Native (ObjC/Kotlin)
    │                                       │
    │  JavaScript 可以直接持有              │
    │  C++ HostObject 的 reference          │
    │                                       │
    └──────── 同步雙向呼叫 ─────────────────┘

四大支柱：

  1. JSI (JavaScript Interface)：輕量 C++ 層，取代舊 bridge。JS 可以直接呼叫 native 方法，同步、不需要 JSON 序列化。
  2. Fabric：新的渲染引擎。Shadow Tree 在 C++ 層計算，支援 React 18 Concurrent Mode。
  3. TurboModules：新的 native module 系統。lazy-load（用到才載入）、type-safe（透過 Codegen）。
  4. Codegen：從 TypeScript 規格自動產生 native binding，消滅手寫膠水碼。
🔑 對你來說：
用 Expo + 新架構開發，你大部分時候不需要手寫 native 程式碼。但當你遇到效能瓶頸或需要自訂原生功能時，理解 JSI 如何運作會讓你從「會用」升級到「懂原理」。

🔄 一個 Touch 事件的完整旅程（新架構）

1. 使用者點擊按鈕
      │
2. iOS/Android 捕捉觸控事件
      │
3. Fabric 找到對應元件，透過 JSI 傳給 Hermes
      │  (直接 C++ 呼叫，無序列化)
      │
4. React 處理事件 (你的 onPress handler)
      │  可能 setState → re-render
      │
5. Fabric diff Shadow Tree，Yoga 計算 layout
      │
6. Mounting layer 把變更套用到原生 view
      │
7. 使用者看到 UI 更新 ✅  (全部在 16ms 內完成，維持 60fps)

📦 Expo 在這個架構中的位置

Expo 不是「RN 之上的另一層抽象」，它是一套開發工具鏈：

  • Expo SDK：預先包裝好的 native module 集合（相機、定位、推播等），你不用自己寫 native 端
  • Expo Go：開發用 app，讓你不用 Xcode/Android Studio 就能在手機上預覽
  • EAS (Expo Application Services)：雲端建置、OTA 更新、商店上傳
  • expo prebuild：從 app.json 自動產生 ios/ 和 android/ 原生專案目錄

Expo app 跑的是同一個 RN runtime，使用同樣的 Hermes engine、同樣的 Fabric renderer。
