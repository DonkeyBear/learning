LESSON 0005 / BLOCKS / ~45 min

打造 3D 方塊與 Loot Table

將方塊放在世界上、拿在手裡，並自定義破壞後的掉落物。

方塊與物品有什麼不同？

在 Minecraft 中，物品 (Item) 只存在於你的物品欄或掉落在地上（成為實體）。而 方塊 (Block) 則固定在世界的 3D 座標系中，擁有方向、狀態（例如門是開著還是關著），並且在被破壞時會觸發掉落事件。

要建立一個完整的方塊體驗，我們需要做兩件事：
1. 註冊世界上的「方塊」。
2. 註冊放在背包裡的「方塊物品 (BlockItem)」。

💡 順帶一提：BlockState 的概念

真實的 Minecraft 方塊遠比這課介紹的複雜。每個方塊可以有多種「狀態 (BlockState)」——例如樓梯有朝向（North/South...）和形狀（Straight/Inner/Outer）的組合，產生出數十種不同的顯示方式。這門課我們從最簡單的「無狀態」方塊入門，等你熟悉之後，BlockState 是一個值得深入的下一步。

步驟一：雙重註冊 (Block & BlockItem)

建立 ModBlocks 類別

就像 ModItems 一樣，我們建立一個 ModBlocks.java 來統一管理方塊。這裡我們將遵循 Minecraft 26 的標準，同時使用 ResourceKey 註冊方塊與對應的物品。

package com.example.block;

import net.minecraft.world.item.Item;
import net.minecraft.world.item.BlockItem;
import net.minecraft.world.level.block.Block;
import net.minecraft.world.level.block.state.BlockBehaviour;
import net.minecraft.core.Registry;
import net.minecraft.core.registries.BuiltInRegistries;
import net.minecraft.core.registries.Registries;
import net.minecraft.resources.Identifier;
import net.minecraft.resources.ResourceKey;

public class ModBlocks {
    // 1. 建立 Block 與 Item 的註冊金鑰
    public static final ResourceKey<Block> MAGIC_BLOCK_KEY = ResourceKey.create(
            Registries.BLOCK, Identifier.parse("myfirstmod:magic_block")
    );
    public static final ResourceKey<Item> MAGIC_BLOCK_ITEM_KEY = ResourceKey.create(
            Registries.ITEM, Identifier.parse("myfirstmod:magic_block")
    );

    // 2. 實例化方塊 (綁定 Block 金鑰)
    public static final Block MAGIC_BLOCK = new Block(
            BlockBehaviour.Properties.of().setId(MAGIC_BLOCK_KEY).strength(3.0f)
    );

    // 3. 實例化方塊物品 (包裝方塊，並綁定 Item 金鑰)
    public static final Item MAGIC_BLOCK_ITEM = new BlockItem(
            MAGIC_BLOCK, 
            new Item.Properties().setId(MAGIC_BLOCK_ITEM_KEY)
    );

    // 4. 註冊方法
    public static void registerModBlocks() {
        // 分別註冊方塊和對應的物品
        Registry.register(BuiltInRegistries.BLOCK, MAGIC_BLOCK_KEY, MAGIC_BLOCK);
        Registry.register(BuiltInRegistries.ITEM, MAGIC_BLOCK_ITEM_KEY, MAGIC_BLOCK_ITEM);
    }
}
💡 順帶一提：為什麼需要 BlockItem？

Block 物件只負責處理世界上的互動（如踩踏、被挖掘）。當你把方塊挖下來變成掉落物，或者拿在手中時，遊戲需要一個繼承自 Item 的物件。BlockItem 就是官方提供的一個「包裝紙」，把方塊偽裝成物品讓你拿在手上！這也就是為什麼我們同時需要 Registries.BLOCK 和 Registries.ITEM 的金鑰。

步驟二：方塊的資源檔架構

方塊的渲染比單純的物品複雜，我們需要建立四個資源檔！

💡 順帶一提：assets/ vs. data/ 的分界

Minecraft 把資源分成兩個目錄，這和前端的 public/（靜態資源）與 api/（後端邏輯）的概念如出一轍。assets/ 是客戶端資源（貼圖、模型、音效），屬於「怎麼看起來像什麼」。data/ 是伺服器端邏輯（Loot Table、配方、標籤），屬於「遊戲規則是什麼」。區分清楚這兩個目錄，可以避免非常多「材質沒讀到」或「掉落物沒生效」的 debug 困境。

1. Blockstates (方塊狀態)

這決定了方塊在不同狀態下（例如朝向南邊或北邊）要使用哪個模型。對於普通的方塊，我們只需要一種狀態。
assets/myfirstmod/blockstates/magic_block.json

{
  "variants": {
    "": { "model": "myfirstmod:block/magic_block" }
  }
}

2. 幾何模型 (Block Model) 與貼圖 (Texture)

定義這個方塊是一個正方體 (cube)，並且六面都貼同一張圖。
assets/myfirstmod/models/block/magic_block.json

{
  "parent": "minecraft:block/cube_all",
  "textures": {
    "all": "myfirstmod:block/magic_block"
  }
}

準備一張圖片放在：assets/myfirstmod/textures/block/magic_block.png

3. 背包裡的物品顯示 (Item Model Definition)

還記得上一課學到的 Minecraft 26 新版架構嗎？因為我們有 BlockItem，所以也必須為它定義 items/*.json。
assets/myfirstmod/items/magic_block.json

{
  "model": {
    "type": "minecraft:model",
    "model": "myfirstmod:block/magic_block"
  }
}

注意：這裡的 model 指向的是 block/magic_block，代表拿在手上的方塊長得跟地上的一模一樣。

步驟三：破壞與掉落 (Loot Tables)

在舊時代，工程師會寫 if (broken) return magic_sword;。但現在的 Minecraft 是 Data-Driven (資料驅動) 的！掉落物完全由 JSON 控制。這個 JSON 必須放在 data/ 目錄，而不是 assets/（因為掉落屬於遊戲邏輯規則，不屬於客戶端視覺資源）。

建立檔案：src/main/resources/data/myfirstmod/loot_table/blocks/magic_block.json

{
  "type": "minecraft:block",
  "pools": [
    {
      "rolls": 1,
      "entries": [
        {
          "type": "minecraft:item",
          "name": "myfirstmod:magic_sword"
        }
      ],
      "conditions": [
        {
          "condition": "minecraft:survives_explosion"
        }
      ]
    }
  ]
}
💡 順帶一提：資料驅動 (Data-Driven) 的威力

透過 JSON 定義掉落物，意味著未來玩家可以透過「資料包 (Datapack)」輕鬆修改你的模組！他們不需要會寫 Java，只要放一個新的 magic_block.json 覆蓋原本的路徑，就能讓這個方塊改掉鑽石或蘋果。這是現代 Minecraft 模組生態能如此繁榮的關鍵設計。

💡 順帶一提：Loot Table 與 SpringBoot 設定驅動的共鳴

在 SpringBoot 中，我們習慣把行為配置從程式碼中抽出來，放進 application.yml。Loot Table 的設計哲學完全一樣——把「規則」外化成資料檔，讓邏輯與設定分離。未來學 SpringBoot 時，你會發現這種「資料驅動設定」的思維無處不在。
