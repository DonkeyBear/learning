🚪 API 網關 — 當 Nginx 不夠用時

Lesson 0004 · ⏱ ~20 min · architecture · ← Lesson 0003

從 1 個服務到 20 個服務

Lesson 0003 你學了 Nginx 反向代理，它的設定長這樣： 

location / { proxy_pass http://fastapi_backend; }

一個後端、一個 upstream、一條規則。這在服務只有 1 個時完美運作。
但當你的系統長成這樣時，Nginx 開始吃力了： 

                    ┌─────────────┐
                    │  用戶服務     │ :8001
                    │  (FastAPI)   │
                    └─────────────┘
                    ┌─────────────┐
  客戶端 ──→ Nginx ──│  訂單服務     │ :8002
                    │  (FastAPI)   │
                    └─────────────┘
                    ┌─────────────┐
                    │  支付服務     │ :8003
                    │  (FastAPI)   │
                    └─────────────┘

   Nginx config 會變成一堆 location 規則
   每新增一個服務就要手動改 config + reload
   認證邏輯散落在每個服務中
   沒有統一的監控、限流、日誌

API Gateway（API 網關）就是為了解決這個「多服務管理」問題而生的。

🎯 本課目標：能清楚區分反向代理和 API 網關的職責邊界， 知道何時該用哪個，並能在面試中畫出一個微服務架構下的 API Gateway 部署圖。

第一步：Nginx vs API Gateway — 不是替代，是進化

一句話區分

Nginx 是 Layer 4/7 的反向代理——它關心的是「怎麼把請求轉發到正確的機器」。

API Gateway 是應用層的閘道——它關心的是「這個請求應該被轉發嗎？轉發前需要做什麼？」 

┌── Nginx 做的事（反向代理層）─┐   ┌── API Gateway 額外做的事 ──┐
│                             │   │                            │
│  ✅ SSL termination         │   │  ✅ 路由：/users → 用戶服務   │
│  ✅ 靜態檔案                │   │  ✅ 鑑權：JWT 驗證集中處理    │
│  ✅ 負載均衡到 upstream     │   │  ✅ 限流：每用戶每秒 100 次   │
│  ✅ 基本 header 改寫        │   │  ✅ API 版本：/v1, /v2 路由   │
│  ✅ 連線緩衝與超時          │   │  ✅ 請求/回應轉換            │
│                             │   │  ✅ 服務發現（自動偵測新服務） │
│                             │   │  ✅ 金鑰管理 + 計費          │
│                             │   │  ✅ 統一監控與日誌           │
│                             │   │  ✅ Circuit Breaker         │
└─────────────────────────────┘   └────────────────────────────┘
💡 實務上常見的組合：Nginx 在最前面處理 SSL 和靜態檔案，然後把 API 請求轉發給 API Gateway（Kong / Traefik / APISIX），Gateway 再做路由、鑑權、限流，最後轉給各個微服務。兩者協同工作，不是二選一。

第二步：API Gateway 的核心能力

① 路由與服務發現

不再是寫死 proxy_pass http://127.0.0.1:8001，而是動態路由：

# Kong 的路由規則（宣告式，不用手寫 location）
GET    /users          → user-service      (內部自動負載均衡)
GET    /users/:id      → user-service
POST   /orders         → order-service      ← 需要 auth
DELETE /orders/:id     → order-service      ← 需要 auth + admin role
POST   /payments       → payment-service    ← 需要特殊的 API key

當一個新的服務上線，Gateway 透過服務發現（Consul、K8s Service、Docker DNS）自動知道它的位置，不需要手動改 config。

② 集中式鑑權

在 Nginx 時代，每個 FastAPI 服務都要自己寫 JWT 驗證。Gateway 把這件事抽出來：

  Request: GET /orders/42
  Authorization: Bearer <JWT>
         │
         ▼
  ┌─────────────────────┐
  │  API Gateway        │
  │  ① 驗證 JWT 簽名    │  ← 一次處理，所有服務受益
  │  ② 解析 user_id     │
  │  ③ 檢查 rate limit  │
  │  ④ 附加 X-User-ID   │
  │     header 轉發      │
  └─────────────────────┘
         │
         ▼
  ┌─────────────┐
  │ order-service│ ← 直接讀 X-User-ID，不需要再驗 JWT
  └─────────────┘

③ Rate Limiting（限流）

Nginx 的 limit_req 只能做到「每個 IP 每分鐘 N 次」。API Gateway 可以做到：

# 精細的限流策略
plan: free      → 每分鐘 10 次請求
plan: pro       → 每分鐘 1000 次請求
plan: enterprise → 每分鐘 10000 次請求

# 或是按 endpoint 限流
POST /payments   → 每秒最多 5 次（防重複扣款）
GET  /search     → 每秒最多 50 次（防爬蟲）

④ 請求/回應轉換

舊版 API 回傳 user_name，新版前端要 username？Gateway 可以即時改寫，不用動後端程式碼。

⑤ 還有這些能力——面試加分項

Circuit Breaker 當訂單服務連續失敗 5 次，自動熔斷，直接回 503，防止級聯故障
API Versioning /v1/users → 舊服務、/v2/users → 新服務，平滑遷移
Request Deduplication 偵測重複請求（如使用者連點兩下付款），只轉發一次
Response Caching Gateway 層快取 GET 回應，減輕後端負載
Canary Release 10% 流量導向新版服務，觀察錯誤率後逐步擴大

第三步：常見的 API Gateway 方案

開源方案（你可以自己部署的）

Gateway 特點 適合場景
Kong 基於 Nginx + Lua，外掛生態豐富（JWT, OAuth2, rate limiting, logging） 中大型微服務，需要豐富的外掛生態
Traefik Go 語言，原生支援 Docker/K8s，自動服務發現 容器化環境，追求零配置自動發現
Apache APISIX 基於 Nginx + LuaJIT，高效能，支援動態設定 高效能要求，需要動態變更路由
Caddy 自動 HTTPS，設定極簡（但 API Gateway 功能較少） 小型專案，簡單的反向代理 + SSL

雲端方案（付費但免維護）

  • AWS API Gateway — 與 Lambda、Cognito 深度整合
  • GCP Apigee — Google 的企業級方案
  • Azure API Management — 與 Azure AD 整合
⚠️ 不要為了用而用：如果你的服務只有 2-3 個，一個 Nginx 就夠了。 API Gateway 引入額外的複雜度（部署、監控、延遲 +5-20ms）。 先有痛點（管理成本太高、鑑權邏輯重複、缺乏統一監控），再引入 Gateway。

第四步：回顧地圖 — Gateway 坐哪裡？

回憶 Lesson 0001 的 request 路徑圖。API Gateway 坐哪裡？

  瀏覽器 → DNS → TCP → TLS
    ↓
  ⑦a Nginx (SSL termination, 靜態檔案)
    ↓
  ⑦b API Gateway ← 🆕 本課的主題
    │   (路由、鑑權、限流、版本、轉換)
    ↓
  ⑧ uvicorn → ⑨ FastAPI → ⑩ Middleware → ⑪ 路由函數 → ⑫ 回應返回

Gateway 本質上是 ⑦（反向代理層）的進化版。它依然是 request 路徑上的一個中間站， 但它比 Nginx 多了一層「應用層智慧」——知道你的 API 結構、知道誰在呼叫、知道該不該放行。

📋 本課重點摘要

  1. Nginx 是傳輸層反向代理（轉發請求），API Gateway 是應用層閘道（理解 API 語義後做決策）。
  2. Gateway 的核心能力：路由、集中鑑權、限流、版本管理、請求轉換、熔斷、金絲雀部署。
  3. Gateway 和 Nginx 是協同關係，不是替代關係——常見架構是 Nginx（SSL + 靜態檔案）→ Gateway（路由 + 鑑權）→ 後端服務。
  4. 開源選擇：Kong（生態豐富）、Traefik（容器原生）、APISIX（高效能）。
  5. 不要過早引入——服務少於 5 個時 Nginx 就夠了。先有痛點再升級。

🎤 面試回答模板：「你們的微服務之間怎麼溝通？有沒有用 API Gateway？」

「我們使用 API Gateway 作為所有微服務的統一入口。Gateway 負責幾件事： 第一，路由——根據 URL path 把請求導向對應的服務； 第二，集中鑑權——在 Gateway 層驗證 JWT，通過後把 user_id 放進 header 傳給後端，避免每個服務重複實作； 第三，限流——根據使用者的方案（free/pro/enterprise）做差異化 rate limiting； 第四，觀測性——Gateway 層統一收集 request log、metrics 和 tracing。 我們選用 Kong，因為它的外掛生態成熟，JWT、OAuth2、rate limiting 都有現成的外掛可以用。 在 Gateway 前面還有一層 Nginx 處理 SSL termination 和靜態檔案。」
