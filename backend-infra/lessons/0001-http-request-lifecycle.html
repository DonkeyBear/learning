📡 HTTP Request 的完整生命週期

Lesson 0001 · ⏱ ~15 min · foundation

為什麼先學這個？

你已經能用 FastAPI 寫出 @app.get("/items")，但你有沒有想過：從使用者打開瀏覽器，到你的 Python 程式碼被執行，中間到底發生了什麼事？

CORS、反向代理、API 網關、鑑權——這些你之後要學的所有東西，都座落在這個 request 路徑的某個位置上。如果不先理解這條路徑的全貌，後面的每個主題都會像一座孤島。先有地圖，再認路。

🎯 本課目標：學完這堂課後，你能從頭到尾畫出一張 HTTP request 的旅程圖，並正確說出每一站的名稱和職責。

旅程開始：一個 Request 的一生

假設使用者在瀏覽器輸入 https://api.example.com/users/42，以下是完整旅程： 

┌─────────────────────────────────────────────────────────────────┐
│  ① 瀏覽器                 ② DNS 解析              ③ TCP 連線      │
│  輸入 URL ──────────→ 取得伺服器 IP ──────────→ 三次握手建立     │
│                          93.184.216.34             TCP connection  │
└─────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐
│  ④ TLS 握手               ⑤ HTTP Request          ⑥ 網路傳輸      │
│  HTTPS 加密協商 ──────→ 發送 GET /users/42 ──→ 封包經過路由器    │
│  (證書驗證 + 密鑰交換)     + Headers + Body        到達 VM 的網路卡 │
└─────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐
│  ⑦ 反向代理 (Nginx)       ⑧ ASGI Server          ⑨ FastAPI App   │
│  接收請求、做初步處理 ──→ uvicorn 轉發 ────────→ 你的 Python 程式  │
│  (SSL termination、       (ASGI protocol)         路由 → 函數執行   │
│   static files、限流)                                       │
└─────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐
│  ⑩ 中介層 (Middleware)    ⑪ 路由處理             ⑫ 回應返回       │
│  CORS check ──────────→ /users/42 對應的 ──→ JSON Response       │
│  Auth check              路徑函數被呼叫          沿原路返回瀏覽器    │
│  Request ID                                                      │
└─────────────────────────────────────────────────────────────────┘

上面這張圖就是後端工程師的「世界地圖」。我們來逐步解析每個站點。

階段一：建立連線（①–④）

①
瀏覽器解析 URL 瀏覽器從 https://api.example.com/users/42 中拆出：通訊協定 (HTTPS)、主機名稱 (api.example.com)、路徑 (/users/42)。
②
DNS 查詢 瀏覽器問 DNS 伺服器：「api.example.com 的 IP 是多少？」→ 得到 93.184.216.34。這是網際網路的電話簿。
③
TCP 三次握手 瀏覽器與伺服器建立可靠的 TCP 連線。SYN → SYN-ACK → ACK。這發生在 OSI 第 4 層（傳輸層），與 HTTP 無關——它只負責「建立一條可靠的通道」。
④
TLS 握手 因為是 HTTPS，TCP 連線建立後還需要 TLS 加密協商。伺服器出示 SSL 憑證、雙方協商加密演算法、交換對稱密鑰。完成後，之後的所有 HTTP 資料都是加密的。
💡 關鍵理解：TCP（③）和 TLS（④）是兩層獨立的握手。TCP 先建立「能通話的線路」，TLS 再把這條線路「加密」。你的 FastAPI 程式碼完全不知道這些事——它們在抵達你的程式之前就處理完了。

階段二：請求到達伺服器（⑤–⑦）

⑤
瀏覽器發送 HTTP Request 一個標準的 HTTP request 長這樣：
GET /users/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGci...
Accept: application/json
⑥
網路路由 封包經過多個路由器在網際網路上跳轉，最終到達你 VM 的網路卡。你的雲端供應商（GCP/AWS/Azure）的防火牆可能在此處做第一層過濾。
⑦
反向代理（Nginx / Caddy） 這是你伺服器上的「門房」。它接收 request，可以做：
• SSL Termination：幫你解密 HTTPS，讓後面的 FastAPI 只需要處理 HTTP
• 靜態檔案：直接回傳圖片/CSS/JS，不勞煩 Python
• 負載均衡：把 request 分配給多個 uvicorn process
• 限流 / IP 黑名單：在請求碰到你的程式之前就擋掉惡意流量
→ 然後把 request 轉發給 uvicorn（通常在 127.0.0.1:8000）
⚠️ 常見誤解：很多人以為 uvicorn main:app --host 0.0.0.0 --port 80 就是「部署」。在生產環境中，你不應該讓 uvicorn 直接暴露在 80/443 port。前面一定要放 Nginx 或 Caddy。原因在後面的課程會詳細討論。

階段三：伺服器內部處理（⑧–⑪）

⑧
ASGI Server (uvicorn) uvicorn 接收 Nginx 轉發來的 HTTP request，依照 ASGI 協議 將其轉換成 Python 能理解的資料結構（scope、receive、send），然後傳給 FastAPI app。
⑨
FastAPI 框架 FastAPI 接管 ASGI 傳來的 request，建立 Request 物件。它解析 URL、headers、body，然後開始走內部管線。
⑩
中介層（Middleware） 在抵達你的路由函數之前，request 會先經過一層層 middleware。每一層可以：檢查、修改、或短路（直接回傳 response）。例如：
• CORS Middleware：檢查 Origin header，決定是否允許跨域
• Auth Middleware：驗證 JWT token，附加 user 資訊到 request
• Request ID Middleware：為每個 request 加上唯一 ID 以便追蹤日誌
⑪
路由匹配與函數執行 FastAPI 根據 URL 路徑 /users/42 和 HTTP method GET 找到對應的函數。你的 Python 程式碼終於被執行了。
🔑 關鍵洞察：你的程式碼（⑪）只佔整個旅程的 1/12。前面 10 步和最後 1 步（回應返回）都不是你寫的，但它們決定了你的服務是否安全、快速、可靠。這就是為什麼你需要學這些「底層知識」。

階段四：回應返回（⑫）

你的函數 return 一個 JSON 之後，response 沿著完全相同的路徑反向回去：middleware → ASGI → Nginx → 網路 → 瀏覽器。每一層都有機會再處理 response（例如加上 security headers）。

這張地圖如何連接後續課程

🗺 你在這張地圖上學到的，之後會用來理解：

  • CORS → 發生在 ⑩ Middleware 層，檢查 request 的 Origin header
  • 反向代理 → 就是 ⑦ Nginx，理解它為什麼是必備的基礎設施
  • API 網關 → ⑦ 的進化版，多了路由、鑑權、rate limiting、API versioning
  • 鑑權 (Auth) → 可在 ⑦（網關層）或 ⑩（middleware 層）執行
  • 資安 → 每一層都有對應的安全威脅與防禦策略

📋 本課重點摘要

  1. 一個 HTTP request 的旅程有 12 個主要階段，你的 FastAPI 程式碼只佔其中 1 個。
  2. 反向代理（Nginx）是你的伺服器門房，處理 SSL、靜態檔案和流量控制。
  3. Middleware 是 request 進入路由前的管線，CORS、Auth 檢查都發生在這裡。
