🗄 資料庫設計 — 索引、正規化、N+1、連線池

Lesson 0007 · ⏱ ~25 min · database · ← Lesson 0006

你用 Firebase 存資料，然後呢？

你會用 Firebase CRUD，這很棒。但面試官會問的問題通常是這些： 「這個查詢為什麼慢？」「你有沒有用索引？」「N+1 問題怎麼解？」 本課用關聯式資料庫（PostgreSQL/MySQL）的視角補足你的知識盲區， 並指出哪些概念 Firestore 已經幫你處理了、哪些還沒有。

🎯 本課目標：能解釋索引原理、避開 N+1 問題、設定連線池， 並理解關聯式 vs NoSQL 的選擇邏輯。

第一步：關聯式 vs NoSQL — 你已經在用 NoSQL 了

Firebase Firestore 幫你處理了什麼？

Firestore 是文檔型 NoSQL。它自動幫你做了：

  • ✅ 水平擴展：資料自動分片到多台機器
  • ✅ 索引：每個欄位都自動建了單欄索引
  • ✅ 連線管理：SDK 內部處理了連線池
  • ✅ 即時更新：WebSocket 監聽（關聯式 DB 做不到）

但也意味著你沒學到這些——因為它們被藏起來了。面試中這些都是考點。

📊 關聯式資料庫 (PostgreSQL / MySQL)

  • 📦 資料結構：Table (表格)
  • 🔗 關聯方式：JOIN + Foreign Key (外鍵)
  • 💬 查詢語言：標準 SQL 語法
  • ⚡ 擴展方式：垂直擴展 (增加單機 RAM/CPU)
  • 📋 Schema：嚴格限制 (需要 Migration)
  • 🔐 ACID 事務：完整嚴格支援
  • 🎯 適合場景：資料間關係複雜、需要複雜 JOIN 查詢與資料高度一致的系統（如金融、交易系統）。

⚡ NoSQL 資料庫 (Firestore / MongoDB)

  • 📦 資料結構：Collection / Document (集合文件)
  • 🔗 關聯方式：巢狀文件嵌入 (Denormalize) 或手動查詢
  • 💬 查詢語言：各資料庫專屬的 SDK / API
  • ⚡ 擴展方式：水平擴展 (增加多台機器做 Sharding)
  • 📋 Schema：動態彈性 (Schemaless)
  • 🔐 ACID 事務：有限支援（依不同 NoSQL 特性而異）
  • 🎯 適合場景：樹狀或大文檔型資料結構、需要 WebSocket 即時同步、高併發讀寫、或是快速迭代開發 MVP。
💡 面試金句：「我們用 Firestore 是因為初期需要快速開發和即時同步功能。 但隨著業務成長，某些需要複雜查詢和強一致性的模組（如訂單系統）， 我們正在評估遷移到 PostgreSQL。」
→ 這表示你理解兩種 DB 的取捨，不是只會用 Firebase。

第二步：索引 — 為什麼查詢突然變慢？

沒有索引時：全表掃描

-- 100 萬筆資料的 users 表
SELECT * FROM users WHERE email = 'alice@example.com';
-- PostgreSQL 要從頭到尾掃描 100 萬行 → 300ms

-- 加上索引後：
CREATE INDEX idx_users_email ON users(email);
-- PostgreSQL 用 B-tree 直接跳到目標 → 0.1ms

常見索引類型

索引類型 用途 範例
B-tree（預設） 等於、大於小於、範圍查詢、排序 WHERE email = ?
複合索引 多個欄位一起查詢 WHERE user_id=1 AND status='active'
Unique 索引 保證欄位值唯一（email、username） UNIQUE(email)
全文索引 模糊搜尋、關鍵字搜尋 WHERE content LIKE '%keyword%'
Partial 索引 只對符合條件的 row 建索引 WHERE deleted_at IS NULL
⚠️ 索引不是免費的：每個索引都會拖慢寫入（INSERT/UPDATE/DELETE）速度， 並佔用磁碟空間。不要「每欄都加」，而是對查詢中 WHERE、JOIN、ORDER BY 的欄位加索引。

第三步：N+1 問題 — 後端面試必考

這可能是 ORM 使用者最容易踩的效能陷阱。

# 🚨 N+1 問題：1 次查詢拿到 100 個 user，
#        然後對每個 user 再發 1 次查詢拿 orders
#        總共 1 + 100 = 101 次查詢！

users = db.query(User).all()                # 1 次查詢，拿到 100 個 user
for user in users:
    orders = db.query(Order).filter(        # ← 對「每個」user 再查一次！
        Order.user_id == user.id
    ).all()
    print(f"{user.name} has {len(orders)} orders")
# 總查詢次數：101（100 users + 1）

# ✅ 解法一：Eager Loading（預先載入）
from sqlalchemy.orm import joinedload

users = db.query(User).options(
    joinedload(User.orders)   # ← 一次 JOIN 把所有 orders 也拿回來
).all()
# 總查詢次數：1

# ✅ 解法二：批次查詢（如果不能 JOIN）
user_ids = [u.id for u in users]
all_orders = db.query(Order).filter(
    Order.user_id.in_(user_ids)   # ← 一次查詢拿所有 orders
).all()

# 在 Python 中分組
orders_by_user = {}
for order in all_orders:
    orders_by_user.setdefault(order.user_id, []).append(order)
# 總查詢次數：2
💡 Firestore 使用者的注意事項：Firestore 的資料是巢狀文件（document → subcollection）， 讀取 subcollection 是獨立的查詢。如果你遍歷 100 個 user document， 然後對每個 user 去讀 subcollection，這就是 Firestore 版的 N+1！ 解法也是一樣：用 collectionGroup 一次查詢，或重新設計資料結構來避免。

第四步：連線池（Connection Pool）

資料庫連線很貴——建立 TCP、TLS、認證。連線池就是預先建立一批連線，重複使用。

❌ 沒有連線池 (每次重新建立)

  • 📬 Request 1:
    • 建立 TCP/TLS 連線 ➔ 50ms
    • 執行 SQL 查詢 ➔ 5ms
    • 關閉資料庫連線 ➔ 10ms
    ⏱️ 總耗時：65ms
  • 📬 Request 2:
    • 建立 TCP/TLS 連線 ➔ 50ms
    • 執行 SQL 查詢 ➔ 5ms
    • 關閉資料庫連線 ➔ 10ms
    ⏱️ 總耗時：65ms
  • 📊 1,000 個請求總耗時：
    ⌛ 累計花費約 65 秒 (建立連線成為系統最大瓶頸)

✅ 有連線池 (重複使用連線)

  • 📬 Request 1:
    • 從池中直接租借連線 ➔ 0ms
    • 執行 SQL 查詢 ➔ 5ms
    • 歸還連線回池中 ➔ 0ms
    ⏱️ 總耗時：5ms
  • 📬 Request 2:
    • 從池中直接租借連線 ➔ 0ms
    • 執行 SQL 查詢 ➔ 5ms
    • 歸還連線回池中 ➔ 0ms
    ⏱️ 總耗時：5ms
  • 📊 1,000 個請求總耗時：
    🚀 累計花費僅 5 秒 (效能提升達 13 倍！) 
# FastAPI + SQLAlchemy 連線池設定
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

DATABASE_URL = "postgresql://user:pass@localhost/dbname"

engine = create_engine(
    DATABASE_URL,
    pool_size=20,            # 池中常駐 20 條連線
    max_overflow=10,         # 池滿時最多再開 10 條臨時連線
    pool_timeout=30,         # 等連線等 30 秒還拿不到就報錯
    pool_recycle=3600,       # 每小時回收連線（防止連線過期）
    pool_pre_ping=True,      # 拿連線前先 ping 看還活著嗎
)

SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

# 在 FastAPI 中使用（dependency）
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()  # 歸還連線到池中，不是真的關掉 TCP

第五步：交易（Transaction）與 ACID

這在 Firestore 中稱為 batch write / transaction，在 SQL 中更強大。

# 轉帳：從 A 扣 100、加到 B
# 🚨 沒有交易：如果第二步失敗，錢就消失了！

# ✅ 用交易確保原子性
with engine.begin() as conn:        # begin() = 開始交易
    conn.execute("UPDATE accounts SET balance = balance - 100 WHERE id = 1")
    conn.execute("UPDATE accounts SET balance = balance + 100 WHERE id = 2")
    # ↑ 任何一步失敗 → 自動 rollback，全部復原
    # ↑ 全部成功 → commit，同時生效

ACID 一分鐘速記

Atomicity 全部成功或全部失敗 轉帳不會扣了 A 沒加 B
Consistency 交易前後資料都符合規則 balance 永遠 ≥ 0（check constraint）
Isolation 併發交易不互相干擾 兩人同時轉帳不會出錯
Durability Commit 後的資料不會消失 停電重開後資料還在

📋 本課重點摘要

  1. 關聯式 vs NoSQL：Firestore 適合快速開發和即時同步；PostgreSQL 適合複雜查詢和強一致性。
  2. 索引：B-tree 把查詢從 O(n) 降到 O(log n)。但每次寫入都要更新索引，不要每欄都加。
  3. N+1 問題：1 次查主表 + N 次查關聯 = N+1 次。用 joinedload（eager loading）或批次查詢解決。
  4. 連線池：重複使用 DB 連線，避免每次都重新建立 TCP/TLS。SQLAlchemy 內建支援。
  5. ACID：交易保證原子性（全部都做或全部不做）——轉帳不會掉錢。

🎤 面試回答模板：「你的 API 發現某個端點很慢，怎麼排查？」

「我會從幾個方向排查：第一，看是否有 N+1 問題——用 SQLAlchemy 的 echo=True 或資料庫的 slow query log 看查詢數量和時間。如果是 N+1，改用 eager loading。 第二，檢查是否缺索引——用 EXPLAIN ANALYZE 看查詢計畫， 如果出現 Seq Scan（全表掃描）就表示需要索引。第三，檢查連線池設定—— 連線池太小會導致請求排隊等連線；太大會浪費資料庫資源。 第四，考慮加快取（Redis）來減少重複查詢。」
