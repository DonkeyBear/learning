🚧 CORS 深入解析

Lesson 0002 · ⏱ ~20 min · security · ← Lesson 0001

你一定看過這個錯誤

Access to fetch at 'https://api.yourdomain.com/users'
from origin 'https://frontend.yourdomain.com' has been blocked
by CORS policy: No 'Access-Control-Allow-Origin' header
is present on the requested resource.

這大概是後端開發者最常遇到的「它不是 bug，它是設計」的時刻。 你寫的 API 明明用 curl 或 Postman 測試都正常，但前端一呼叫就爆掉。 為什麼？

🎯 本課目標：學完後，你能解釋 CORS 為什麼存在、preflight 是什麼、 以及如何正確設定 FastAPI 的 CORSMiddleware——並在面試中給出有深度的回答。

第一步：Same-Origin Policy（同源政策）

要理解 CORS，必須先理解它要放鬆的那個規則：Same-Origin Policy (SOP)。

🔐 Same-Origin Policy — 瀏覽器的預設安全規則

瀏覽器預設禁止一個網頁的 JavaScript 讀取另一個「來源」(origin) 的資源。 這是瀏覽器內建的安全機制，防止惡意網站偷取你在其他網站上的資料。

什麼是「同一個 origin」？三個要素必須完全一樣：

  • Scheme（協議）：https://
  • Host（主機）：api.example.com
  • Port（埠號）：:443（https 預設）

Origin 比對練習

以下哪個 URL 和 https://api.example.com:443/users 是同一個 origin？

你的網頁：      https://frontend.example.com
你想呼叫的 API： https://api.example.com

Scheme 相同 ✅  (都是 https)
Host 不同   ❌  (frontend.example.com ≠ api.example.com)
Port 相同   ✅  (都是 443)

→ 不同源！瀏覽器會阻擋這個請求。
💡 關鍵理解：SOP 只存在於瀏覽器中。curl、Postman、伺服器之間的 HTTP 呼叫完全不受 SOP 限制。 這就是為什麼你的 API 在 Postman 中正常，但在瀏覽器中壞掉——SOP 是瀏覽器在執行，不是你的伺服器在執行。

第二步：CORS 登場

但現代 Web 架構中，前端（frontend.example.com）和後端（api.example.com） 分離部署是非常普遍的。CORS（Cross-Origin Resource Sharing，跨來源資源共享）就是一組規則， 讓伺服器可以「告訴瀏覽器」：「雖然我們不同源，但這個請求是安全的，你可以放行。」

CORS 的兩種請求模式

模式一：簡單請求（Simple Request）

同時滿足以下全部條件時，瀏覽器會直接發送請求，不先問伺服器：

  • Method 是 GET、HEAD、POST 三者之一
  • 只使用「簡單 headers」：Accept、Accept-Language、Content-Language、Content-Type
  • Content-Type 只能是 text/plain、multipart/form-data、application/x-www-form-urlencoded
  • 沒有 ReadableStream、沒有 XMLHttpRequest.upload 事件監聽器

→ 瀏覽器直接發 request，然後檢查 response 的 CORS header 來決定要不要把結果給前端 JavaScript。

模式二：預檢請求（Preflight Request）

不符合簡單請求的條件時（例如用了 PUT、DELETE、Authorization header、Content-Type: application/json），瀏覽器會先發一個 OPTIONS 請求去問伺服器：「我可以發這個請求嗎？」

這就像進去一間大樓之前先問櫃檯：「我可以進去嗎？我可以帶這個包包嗎？我可以去 3 樓嗎？」——櫃檯說 OK 之後你才真的上樓。

┌─ Preflight 流程 ─────────────────────────────────────┐
│                                                       │
│  瀏覽器                      API 伺服器                 │
│    │                            │                     │
│    │  OPTIONS /users/42         │                     │
│    │  Origin: frontend.app      │                     │
│    │  Access-Control-Req-Method: DELETE               │
│    │  Access-Control-Req-Headers: Authorization       │
│    │ ─────────────────────────→ │                     │
│    │                            │ 伺服器檢查：          │
│    │                            │ 「我允許 DELETE 嗎？」  │
│    │                            │ 「我允許 Auth 嗎？」   │
│    │                            │ 「我允許這個 origin?」 │
│    │                            │                     │
│    │  200 OK                    │                     │
│    │  Access-Control-Allow-Origin: frontend.app       │
│    │  Access-Control-Allow-Methods: GET, POST, DELETE │
│    │  Access-Control-Allow-Headers: Authorization     │
│    │ ←───────────────────────── │                     │
│    │                            │                     │
│    │  ✅ 預檢通過！              │                     │
│    │                            │                     │
│    │  DELETE /users/42          │                     │
│    │  (帶上真正的 request)       │                     │
│    │ ─────────────────────────→ │                     │
│    │                            │                     │
│    │  200 OK (response)         │                     │
│    │ ←───────────────────────── │                     │
└───────────────────────────────────────────────────────┘

第三步：關鍵 CORS Headers

Response Headers（伺服器回應給瀏覽器的）

Header 作用
Access-Control-Allow-Origin 指定哪個 origin 可以存取。可以是 *（全部）或特定 origin。
Access-Control-Allow-Methods 允許的 HTTP methods（只用於 preflight response）。
Access-Control-Allow-Headers 允許的非簡單 request headers（只用於 preflight）。
Access-Control-Allow-Credentials 設為 true 時，允許攜帶 cookie / Authorization header。此時 Allow-Origin 不能是 *。
Access-Control-Max-Age Preflight 結果可以快取多久（秒），避免每次都發 OPTIONS。
⚠️ 面試常見陷阱：Access-Control-Allow-Origin: * 和 credentials: true 不能同時使用。如果前端需要帶 cookie 或 Authorization header，你必須指定明確的 origin（不能用萬用字元）。

第四步：FastAPI 中的 CORS 設定

這是你可能已經在你的專案中寫過，但不一定完全理解每一行在做什麼的程式碼：

from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

# CORS 中介層 — 必須在其他 middleware 之前加入
app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "https://frontend.example.com",  # ✅ 明確列出
        "http://localhost:3000",          # ✅ 本地開發
    ],
    allow_credentials=True,   # 允許攜帶 cookie / auth header
    allow_methods=["GET", "POST", "PUT", "DELETE", "OPTIONS"],
    allow_headers=["*"],      # 允許所有 request headers
    expose_headers=["X-Request-ID"],  # 讓前端 JS 能讀取這些 response headers
    max_age=600,              # preflight 快取 10 分鐘
)
🔑 回憶 Lesson 0001 的地圖：CORS 檢查發生在 ⑩ Middleware 層。 當一個跨域 request 進來時，FastAPI 的 CORSMiddleware 會在 request 到達你的路由函數之前就攔截 OPTIONS（preflight）請求，並自動加上正確的 CORS response headers。 對於一般請求，它會在 response 返回時加上 CORS headers。

🛠 實作練習：在你的專案中驗證 CORS

任務：觀察 Preflight

  1. 打開你的 FastAPI 專案，確認 CORSMiddleware 已設定
  2. 打開瀏覽器 DevTools → Network 分頁
  3. 從前端（另一個 origin）對 API 發送一個帶 Authorization header 的 PUT 請求
  4. 在 Network 中觀察：你會看到兩個請求——一個 OPTIONS（preflight）和一個 PUT
  5. 點進 OPTIONS 請求，查看 response headers 中的 Access-Control-Allow-* 系列

📋 本課重點摘要

  1. Same-Origin Policy 是瀏覽器的安全規則，限制跨域讀取；CORS 是放鬆這個規則的機制。
  2. Origin = scheme + host + port，三者缺一不可。
  3. 簡單請求（GET/POST + 基本 headers）直接發送；非簡單請求（PUT/DELETE/JSON/自訂 header）需要先發 OPTIONS preflight。
  4. CORS 錯誤是瀏覽器在阻擋，不是伺服器。curl/Postman 不受影響。
  5. Access-Control-Allow-Origin: * 和 credentials: true 不能共存——這是面試經典題。

🎤 面試回答模板：「請解釋 CORS」

「CORS 是瀏覽器的安全機制。瀏覽器預設執行 Same-Origin Policy，禁止網頁的 JavaScript 跨域讀取資源。 CORS 讓伺服器可以透過特定的 HTTP response headers（如 Access-Control-Allow-Origin） 告訴瀏覽器哪些跨域請求是安全的。對於非簡單請求（如 PUT、DELETE、帶 Authorization header 的請求）， 瀏覽器會先發一個 OPTIONS preflight 請求來確認伺服器的 CORS 政策。 這整個機制只存在於瀏覽器中——伺服器之間的 HTTP 呼叫不受 CORS 限制。」
