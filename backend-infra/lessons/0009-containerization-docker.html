🐳 容器化 — Docker 與 Docker Compose

Lesson 0009 · ⏱ ~20 min · devops · ← Lesson 0008

「在我電腦上可以跑啊」— 這句話終結者

你用 uvicorn 部署到 VM 上。但如果 VM 重灌了？如果換了一台機器？ 如果團隊新成員要 setup 環境？每次都要重裝 Python、pip install、設定 Nginx、安裝 Redis...

Docker 把「整台機器的狀態」打包成一個檔案。 你在任何地方執行這個檔案，都會得到完全相同的環境——Python 3.12、相同的相依套件、相同的設定檔。

🎯 本課目標：能寫出 FastAPI 的 Dockerfile 和 docker-compose.yml， 把 Nginx + FastAPI + Redis + PostgreSQL 全部用一個指令啟動。

第一步：Docker 不是虛擬機

這是面試常問的概念題。

💻 虛擬機器 (Virtual Machine)

App A
Bin/Lib
App B
Bin/Lib
Guest OS
Guest OS
Hypervisor (虛擬化層)
Host OS (實體機系統)
Hardware (實體硬體)
  • ⚠️ 每個 VM 有獨立的 OS
  • ⏱️ 啟動時間：30 秒 - 2 分鐘
  • 💾 佔用空間：通常為數 GB 級別

🐳 Docker 容器 (Container)

App A
Bin/Lib
App B
Bin/Lib
Docker Engine (引擎)
Host OS (共用系統核心 Kernel)
Hardware (實體硬體)
  • ✨ 共用 Host OS 核心 (Kernel)
  • 🚀 啟動時間：僅需 0.5 - 3 秒
  • 💾 佔用空間：通常僅數十 MB - 數百 MB

一句話記住

VM：虛擬出一整台電腦（包含作業系統）。隔離性強、資源消耗大。

Container：只隔離應用程式和它的相依套件，共用 Host 的 OS kernel。輕量、快速。

第二步：為你的 FastAPI 寫 Dockerfile

Dockerfile 是「建構映像檔的食譜」——告訴 Docker 這個容器裡要裝什麼。

# Dockerfile — 放在你的 FastAPI 專案根目錄

# ① 基底映像：官方的輕量 Python
FROM python:3.12-slim

# ② 設定工作目錄（容器內的資料夾）
WORKDIR /app

# ③ 先複製 requirements，利用 Docker 快取層
#    （這樣 pip install 只有在 requirements 變動時才重跑）
COPY requirements.txt .

# ④ 安裝依賴
RUN pip install --no-cache-dir -r requirements.txt

# ⑤ 複製專案程式碼
COPY . .

# ⑥ 建立非 root 使用者（資安基本動作）
RUN useradd --create-home appuser && chown -R appuser:appuser /app
USER appuser

# ⑦ 告訴 Docker 這個容器監聽哪個 port（文件用途，不實際綁定）
EXPOSE 8000

# ⑧ 啟動指令
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
🔑 COPY requirements 和 COPY . 為什麼要分開？
Docker 會快取每一個步驟。如果 requirements.txt 沒變， 下次 build 時 pip install 這層直接從快取拿，不用重跑。 如果合在一起寫，只要任何程式碼變動，pip install 就要重跑——浪費時間。

第三步：Docker Compose — 一鍵啟動整套服務

你的架構到現在包含：Nginx + FastAPI + PostgreSQL + Redis。Compose 讓它們一起啟動。

# docker-compose.yml — 放在專案根目錄
version: "3.9"

services:
  # === PostgreSQL ===
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: ${DB_PASSWORD}   # ← 從 .env 讀取，不寫死在檔案
      POSTGRES_DB: myapp
    volumes:
      - pgdata:/var/lib/postgresql/data   # 資料持久化（重啟不消失）
    healthcheck:                           # 健康檢查：告訴 Compose 何時 DB 準備好了
      test: ["CMD", "pg_isready", "-U", "myuser"]
      interval: 5s
      retries: 5

  # === Redis ===
  redis:
    image: redis:7-alpine
    volumes:
      - redisdata:/data
    command: redis-server --appendonly yes  # 開啟 AOF 持久化

  # === FastAPI ===
  api:
    build: .                     # 用當前目錄的 Dockerfile 建構
    command: uvicorn main:app --host 0.0.0.0 --port 8000
    environment:
      DATABASE_URL: postgresql://myuser:${DB_PASSWORD}@db:5432/myapp
      REDIS_URL: redis://redis:6379
      # ↑ 注意：host 是服務名稱（db、redis），不是 localhost！
    depends_on:
      db:
        condition: service_healthy   # 等 DB 健康檢查通過才啟動
      redis:
        condition: service_started
    volumes:
      - .:/app            # 開發模式：本地程式碼即時同步到容器
    ports:
      - "8000:8000"

  # === Nginx ===
  nginx:
    image: nginx:alpine
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro  # ro = read-only
    ports:
      - "80:80"           # 對外暴露 80 port
    depends_on:
      - api

# === 命名 volume（Docker 管理，資料不會因容器刪除而消失）===
volumes:
  pgdata:
  redisdata:

開發 vs 生產的 Docker 差異

開發環境 生產環境
程式碼 volumes: .:/app（hot reload） COPY 進映像檔（不可變）
uvicorn --reload 多 worker（--workers 4）
Debug DEBUG=True, verbose log DEBUG=False
Port 直接暴露 只透過 Nginx 暴露 80/443

第四步：常用 Docker 指令速查

# 建構
docker build -t myapp .              # 從 Dockerfile 建構映像檔

# 執行
docker run -d -p 8000:8000 myapp    # -d: 背景, -p: port 映射

# 查看
docker ps                            # 執行中的容器
docker ps -a                         # 所有容器（含已停止）
docker logs -f container_name        # 即時看 log

# Compose
docker-compose up -d                 # 啟動所有服務（背景）
docker-compose down                  # 停止並刪除所有服務
docker-compose logs -f api           # 只看 api 服務的 log
docker-compose exec api bash         # 進入 api 容器的 shell

# 清理
docker system prune -a               # 刪除所有未使用的映像、容器、網路

🔰 練習

Q1. Docker 容器和虛擬機（VM）最關鍵的區別是什麼？

Q2. Dockerfile 中，為什麼要分開 COPY requirements.txt 和 COPY .？

Q3. docker-compose.yml 中 depends_on 的 condition: service_healthy 解決了什麼問題？

Q4. Docker Compose 中，api 服務要連到 PostgreSQL，連線字串的 host 應該寫什麼？

📋 本課重點摘要

  1. Container ≠ VM：容器共用 Host kernel、沒有自己的 OS，所以啟動快、體積小。
  2. Dockerfile 分層快取：先 COPY requirements → pip install → 再 COPY 程式碼。不變的層直接從快取拿。
  3. docker-compose 管理多服務：一鍵啟動 Nginx + FastAPI + Redis + PostgreSQL。
  4. 容器間通訊：host = 服務名稱（如 db、redis），不是 localhost。
  5. volume：資料庫檔案和 Redis 資料要存在 volume 中，容器刪除後資料才不會消失。
  6. 開發 vs 生產：開發用 volume 掛載 hot reload；生產把程式 COPY 進映像檔、多 worker。

🎤 面試回答模板：「你有用過 Docker 嗎？為什麼要用？」

「我們用 Docker 來確保開發、測試、生產環境的一致性。 每個服務（FastAPI、PostgreSQL、Redis、Nginx）各跑在自己的容器中， 用 docker-compose 一鍵啟動整套環境——新人第一天就能把整個後端跑起來。 容器化還解決了『在我電腦上可以跑』的問題：Docker 映像檔包含了 OS 層級的相依性， 在任何機器上跑出來的行為都完全一樣。 在生產環境中，我們把映像檔推到 container registry，VM 上只需要裝 Docker 就能部署， 不需要手動裝 Python、pip install 或設定 Nginx。」
