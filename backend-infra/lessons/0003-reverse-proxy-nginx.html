🔀 反向代理與 Nginx 實戰

Lesson 0003 · ⏱ ~20 min · infra · ← Lesson 0002

你的 uvicorn 正在裸奔

如果你曾經這樣部署： 

uvicorn main:app --host 0.0.0.0 --port 80

那你的 FastAPI 應用程式正直接暴露在網際網路上，沒有任何防護。 這就像在戰場上不穿盔甲——不是不能用，是你不該這樣做。 本課教你如何用 Nginx 反向代理為你的服務穿上盔甲。

🎯 本課目標：能寫出一個生產環境等級的 Nginx config，把 request 正確轉發給 uvicorn， 並清楚解釋反向代理在架構中的角色。

第一步：正向代理 vs 反向代理

面試常問：「正向代理和反向代理有什麼區別？」

一句話記住

正向代理：客戶端知道目標伺服器，但透過代理去訪問（代理客戶）。

反向代理：客戶端以為代理就是伺服器，代理在背後轉發（代理伺服器）。

第二步：為什麼 uvicorn 不該直接暴露？

uvicorn 的設計定位

uvicorn 是一個 ASGI 應用伺服器，它的設計目標是高效執行 Python 非同步程式碼，而不是處理網際網路的各種威脅和複雜性。以下是它不擅長的事：

問題 uvicorn 的弱點 Nginx 的解法
😵 慢速攻擊 (Slowloris) 會被慢速連線耗盡連線池 內建 request 超時與 buffer 機制
🔐 SSL/TLS Python 處理 SSL 效能差 C 語言級的 SSL termination
📁 靜態檔案 用 Python 送靜態檔是浪費 高效 static file serving
🔄 負載均衡 單一 process，無法分配流量 upstream 模組，支援多種算法
🛡 請求過濾 沒有內建 rate limiting limit_req 模組，IP 黑名單
🔧 零停機重啟 重啟時會斷線 graceful reload，不丟失連線

第三步：Nginx 設定檔實戰

以下是一個可以直接用在生產環境的 Nginx 設定，把 HTTPS 請求轉發給本機的 uvicorn：

# /etc/nginx/sites-available/api.example.com

# ① 定義 upstream：uvicorn 的位址（可以有多個 process）
upstream fastapi_backend {
    # 最少連線數演算法：把新請求送給目前連線最少的 worker
    least_conn;

    # uvicorn 通常跑在 127.0.0.1:8000（只監聽本機）
    server 127.0.0.1:8000;
    server 127.0.0.1:8001;  # 可以跑多個 uvicorn process！

    # 如果某個 worker 連續 3 次失敗，暫時踢出 30 秒
    # max_fails=3 fail_timeout=30s;
}

# ② HTTP → HTTPS 強制跳轉
server {
    listen 80;
    server_name api.example.com;
    return 301 https://$server_name$request_uri;
}

# ③ 主要的 HTTPS 伺服器設定
server {
    listen 443 ssl http2;
    server_name api.example.com;

    # === SSL 憑證 ===
    ssl_certificate     /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;

    # === SSL 安全強化 ===
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;

    # === 連線限制 ===
    client_max_body_size 10M;          # 限制上傳大小
    client_body_timeout 12s;           # request body 超時
    client_header_timeout 12s;         # request header 超時

    # === Gzip 壓縮 ===
    gzip on;
    gzip_types application/json text/plain;

    # === 靜態檔案：Python 完全不碰 ===
    location /static/ {
        alias /var/www/api/static/;
        expires 30d;                   # 瀏覽器快取 30 天
        add_header Cache-Control "public, immutable";
    }

    # === API 請求：全部轉發給 uvicorn ===
    location / {
        # 這些 header 讓 FastAPI 知道原始請求的資訊
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 轉發到 upstream 定義的後端群組
        proxy_pass http://fastapi_backend;

        # WebSocket 支援（FastAPI 的 WebSocket 需要）
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        # Proxy 層超時
        proxy_read_timeout 30s;
        proxy_connect_timeout 5s;
    }
}
💡 關鍵 header 解釋：
X-Real-IP：告訴 uvicorn 真正的客戶端 IP（否則 uvicorn 只會看到 127.0.0.1）
X-Forwarded-For：完整的代理鏈 IP 列表
X-Forwarded-Proto：原始請求是 http 還是 https（FastAPI 需要這個來生成正確的 URL）
Host：原始請求的 Host header，讓 FastAPI 知道是哪個域名被呼叫

各司其職：Nginx 用 C 語言的 event-driven 模型處理大量外部連線、SSL、靜態檔案； Python 專注在業務邏輯上。這是「讓對的工具做對的事」。

🛠 實作練習：在你的 VM 上設定 Nginx

任務：親手部署 Nginx + FastAPI

  1. SSH 進入你的 VM
  2. sudo apt install nginx（或對應的套件管理器）
  3. 建立設定檔：sudo nano /etc/nginx/sites-available/api，貼上本課的設定模板
  4. sudo ln -s /etc/nginx/sites-available/api /etc/nginx/sites-enabled/
  5. sudo nginx -t（測試設定檔語法是否正確）
  6. sudo systemctl reload nginx
  7. 確保 uvicorn 在背景執行：uvicorn main:app --host 127.0.0.1 --port 8000 &
    8. → 注意現在是 127.0.0.1，不是 0.0.0.0！
  8. 用瀏覽器打開你的域名——請求現在走 Nginx → uvicorn

📋 本課重點摘要

  1. 正向代理代理客戶端（你 → proxy → 網站），反向代理代理伺服器（網站 → proxy → backend）。
  2. uvicorn 不該直接暴露到網際網路上——它缺乏 SSL 效能、連線防護、rate limiting、靜態檔案服務等能力。
  3. Nginx 用 C 語言的 event-driven 模型（epoll）處理外部連線，Python 專注跑業務邏輯——各司其職。
  4. SSL Termination 發生在 Nginx 層：外部 HTTPS → Nginx 解密 → 內部 HTTP 轉發 uvicorn。
  5. X-Forwarded-* headers 是反向代理的關鍵：它們讓後端知道真實客戶端的資訊。

🎤 面試回答模板：「為什麼需要反向代理？」

「應用伺服器（如 uvicorn/Gunicorn）的設計目標是執行業務邏輯，不是處理網際網路流量。 反向代理（Nginx）提供了幾個關鍵能力：第一，SSL termination——C 語言級的加解密遠比 Python 高效； 第二，靜態檔案服務——不需要讓 Python process 處理這些請求； 第三，緩衝與防護——對抗慢速攻擊、限制 request 大小、rate limiting； 第四，負載均衡——把流量分配給多個 backend process； 第五，零停機部署——Nginx 可以在 backend 重啟時保持連線。 在生產環境中，應用伺服器永遠不應該直接暴露在 80/443 port。」
