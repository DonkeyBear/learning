為什麼面試要考系統設計？

初階後端面試考你「會不會寫 code」。中高階面試考你「會不會想」—— 給你一個模糊的需求，你能不能把它拆解成具體的元件，做出合理的取捨， 並用前面九堂課學到的知識來支撐你的設計決策。

🎯 本課目標：掌握系統設計面試的四步框架，能夠在 45 分鐘內從零設計出一個可擴展的系統， 並在每個決策點引用前面學到的概念來支撐你的論點。

第一步：系統設計的四步框架

面試中最常見的失敗模式是「一上來就畫架構圖」。正確的做法是先釐清需求。

🔄 四步框架（35-45 分鐘）

# 階段 時間 你要做的事
1 釐清需求 5-8 min 問清楚功能範圍、使用者量、讀寫比、延遲要求
2 估算規模 3-5 min 算 QPS、儲存量、頻寬——數字驅動設計
3 高層設計 10-15 min 畫架構圖、定義 API、選資料庫
4 深度探討 10-15 min 挑 2-3 個瓶頸深入：怎麼擴展、怎麼處理故障
⚠️ 最常見的錯誤：跳過需求釐清，直接開始畫圖。這會讓你做出錯誤的假設。 面試官想看的是你問問題的能力——好的工程師先理解問題，再設計解法。

第二步：實戰演練 — 設計一個縮網址服務

題目：「設計一個像 bit.ly 的縮網址服務。」這是系統設計面試的經典入門題。

階段 1：釐清需求（你要主動問！）

🙋 你應該問的問題

  • 「使用者需要登入才能縮網址嗎？還是匿名也可以用？」
    → 面試官：匿名可用，但註冊用戶可以看到自己的縮網址統計
  • 「生成的短網址有 expiration 嗎？」
    → 面試官：預設永遠有效，但付費用戶可以設過期時間
  • 「大概有多少使用者？每天生成多少短網址？」
    → 面試官：每天 1 億次寫入，讀取是寫入的 10 倍（10 億次 redirect）
  • 「短網址多長？」
    → 面試官：7 個字元，由 [a-zA-Z0-9] 組成
  • 「redirect 的延遲要求？」
    → 面試官：要在 10ms 內完成 redirect

階段 2：估算規模（數字驅動設計）

📊 QPS 吞吐量

寫入：1 億 / 天 ≈ 1,200 QPS

讀取：10 億 / 天 ≈ 12,000 QPS

* 讀寫比約為 10 : 1

💾 儲存空間估算

每筆資料：約 130 bytes

每天新增：13 GB / 天

五年容量：約 24 TB

short_url (7B) + long_url (100B) + user_id (8B) + created_at (8B)

💡 核心架構決策

分散式儲存：單一 DB 無法負荷 24TB 級區

快取快取層：應對 12,000 QPS 熱點讀取

非同步寫入：避免統計寫入阻塞關鍵路徑

🔑 回憶 Lesson 0007-0008：12,000 QPS 的讀取量，PostgreSQL 撐得住但很勉強。 但 10ms 延遲要求 + 讀寫比 10:1 → 這是 Cache-Aside 模式的完美場景（Lesson 0008）！ 熱門 URL 的快取命中率會非常高。

階段 3：高層設計

API 設計：

# 建立短網址
POST /api/shorten
  Body:   { "long_url": "https://...", "custom_alias?": "mylink" }
  Header: Authorization: Bearer <JWT> (optional)
  → 201  { "short_url": "https://short.ly/abc1234" }

# 重定向（核心功能）
GET /{short_code}
  → 302  Location: https://original-long-url.com/...

# 查詢統計（登入用戶）
GET /api/stats/{short_code}
  Header: Authorization: Bearer <JWT>
  → 200  { "clicks": 15234, "created_at": "..." }

短碼生成演算法：

# 選項 A：Hash 長網址 → 取前 7 字元（有碰撞風險）
# 選項 B：Base62 編碼自增 ID → 保證不重複（本設計採用）
# 選項 C：預先生成一批短碼存起來（像航空公司的票號池）

import string
BASE62 = string.ascii_letters + string.digits  # a-z, A-Z, 0-9 = 62 chars

def encode_base62(num: int) -> str:
    """把數字轉成 7 位 base62 短碼"""
    chars = []
    for _ in range(7):
        chars.append(BASE62[num % 62])
        num //= 62
    return ''.join(reversed(chars))

# 62^7 = 3.5 兆個短碼，夠用幾十年
# Snowflake ID → base62 → 7 字元短碼

階段 4：深度探討 — 主動提出瓶頸和解法

🔴 瓶頸 1：熱門 URL 的讀取壓力

問題：某個爆紅的短網址可能有 100 萬 QPS。

解法：Redis 快取 + CDN。在 Cloudflare 層設 Cache-Control header，熱門短網址直接從 CDN 邊緣節點回應 302，根本不進到後端。

🔴 瓶頸 2：短碼生成需要全域唯一的 ID

問題：多台 API server 同時生 ID 不能碰撞。

解法：Twitter Snowflake 演算法——41 bits timestamp + 10 bits machine ID + 12 bits sequence number = 在每台機器上本地生成唯一 ID，不依賴資料庫。

🔴 瓶頸 3：資料庫寫入瓶頸

問題：每天 1 億筆寫入，單一 PostgreSQL 不夠。

解法：Sharding——依 short_code 的前 2 個字元來分片。62² = 3844 個 shard。 每個 shard 每天只處理 1 億 / 3844 ≈ 2.6 萬筆寫入，輕鬆應付。

📋 系統設計面試的四步框架（背起來）

  1. 釐清需求（5-8 min）— 功能範圍、使用者量、讀寫比、延遲要求。問！

  • 「讀寫比 10:1 → 用 Cache-Aside 模式，Redis 存熱門資料」
