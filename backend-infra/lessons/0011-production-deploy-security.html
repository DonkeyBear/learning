🔐 自建後端部署安全 — 從 VM 到應用層的全面加固

Lesson 0011 · ⏱ ~30 min · security · ← Lesson 0010

你的 API 上線了，然後呢？

你寫好了 FastAPI、通過了測試、docker build 成功——推到 fly.io 或 GCP VM 上之後， 真正的考驗才剛開始。公開到網路上幾小時內，你的伺服器就會被自動化掃描器探測、 被 bot 嘗試常見漏洞、被爬蟲抓取。這不是「會不會被攻擊」的問題，而是「什麼時候」的問題。

🎯 本課目標：掌握將 FastAPI 後端部署到 GCP VM 或 fly.io 時，從作業系統、網路層、反向代理到應用層的完整安全加固清單。 學完後你能自信地回答：「我把 API 部署到生產環境時，做了哪些安全措施」。

🏰 分層防禦模型

資安的黃金法則：沒有一層防禦是完美的，但多層疊加可以讓攻擊者付出極高代價。 這就是 Defense in Depth（縱深防禦）。在你的部署架構中，每一層都有它的角色：

各層防禦對應的威脅

層級 抵擋的攻擊 工具
平台層 大流量 DDoS、端口掃描、IP 信譽過濾 GCP VPC Firewall / fly.io Proxy

☁️ 第一層：平台選擇與配置

選項 A：GCP Compute Engine VM

你完全掌控 OS——自由度高，但所有安全責任也在你身上（Shared Responsibility Model）。

GCP VM 部署安全清單

  1. VPC Firewall Rules：預設 deny all ingress，只開放 80/443（必要時加 22）
  2. SSH 不允許來自 0.0.0.0/0：限定你的固定 IP 或用 Identity-Aware Proxy (IAP) 隧道
  3. 關閉所有不必要的 port：PostgreSQL (5432)、Redis (6379) 等絕對不能對公網開放
  4. 使用 Service Account 而非個人帳號：VM 綁定最小權限的 SA
  5. 開啟 VPC Flow Logs：記錄所有網路流量，供日後稽查
# 檢查 VM 上哪些 port 在監聽（從 VM 內部執行）
ss -tlnp

# 常見應關閉的服務（除非你真的需要）
sudo systemctl disable --now cups avahi-daemon bluetooth
# cups:    列印服務
# avahi:   mDNS/Bonjour（本地服務發現）
# bluetooth: 藍牙 — 伺服器不需要

選項 B：fly.io

fly.io 是 PaaS——平台幫你處理很多底層安全。預設 deny all：只有你在 fly.toml 中宣告的 port 才會對外開放。 它的 edge proxy 自帶 Anycast 架構（天然分散大流量）、自動 TLS 憑證、以及 HTTP→HTTPS 強制重導向。

fly.io 部署安全清單

  1. 只暴露需要的 port：fly.toml 中 [[services]] 只宣告 80/443
  2. 強制 HTTPS：internal_port 對外只走 TLS，handlers 中啟用 force_https = true
  3. 使用 Private Networking：App 與 DB 之間的溝通走 fly.io 內網（不經公網）
  4. 善用 Fly Proxy 的 HTTP 正規化：它會幫你加上 Fly-Client-IP、X-Forwarded-Proto 等 header
  5. 注意：fly.io 沒有 IP 白名單功能——如果你的 API 需要來源 IP 限制，要在應用層或搭配 Cloudflare 處理
⚠️ fly.io 的限制：fly.io edge proxy 不提供 WAF、DDoS scrubbing、或 IP reputation filtering。 對抗大流量 DDoS 還是建議前面掛 Cloudflare 或類似服務。fly.io 的 Anycast 可以分散部分流量，但不是專門的 DDoS 防護。 
# fly.toml 安全配置範例
app = "my-fastapi-app"
primary_region = "sin"

[build]
  image = "python:3.12-slim"

[[services]]
  protocol = "tcp"
  internal_port = 8000
  force_https = true

  [[services.ports]]
    port = 80
    handlers = ["http"]
  [[services.ports]]
    port = 443
    handlers = ["tls", "http"]
    [services.ports.tls_options]
      alpn = ["h2", "http/1.1"]
      versions = ["TLSv1.2", "TLSv1.3"]

[[vm]]
  cpu_kind = "shared"
  cpus = 1
  memory_mb = 512

🐧 第二層：作業系統加固（防提權）

這是你問的「如何防範提權」的核心。特權提升攻擊遵循固定套路： 攻擊者取得低權限 shell → 枚舉系統弱點 → 利用漏洞取得 root。 你的目標是讓每一步都極度困難。

SSH 防禦：這是你的大門

# /etc/ssh/sshd_config 的關鍵設定
# 1. 禁止 root 直接登入
PermitRootLogin no

# 2. 禁止密碼登入 — 只用 SSH Key
PasswordAuthentication no
PubkeyAuthentication yes

# 3. 如果你有固定 IP，限制 SSH 只接受特定來源
# （在 /etc/hosts.allow 中設定，或直接用 iptables）
# 更安全的做法：改用 GCP IAP Tunnel 或 Tailscale，完全不開 22 port

# 4. 換掉預設的 22 port（降低自動化掃描的噪音）
Port 2222

# 5. 限制可以使用 SSH 的使用者
AllowUsers deployuser

# 修改後重啟
sudo systemctl restart sshd

Fail2Ban：自動封鎖可疑 IP

Fail2Ban 監控 log 檔案，當某個 IP 在短時間內失敗太多次，自動用 iptables 封鎖它。 這是防止 SSH 暴力破解和 API 濫用的第一道自動化防線。 

# /etc/fail2ban/jail.local

# 預設設定
[DEFAULT]
bantime   = 3600       # 封鎖 1 小時
findtime  = 600        # 10 分鐘內的失敗計數
maxretry  = 5          # 失敗 5 次就封鎖
ignoreip  = 127.0.0.1  # 白名單（你自己的固定 IP 也放這裡）

# SSH 防護
[sshd]
enabled   = true
port      = 2222       # 如果你改了 SSH port

# Nginx API 速率限制防護
[nginx-api]
enabled   = true
filter    = nginx-limit-req
logpath   = /var/log/nginx/error.log
maxretry  = 10
findtime  = 60
bantime   = 3600

# 重複違規者 — 加倍懲罰
[recidive]
enabled   = true
logpath   = /var/log/fail2ban.log
bantime   = 86400      # 封鎖 24 小時
findtime  = 86400
maxretry  = 3

防止提權的核心措施

🛡 提權防禦速查表

攻擊向量 防禦措施
SUID 二進位檔被濫用 find / -perm -4000 -type f 2>/dev/null — 移除不必要的 SUID
Sudo 設定不當 嚴格 sudoers 白名單、移除 NOPASSWD、限制 wheel 群組成員
Kernel 漏洞 啟用自動安全性更新（unattended-upgrades）、定期重啟
Cron job 劫持 cron 目錄權限鎖定、腳本中使用絕對路徑
容器逃逸 容器以非 root 執行、關閉 privileged mode、使用 seccomp
弱檔案權限 /etc/shadow、SSH key 設為 600、關鍵檔案用 chattr +i
未修補的 sudo 漏洞 保持 sudo 版本最新 — CVE-2025-32463 (CVSS 9.3) 是活躍攻擊目標

以非 root 執行你的應用

這是最基本也最重要的一步：永遠不要用 root 跑你的 FastAPI。 使用一般使用者或專門的應用使用者。 

# 建立專用的應用使用者（無 shell、無家目錄）
sudo useradd --system --no-create-home --shell /usr/sbin/nologin fastapi

# 你的應用目錄權限
sudo chown -R fastapi:fastapi /opt/myapp
sudo chmod 750 /opt/myapp

# 用 systemd 以該使用者執行
# /etc/systemd/system/fastapi.service:
# [Service]
# User=fastapi
# Group=fastapi
# ...
# Dockerfile：以非 root 執行
FROM python:3.12-slim

# 建立非 root 使用者
RUN groupadd -r fastapi && useradd -r -g fastapi fastapi

WORKDIR /app
COPY --chown=fastapi:fastapi . .

USER fastapi

CMD ["gunicorn", "main:app", "-c", "gunicorn.conf.py"]

防火牆：OS 層的最後防線

# UFW（Ubuntu 上最簡單的方式）
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw allow 2222/tcp    # SSH（如果改了 port）
sudo ufw enable

# iptables（更細粒度控制）
sudo iptables -A INPUT -i lo -j ACCEPT
sudo iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 2222 -s 你的固定IP -j ACCEPT  # SSH 只允許你的 IP
sudo iptables -P INPUT DROP   # 預設拒絕所有其他連入

# 儲存規則（重開機後保留）
sudo apt install iptables-persistent
sudo netfilter-persistent save
⚠️ 關鍵：先允許 SSH 再設定 DROP！如果你先設 DROP 再允許 SSH⋯⋯你就把自己鎖在伺服器外面了。 務必先在 INPUT chain 中加上 SSH 的 ACCEPT 規則，最後才是 -P INPUT DROP。

🔄 第三層：反向代理安全（Nginx）

即使你用 fly.io（它有自己的 proxy），通常還是建議在應用前放一層 Nginx（或在 fly.io 上直接依賴其 proxy）。 若部署在 GCP VM 上，Nginx 是必備。它不只是反向代理，更是你的 Layer 7 防火牆。 

# /etc/nginx/sites-available/api — 安全配置範本
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
limit_conn_zone $binary_remote_addr zone=conn_per_ip:10m;
# 對高敏感路徑（登入、註冊）設定更嚴格的 zone
limit_req_zone $binary_remote_addr zone=login:10m rate=3r/m;

# 用 map 來處理真實客戶端 IP（如果你前面還有 CDN/Proxy）
map $http_x_forwarded_for $client_real_ip {
    default $remote_addr;
    "~*(?P[^,]+)" $ip;
}

server {
    listen 80;
    server_name api.yourdomain.com;

    # 強制 HTTPS
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name api.yourdomain.com;

    # ── TLS 設定 ──
    ssl_certificate     /etc/letsencrypt/live/api.yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.yourdomain.com/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;

    # ── 隱藏伺服器資訊 ──
    server_tokens off;
    proxy_hide_header X-Powered-By;
    more_clear_headers Server;

    # ── 請求大小限制（防止 oversized payload 攻擊）──
    client_max_body_size 10m;
    client_body_buffer_size 128k;

    # ── 連線限制 ──
    limit_conn conn_per_ip 20;       # 每個 IP 最多 20 個並行連線
    limit_req zone=api burst=30 nodelay;  # 通配速率限制

    # ── 安全性 headers ──
    add_header X-Frame-Options "DENY" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header Permissions-Policy "geolocation=(), microphone=(), camera=()" always;

    # ── Proxy 到 FastAPI ──
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 防止 slow loris / 慢速攻擊
        proxy_read_timeout 30s;
        proxy_send_timeout 30s;
        proxy_connect_timeout 10s;
    }

    # 對登入端點更嚴格的速率限制
    location /api/auth/ {
        limit_req zone=login burst=5 nodelay;
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 阻擋對敏感路徑的外部直接存取
    location ~ ^/(admin|\.env|\.git|config) {
        return 404;
    }
}
💡 你需要 nginx-extras才能使用 more_clear_headers： sudo apt install nginx-extras。或是省略那行，只用 server_tokens off。

🐍 第四層：FastAPI 應用安全

Security Headers Middleware

# middleware/security.py
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response

class SecurityHeadersMiddleware(BaseHTTPMiddleware):
    """為每個 response 加上安全性 headers"""

    async def dispatch(self, request: Request, call_next):
        response: Response = await call_next(request)
        response.headers["X-Frame-Options"] = "DENY"
        response.headers["X-Content-Type-Options"] = "nosniff"
        response.headers["X-XSS-Protection"] = "1; mode=block"
        response.headers["Referrer-Policy"] = "strict-origin-when-cross-origin"
        response.headers["X-Permitted-Cross-Domain-Policies"] = "none"
        response.headers["Permissions-Policy"] = (
            "geolocation=(), microphone=(), camera=()"
        )
        # 只有在 HTTPS 的情況下才加 HSTS
        if request.url.scheme == "https":
            response.headers["Strict-Transport-Security"] = (
                "max-age=31536000; includeSubDomains"
            )
        return response


# main.py
from middleware.security import SecurityHeadersMiddleware

app.add_middleware(SecurityHeadersMiddleware)

🚦 重點：API 濫用防護 — 速率限制與 IP 封鎖

這是你問題的核心：不需要鑑權的後端，怎麼防止被濫用？ 答案是分層的速率限制策略。

方案一：slowapi（應用層）

slowapi 是 FastAPI 最成熟的速率限制套件，支援 in-memory 和 Redis 兩種後端。 它也支援 multi-tier limits——同一個端點可以同時有多層限制。 

# 安裝：pip install slowapi redis
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.util import get_remote_address
from slowapi.errors import RateLimitExceeded
from fastapi import FastAPI, Request

# 真實 IP 提取（處理反向代理的情況）
def get_real_ip(request: Request) -> str:
    """優先從 X-Forwarded-For 取得真實客戶端 IP"""
    forwarded = request.headers.get("X-Forwarded-For")
    if forwarded:
        return forwarded.split(",")[0].strip()
    # fly.io 自訂 header
    fly_ip = request.headers.get("Fly-Client-IP")
    if fly_ip:
        return fly_ip
    return request.client.host if request.client else "unknown"

# 自訂 key function：API key 優先，fallback 到 IP
def get_rate_limit_key(request: Request) -> str:
    api_key = request.headers.get("X-API-Key")
    if api_key:
        return f"apikey:{api_key}"
    return f"ip:{get_real_ip(request)}"

# Redis backend（多 worker / 多實例必備）
limiter = Limiter(
    key_func=get_rate_limit_key,
    storage_uri="redis://localhost:6379",  # 生產環境用 Redis
    default_limits=["200/hour", "1000/day"],
    headers_enabled=True,       # 自動加上 X-RateLimit-* headers
    in_memory_fallback_enabled=True,  # Redis 掛了也不崩潰
    strategy="moving-window",   # 滑動視窗（最準確）
)

app = FastAPI()
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)
# 不同端點的分層速率限制

from slowapi import Limiter
from slowapi.util import get_remote_address

limiter = Limiter(key_func=get_remote_address)

# 🔓 公開端點 — 無需鑑權，但限制較嚴
@app.get("/api/public/items")
@limiter.limit("100/minute")        # 每分鐘 100 次
@limiter.limit("1000/hour")         # 每小時 1000 次
async def public_items(request: Request):
    return {"items": [...]}

# 🔐 登入端點 — 極嚴格，防止暴力破解
@app.post("/api/auth/login")
@limiter.limit("3/minute;10/hour;20/day")
async def login(request: Request):
    ...

# 📝 註冊端點
@app.post("/api/auth/register")
@limiter.limit("2/minute;10/day")  # 防止大量假帳號
async def register(request: Request):
    ...

# 🛠 管理端點 — 只有特定 IP 能存取
@app.get("/api/admin/stats")
@limiter.limit("30/minute")
async def admin_stats(request: Request):
    ...

# 全域 fallback（在上面 default_limits 已設定為 "200/hour"）

未鑑權 API 的推薦限制值

端點類型 限制值 原因
公開讀取 (GET) 60-100 / minute 足夠正常使用，防止大量爬取
公開寫入 (POST) 20-30 / minute 寫入成本高，應更嚴格
登入 3 / minute 防止憑證暴力猜解
註冊 2 / minute · 10 / day 防止大量假帳號
全域 fallback 200 / hour · 1000 / day 最後一道防線

方案二：Nginx 層的速率限制

搭配應用層的 slowapi，Nginx 層的 limit_req 可以在請求到達你的 Python 程式之前就拒絕， 節省伺服器資源。這是對抗大量濫用請求的關鍵。 

# 在 Nginx 中定義速率限制 zone
# 10m = 10 MB 的共享記憶體（約可追蹤 16 萬個 IP）

# rate=10r/s → 每秒 10 個請求（平滑速率）
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

# 用 URI 來區分不同端點的 zone（更精準）
limit_req_zone $binary_remote_addr zone=login:10m rate=2r/m;

server {
    # ...

    # 應用在特定路徑
    location /api/auth/login {
        limit_req zone=login burst=3 nodelay;
        # burst=3   → 允許瞬間 3 個請求的緩衝
        # nodelay   → 超過 rate 的請求直接拒絕（不要排隊）
        proxy_pass http://backend;
    }

    location /api/ {
        limit_req zone=api burst=30 nodelay;
        limit_conn conn_per_ip 20;    # 同時只能 20 個連線
        proxy_pass http://backend;
    }
}

# 當被限制時，Nginx 回傳 503。
# Fail2Ban 監控這個 503，自動封鎖重複觸發的 IP（見前面 jail.local）

方案三：Fail2Ban 動態封鎖

這是真正的「鎖 IP」機制。當同一個 IP 在短時間內反覆觸發速率限制，Fail2Ban 直接在 OS 防火牆層封鎖它， 連 Nginx 都不會碰到這個 IP 的後續請求。 

# 建立自訂 filter：/etc/fail2ban/filter.d/nginx-api-abuse.conf
# [Definition]
# failregex = ^<HOST> -.*"(GET|POST).*" (429|403) .*$
#             ^<HOST> -.*limiting requests.*$
# ignoreregex =

# Jail 設定追加到 /etc/fail2ban/jail.local：
# [nginx-api-abuse]
# enabled  = true
# filter   = nginx-api-abuse
# logpath  = /var/log/nginx/access.log
# maxretry = 15        # 15 次 429/403
# findtime = 60        # 60 秒內
# bantime  = 7200      # 封鎖 2 小時
# action   = iptables-allports[name=API-Abuse]

🌊 DDoS 和端口掃描：你到底需要擔心什麼？

DDoS 攻擊的現實

誠實答案：如果你是一台小 VM 或 fly.io 實例，單靠自己的力量擋不住真正的 DDoS。 專業的 DDoS 攻擊流量可以到數十 Gbps——你整台 VM 的網路頻寬可能只有 1-10 Gbps。 「被 DDoS」的實際結果通常是你的雲端帳單爆炸，而不是你的伺服器撐住。

DDoS 三層防禦策略

  1. 前置 CDN / DDoS 清洗服務（對抗大流量攻擊的唯一解）
    Cloudflare（有免費方案）、AWS Shield、GCP Cloud Armor。
    把你的 DNS 指向 Cloudflare → 它幫你吸收/過濾惡意流量 → 只轉送乾淨流量到你的伺服器。
  2. 平台層的速率限制
    GCP VPC Firewall 可以限制 per-IP 的連線速率。
    fly.io 的 Anycast 架構可以將流量分散到最近邊緣節點吸收部分衝擊。
  3. 應用層的資源保護
    即使流量到了你的伺服器，Nginx + Fail2Ban + slowapi 確保你的應用不會被單一來源癱瘓。
⚠️ 現實檢驗：對個人專案或小型服務而言，被真正的 DDoS 攻擊的機率不高。 你更可能遇到的是單一 IP 的濫用爬蟲或自動化掃描——這些用 rate limiting 和 fail2ban 就能有效防禦。 但如果你跑的是有商業價值的服務，Cloudflare 免費方案是最划算的投資。

端口掃描：不用太擔心

端口掃描是攻擊者的第一步——他們掃你的 IP 看哪些 port 開著，再針對性地攻擊。 但它本身不是威脅。你的防禦策略是：

  1. 只開必要的 port（80/443），其他全部關閉 — 讓掃描器找不到任何可攻擊的入口
  2. 不要用常見 port跑敏感服務（如 SSH 改 2222、DB 永遠不公開）
  3. Fail2Ban 會自動封鎖那些嘗試連到不存在 port 的行為（如果你設了對應的 filter）
  4. 不需要 port knocking 這種複雜機制——SSH key 認證 + Fail2Ban 已經夠安全
💡 心態調整：不要把端口掃描當成「被攻擊了」——把它當成背景噪音。 網路上每台公開 IP 的機器，每天都會被自動化掃描數十到數百次。 只要你的 port 是關閉的，掃描器就找不到任何東西。

📋 完整部署安全清單

☑️ 上線前逐一檢查

平台層

  • ☐ 防火牆只開放 80/443（和特定 IP 的 SSH）
  • ☐ SSH 不允許密碼登入、不允許 root 登入
  • ☐ 若用 GCP VM：VPC Firewall default deny + 只允許必要的 ingress
  • ☐ 若用 fly.io：force_https = true、只暴露需要的 port
  • ☐ 考慮掛上 Cloudflare（免費方案即可獲得基本 DDoS 防護 + CDN）

OS 層

  • ☐ unattended-upgrades 啟用安全性自動更新
  • ☐ Fail2Ban 啟用（SSH + Nginx API 防護）
  • ☐ 關閉不必要的服務（cups、avahi、bluetooth...）
  • ☐ /tmp 掛載為 noexec、nosuid
  • ☐ 應用以非 root 使用者執行
  • ☐ 檢查 SUID 二進位檔（移除不必要的）

反向代理層

  • ☐ Nginx server_tokens off + 隱藏版本資訊
  • ☐ client_max_body_size 限制請求大小
  • ☐ Security Headers（HSTS、X-Frame-Options、CSP、X-Content-Type-Options）
  • ☐ limit_req + limit_conn 速率與連線限制
  • ☐ 阻擋敏感路徑（/.env、/.git、/admin）的直接存取
  • ☐ TLS 1.2+ only、禁用不安全的 cipher

應用層

  • ☐ CORS allow_origins 明確指定（絕不用 *）
  • ☐ TrustedHostMiddleware 限制允許的 hostname
  • ☐ Security Headers Middleware
  • ☐ slowapi 分層速率限制（public/api/auth 不同限制值）
  • ☐ 所有端點有輸入驗證（Pydantic 預設就有，確保你沒手動繞過）
  • ☐ Error response 不洩漏 stack trace（debug=False）
  • ☐ Gunicorn + Uvicorn worker、非 root 執行
  • ☐ Health check endpoint（/health）

監控

  • ☐ 日誌寫到檔案（或 stdout，由 systemd/journald 收集）
  • ☐ Fail2Ban log 定期檢查
  • ☐ 429 / 403 / 5xx 錯誤率監控（至少設定告警）
  • ☐ 磁碟空間和 CPU 使用率的基本監控

📋 本課重點摘要

  1. 縱深防禦：平台防火牆 → OS 防火牆 → 反向代理 → 應用層，沒有一層是完美的，疊加才安全。
  2. 防提權：SSH key only + 非 root 執行 + 移除 SUID + 定期安全性更新 + SELinux/AppArmor + sudo 審計。
  3. 速率限制三層：Nginx limit_req（擋在到達應用之前）→ slowapi（應用層精細控制）→ Fail2Ban（OS 層永ban）。
  4. 未鑑權 API 防濫用：以 IP 為 key 做速率限制，公開讀 60-100/min、寫入 20-30/min、登入 3/min、全域 200/hour。
  5. DDoS 現實：單一 VM 擋不了大流量 DDoS，掛 Cloudflare 是最划算方案。你更可能遇到的是單一 IP 濫用爬蟲。
  6. 端口掃描：背景噪音，不是攻擊。只開 80/443，其他全部關閉即可無視。
  7. Security Headers：成本最低、效果最好的安全措施之一——HSTS、X-Frame-Options、CSP、X-Content-Type-Options。

🎤 面試回答模板：「你部署後端時會做哪些安全措施？」

「我採用縱深防禦策略。首先是平台層——GCP 上用 VPC Firewall 預設 deny all、 只開 80/443；fly.io 上強制 HTTPS 且只暴露需要的 port。 OS 層方面，SSH 禁用密碼和 root 登入、開 Fail2Ban 自動封鎖可疑行為、 應用以非 root 使用者執行，並啟用自動安全性更新。 反向代理層，Nginx 設定 security headers（HSTS、CSP、X-Frame-Options）、 limit_req 和 limit_conn 做第一道速率控制。 應用層，CORS 明確指定允許的 origin、用 slowapi 做分層速率限制 （公開端點 100/min、登入 3/min）、所有輸入用 Pydantic 驗證。 如果是對外服務，我還會在 DNS 前面掛 Cloudflare 做 DDoS 防護和 CDN 加速。」
