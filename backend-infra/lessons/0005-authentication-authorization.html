🔐 鑑權系統 — JWT、OAuth2、Session

Lesson 0005 · ⏱ ~25 min · security · ← Lesson 0004

面試官問「你們怎麼做登入？」——你怎麼答？

這可能是後端面試中最常問的題目。它不是考你會不會 call 一個 library， 而是考你理解不同機制的 trade-off——何時用 Session、何時用 JWT、OAuth2 在整個流程中扮演什麼角色。

🎯 本課目標：能畫出三種鑑權機制的流程圖，說出各自的 pros/cons， 並在 FastAPI 中正確實作 JWT + OAuth2。

第一步：Authentication ≠ Authorization

先搞清楚這兩個詞——面試中混用會扣分

Authentication（認證） Authorization（授權）
你是誰？
Who are you?

驗證身分：
• 帳號 + 密碼 ✓
• JWT token 有效 ✓
• OAuth2 Google 登入 ✓ 		你可以做什麼？
What can you do?

驗證權限：
• 普通用戶不能刪除別人文章 ✗
• Admin 可以看後台 ✓
• Pro 會員可以用 API ✓

口訣：Authentication = 驗身分（AuthN），Authorization = 驗權限（AuthZ）。 先 AuthN，再 AuthZ。

第二步：三種鑑權機制全比較

方案 A：Session-Based（伺服器端狀態）

sequenceDiagram autonumber actor Browser as 瀏覽器 (Client) participant Server as 伺服器 (Server) participant DB as Redis / 資料庫 Note over Browser,Server: 1. 登入認證階段 Browser->>Server: POST /login
{username, password} Note over Server: 驗證密碼是否正確 Server->>DB: 產生 session_id: "abc123"
存入對照表: "abc123" ➔ {user_id: 42} DB-->>Server: 儲存成功 Server-->>Browser: 200 OK
Set-Cookie: session_id=abc123 Note over Browser,Server: 2. 存取受保護資源階段 Browser->>Server: GET /users/me
Cookie: session_id=abc123 Server->>DB: 查對照表: session_id 為 "abc123" DB-->>Server: 回傳 {user_id: 42} Server-->>Browser: 200 OK {id: 42, name: "..."}

Session 的優缺點

✅ 優點：伺服器可以隨時撤銷（刪除 session）、敏感資料不離開伺服器、實作簡單

❌ 缺點：需要儲存（Redis/DB）、水平擴展時需要共享 session store、不適合 mobile app

📌 適用場景：傳統的伺服器端渲染網站（如 Django Admin、WordPress）

方案 B：JWT（JSON Web Token，無狀態）

🔑 JWT 結構解構

eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiI0MiJ9.qVYU_jH07...

🔴 Header (標頭)

{
  "alg": "HS256",
  "typ": "JWT"
}

定義演算法與 Token 類型

🔵 Payload (載荷)

{
  "sub": "42",
  "exp": 1719000000
}

攜帶的使用者資訊與過期時間（可被讀取，不可篡改）

🟢 Signature (簽名)

HMACSHA256(
  base64(Header) + "." +
  base64(Payload),
  your-secret-key
)

防止 Token 被惡意篡改的安全簽名驗證

🔑 核心原理：Header 與 Payload 僅以 Base64 編碼，任何人都可以直接讀取其內容，因此絕不能在 Payload 放密碼或身分證號等敏感個資。但因為有 Signature 驗證，任何人都無法修改內容，否則伺服器比對簽名時就會失敗。

JWT 的優缺點

✅ 優點：無狀態（伺服器不需存任何東西）、自帶資訊（user_id 在 token 裡）、適合分散式系統和 mobile app

❌ 缺點：無法單方面撤銷（發出去的 token 在過期前都有效）、payload 是明文（只能編碼不能加密）、token 可能變大

📌 適用場景：SPA + API、微服務之間的通訊、mobile app 後端

⚠️ 面試陷阱題：「JWT 要怎麼登出？」
JWT 本身無法被撤銷——它是無狀態的設計。要實現「登出」必須靠額外手段：
① 黑名單：把要廢掉的 token 放進 Redis（但這樣 JWT 就不是無狀態的了）
② 短過期 + Refresh Token：access token 只活 15 分鐘，登出時丟棄 refresh token
③ 客戶端刪除 token：瀏覽器刪掉 token 就無法發請求，但 token 本身仍然有效
→ 這是 JWT 最經典的 trade-off，面試必問。

方案 C：OAuth2 + OpenID Connect

OAuth2 不是一種「登入方式」，它是一種授權框架（還記得 AuthN ≠ AuthZ 嗎？）。OpenID Connect 是在 OAuth2 之上的身份認證層。

sequenceDiagram autonumber actor User as 使用者 participant Front as 你的前端 (Client) participant Back as 你的後端 (Server) participant AuthServer as Google (授權伺服器) User->>Front: 點擊「Google 登入」 Front-->>User: 重導向 (Redirect) 到 Google 登入頁面 User->>AuthServer: 在 Google 頁面上輸入帳密並授權許可 AuthServer-->>Front: 登入成功，重導向回前端並帶上授權碼 (?code=xyz) Front->>Back: 傳送授權碼 code 給後端 (POST /auth/google) Back->>AuthServer: 在伺服器端用 code + Client Secret 交換 Token AuthServer-->>Back: 回傳 id_token & access_token Back-->>Front: 後端驗證成功，發送本站 Session / JWT 給前端 Front-->>User: 顯示登入成功！

OAuth2 四種 Grant Type（面試常考）

Grant Type 用途 安全性
Authorization Code 有後端的 web app ⭐⭐⭐ 最安全
Client Credentials 伺服器對伺服器（無使用者） ⭐⭐⭐
Implicit SPA（純前端）— 已廢棄！ ⭐ 不安全
Password 你直接拿到使用者密碼 ⭐ 不安全
💡 現實中：大多數應用只用 OAuth2 來做「第三方登入」（Sign in with Google/GitHub/Facebook），然後在自家後端再發一個自己的 JWT 給前端。OAuth2 管「你是誰」，你自己的 JWT 管「你在本站的身分和權限」。

第三步：FastAPI 實作 — JWT + OAuth2

# auth.py — FastAPI 鑑權的完整範例
from datetime import datetime, timedelta, timezone
from typing import Optional
from fastapi import FastAPI, Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer
from passlib.context import CryptContext
import jwt  # pip install pyjwt[crypto]

app = FastAPI()

# === 設定 ===
SECRET_KEY = "your-secret-key-at-least-32-chars"  # 生產環境用環境變數
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 15   # access token 短過期
REFRESH_TOKEN_EXPIRE_DAYS = 7     # refresh token 長過期

# === 密碼雜湊 ===
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

# === OAuth2 端點（FastAPI 會自動產生 /docs 中的登入表單）===
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/login")

# === 建立 JWT ===
def create_access_token(data: dict, expires_delta: Optional[timedelta] = None):
    to_encode = data.copy()
    expire = datetime.now(timezone.utc) + (expires_delta or timedelta(minutes=15))
    to_encode.update({"exp": expire, "type": "access"})
    return jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)

def create_refresh_token(data: dict):
    expire = datetime.now(timezone.utc) + timedelta(days=REFRESH_TOKEN_EXPIRE_DAYS)
    to_encode = {**data, "exp": expire, "type": "refresh"}
    return jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)

# === FastAPI Dependency：從 token 解出當前使用者 ===
async def get_current_user(token: str = Depends(oauth2_scheme)):
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Could not validate credentials",
        headers={"WWW-Authenticate": "Bearer"},
    )
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        user_id: str = payload.get("sub")
        if user_id is None:
            raise credentials_exception
        if payload.get("type") != "access":
            raise HTTPException(status_code=401, detail="Use access token, not refresh token")
        return {"user_id": user_id}
    except jwt.ExpiredSignatureError:
        raise HTTPException(status_code=401, detail="Token has expired")
    except jwt.InvalidTokenError:
        raise credentials_exception

# === 登入端點 ===
@app.post("/login")
async def login(username: str, password: str):
    # ① 驗證帳號密碼（查 DB，這裡省略）
    # user = db.get_user(username)
    # if not verify_password(password, user.hashed_password): raise ...

    # ② 發 access token + refresh token
    access_token = create_access_token(data={"sub": "42"})
    refresh_token = create_refresh_token(data={"sub": "42"})
    return {
        "access_token": access_token,
        "refresh_token": refresh_token,
        "token_type": "bearer"
    }

# === 刷新 token ===
@app.post("/refresh")
async def refresh_token(refresh_token: str):
    try:
        payload = jwt.decode(refresh_token, SECRET_KEY, algorithms=[ALGORITHM])
        if payload.get("type") != "refresh":
            raise HTTPException(status_code=401)
        new_access = create_access_token(data={"sub": payload["sub"]})
        return {"access_token": new_access, "token_type": "bearer"}
    except jwt.ExpiredSignatureError:
        raise HTTPException(status_code=401, detail="Refresh token expired, please login again")

# === 受保護的端點 ===
@app.get("/users/me")
async def get_me(current_user: dict = Depends(get_current_user)):
    # current_user 已經被 get_current_user dependency 解出來了
    return {"user_id": current_user["user_id"], "message": "You are authenticated!"}
🔑 Access Token vs Refresh Token 的設計邏輯：
• Access Token：短壽命（15 分鐘），每次請求都帶，洩漏了也只影響 15 分鐘
• Refresh Token：長壽命（7 天），只用來換新的 access token，不頻繁傳輸，洩漏風險較低
• Refresh token 存在 httpOnly cookie 中（JavaScript 無法讀取），access token 存在記憶體中
→ 這是 JWT 最佳實踐的核心。

🔰 練習

Q1. Authentication（認證）和 Authorization（授權）的區別是什麼？

Q2. JWT 最大的缺點是什麼？

Q3. OAuth2 的 Authorization Code Flow 中，code 的作用是什麼？

Q4. Access Token 過期時間為什麼通常設很短（如 15 分鐘）？

📋 本課重點摘要

  1. AuthN ≠ AuthZ：認證是「你是誰」，授權是「你能做什麼」。先認證，再授權。
  2. Session：伺服器端存狀態，安全可控但無法水平擴展。適合 SSR。
  3. JWT：無狀態、自帶資訊、適合分散式系統。缺點是無法撤銷——靠短過期 + Refresh Token 緩解。
  4. OAuth2：授權框架，不是登入方式。Authorization Code Flow 最安全。現實中通常 OAuth2 驗身分 + 自己的 JWT 管權限。
  5. Access + Refresh Token 是 JWT 最佳實踐的核心——短命的 access token 降低洩漏風險，長命的 refresh token 減少重新登入次數。

🎤 面試回答模板：「你們系統怎麼做鑑權？」

「我們使用 JWT-based 的無狀態鑑權，搭配 access token + refresh token 機制。 使用者登入時，後端驗證帳密後發給兩個 token：access token（15 分鐘過期，存在前端記憶體中）、 refresh token（7 天過期，存在 httpOnly cookie 中，JavaScript 無法存取）。 前端每次請求都帶 access token 在 Authorization header 中， 後端用 FastAPI 的 Dependency 來驗證 JWT 簽名和過期時間。 Access token 過期後，前端用 refresh token 去 /refresh 端點換新的 access token， 使用者不需要重新登入。如果有第三方登入需求（Google/GitHub）， 我們走 OAuth2 Authorization Code Flow，拿到 id_token 後再發自己的 JWT。」
