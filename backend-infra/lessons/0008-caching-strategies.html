⚡ 快取策略 — Redis 實戰

Lesson 0008 · ⏱ ~20 min · performance · ← Lesson 0007

資料庫已經有索引了，還要快取幹嘛？

索引把查詢從 300ms 降到 0.1ms，很讚。但 Redis 可以做到 0.01ms—— 而且不消耗資料庫的連線和 CPU。當你的 API 每秒要處理 1000 個請求時， 資料庫撐不住，Redis 撐得住。

🎯 本課目標：能解釋快取的三種模式、寫出 FastAPI + Redis 的整合程式碼、 並說出快取失效的經典解法。

💡 核心規律：越往上層（越接近用戶）速度越快、越節省伺服器資源，但資料一致性的維護難度也越高。

第二步：三種快取模式

模式一：Cache-Aside（旁路快取）— 最常用

應用程式自己管快取邏輯。快取沒中就自己去查 DB，然後回填快取。

# Cache-Aside：FastAPI 實作
import redis.asyncio as redis
import json

r = redis.Redis(host='localhost', port=6379, decode_responses=True)

async def get_user(user_id: int):
    # ① 先看快取
    cached = await r.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)  # ← Cache Hit！直接回傳

    # ② 快取沒中，查資料庫
    user = await db.query(User).filter(User.id == user_id).first()
    if not user:
        return None

    # ③ 回填快取（設 TTL 以防過期資料永久存在）
    await r.setex(f"user:{user_id}", 300, json.dumps(user.to_dict()))
    return user
💡 Cache-Aside 的優缺點：
✅ 簡單、靈活——快取邏輯完全在應用程式掌控中
❌ 程式碼重複（每個查詢都要寫 if-else）
❌ 快取和 DB 之間有時間差——可能拿到舊資料

模式二：Write-Through（寫穿透）

寫入時同時更新快取和資料庫。快取永遠是最新的。

# Write-Through
async def update_user(user_id: int, data: dict):
    # ① 先寫 DB
    await db.execute(update(User).where(User.id == user_id).values(**data))
    # ② 同時更新快取
    await r.setex(f"user:{user_id}", 300, json.dumps(data))
    # → 下次讀取一定命中快取，且是最新資料

模式三：Write-Behind（寫後寫）

先寫快取，非同步再寫 DB。寫入延遲最低，但風險最高。

# Write-Behind（高風險，慎用）
async def increment_view_count(article_id: int):
    # ① 立刻更新 Redis
    await r.incr(f"views:{article_id}")
    # ② 每秒 flush 一次到 DB（背景排程）
    # 如果 Redis 在 flush 前掛掉 → 資料遺失！

三種模式一句話總結

模式 口訣 一致性 適用場景
Cache-Aside 讀的時候順便存一份 可能不一致 讀多寫少（大部分場景）
Write-Through 寫 DB 順便寫快取 強一致 需要即時最新資料
Write-Behind 先寫快取，晚點寫 DB 可能遺失 計數器、高吞吐寫入

第三步：快取失效 — 電腦科學最難的問題之一

There are only two hard things in Computer Science: cache invalidation and naming things. — Phil Karlton

策略一：TTL（Time To Live）— 最簡單

# 設一個過期時間，時間到就自動刪除
await r.setex("user:42", 300, user_json)  # 5 分鐘後過期
# ✅ 簡單、自動
# ❌ 5 分鐘內的資料可能是舊的

策略二：主動失效 — 最精確

# 資料更新時，立刻刪掉對應的快取
async def update_user(user_id: int, data: dict):
    await db.execute(update(User).where(User.id == user_id).values(**data))
    await r.delete(f"user:{user_id}")  # ← 立刻刪除舊快取
    # 下次讀取時 Cache-Aside 會自動回填新的

策略三：Cache Stampede 防護

當熱門資料的快取過期，瞬間大量請求同時去查 DB——可能打掛資料庫。

# 🚨 Cache Stampede：1000 個請求同時發現快取過期
# → 1000 個請求同時去打 DB → DB 被打爆

# ✅ 解法：互斥鎖（只有第一個請求去查 DB，其他等結果）
async def get_user_safe(user_id: int):
    cached = await r.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)

    # 嘗試拿鎖
    lock_key = f"lock:user:{user_id}"
    if await r.setnx(lock_key, "1"):     # setnx = 只有在 key 不存在時才設值
        await r.expire(lock_key, 10)      # 鎖 10 秒後自動釋放（防死鎖）
        try:
            user = await db.query(User).filter(User.id == user_id).first()
            if user:
                await r.setex(f"user:{user_id}", 300, json.dumps(user.to_dict()))
            return user
        finally:
            await r.delete(lock_key)      # 釋放鎖
    else:
        # 別人正在查 DB，等我一下
        await asyncio.sleep(0.1)
        return await get_user_safe(user_id)  # 重試（這次應該命中快取）

第四步：Redis 不只是快取

Redis 常用資料型別 — 面試亮點

型別 用途 範例
String 一般快取、計數器 INCR page:views
Hash 物件快取（可單獨更新欄位） HSET user:42 name "Alice"
List 訊息佇列、最新動態 LPUSH notifications "msg"
Set 去重、共同好友、標籤 SADD tags:article1 "python"
Sorted Set 排行榜、優先佇列 ZADD leaderboard 1000 "Alice"
Pub/Sub 即時通知、WebSocket 廣播 PUBLISH channel "event"
🔑 面試加分點：「我們用 Redis 不只做快取——Sorted Set 做排行榜、 String + INCR 做 rate limiting 計數器、Pub/Sub 做跨 process 的即時通知。」 這表示你理解 Redis 的更多可能性。

📋 本課重點摘要

  1. Cache-Aside（旁路快取）是最常用的模式：讀快取 → 沒中查 DB → 回填。
  2. Write-Through 寫 DB 時同步更新快取，保證一致性。
  3. Write-Behind 先寫快取再非同步寫 DB，最快但有遺失風險。
  4. Cache Stampede：熱資料過期 → 大量請求同時打 DB。用互斥鎖讓只有第一個請求去查 DB。
  5. Redis 不只是快取——Sorted Set（排行榜）、Pub/Sub（即時通知）、String + INCR（限流計數器）。
  6. 預設 Redis 資料在記憶體中，重啟就沒了。開持久化（RDB/AOF）才能恢復。

🎤 面試回答模板：「你們怎麼用快取？」

「我們用 Redis 做 Cache-Aside 模式的應用層快取。讀取 API 時先查 Redis， 沒命中才查 PostgreSQL 並回填快取（TTL 5 分鐘）。寫入時主動刪除對應快取， 讓下次讀取自動載入新資料。對於熱門資料，我們用 Redis 的 SETNX 實作輕量級鎖來防止 Cache Stampede。 除了快取，我們也用 Redis Sorted Set 做排行榜、String INCR 做 rate limiting 計數器。 生產環境中開啟了 AOF 持久化，防止重啟後資料全部遺失。」
