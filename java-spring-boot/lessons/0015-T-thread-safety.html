執行緒安全與 Spring 並行模型

Spring Boot 的 Service 預設是 singleton——整個應用只有一個實例，所有 HTTP 請求共享它。這表示每秒可能有幾十個執行緒同時呼叫同一個 Service 的方法。如果 Service 中有共享的可變狀態，就會發生 race condition。

🎯 學完這堂課你要能回答 「Spring Boot 如何處理多執行緒？」「為什麼 singleton Bean 需要是無狀態的？」「@Transactional 如何保證交易隔離？」「樂觀鎖和悲觀鎖的差異？何時用哪個？」「HashMap 和 ConcurrentHashMap 的差異？」

1. Spring 的執行緒模型

Spring Boot 內嵌的 Tomcat 使用執行緒池（thread pool）處理請求：

請求 1 ──→ [Thread-1] ──→ TaskService.getTask(1)
請求 2 ──→ [Thread-2] ──→ TaskService.getTask(2)    ← 同一個 singleton！
請求 3 ──→ [Thread-3] ──→ TaskService.createTask(...) ← 同一個 singleton！
請求 4 ──→ [Thread-1] ──→ TaskService.deleteTask(5)   ← Thread-1 處理完請求 1 後被重用

關鍵事實：

  • Tomcat 預設約 200 個執行緒，每個請求分配一個
  • 所有執行緒共享同一個 Service singleton
  • 如果 Service 沒有任何共享的可變欄位，就是執行緒安全的
💡 FastAPI 對比 FastAPI 使用 async/await + 事件迴圈（單執行緒），天然避免了多執行緒競爭問題（但仍有 async 的並行問題）。Spring Boot 是多執行緒模型——一個請求一個執行緒，所以執行緒安全是必須面對的課題。JavaScript/TypeScript 的單執行緒經驗在這裡不適用。

2. 惡魔在狀態裡：Race Condition 實例

以下是一個不安全的 Service——這是面試中常考的負面案例：

@Service
public class CounterService {
    private int count = 0;    // ⚠️ 共享的可變狀態！

    public int increment() {
        return ++count;       // ⚠️ ++ 不是原子操作！（讀-改-寫 三步驟）
    }
}

兩個執行緒同時呼叫 increment()：

Thread-1 讀取 count = 0
Thread-2 讀取 count = 0    ← 在 Thread-1 寫回之前讀到舊值
Thread-1 寫回 count = 1
Thread-2 寫回 count = 1    ← 覆蓋了 Thread-1 的結果！

預期：2
實際：1（遺失了一次遞增）

解法有三層，選對很重要：

// 解法 1：用原子類別（單一變數的遞增遞減 → AtomicInteger）
@Service
public class CounterService {
    private final AtomicInteger count = new AtomicInteger(0);
    public int increment() { return count.incrementAndGet(); }
}

// 解法 2：用 synchronized（多個變數需要原子操作）
@Service
public class TransferService {
    public synchronized void transfer(Account a, Account b, int amount) { ... }
}

// 解法 3：消除狀態（最推薦）——把狀態移到資料庫
@Service
public class CounterService {
    private final CounterRepository repo;
    // 沒有實例欄位！每次請求狀態都在資料庫中
    public int increment() { ... }  // UPDATE counters SET value = value + 1
}
🔥 面試黃金法則 stateless Service = 執行緒安全的 Service。Spring 的最佳實踐是讓 Service 和 Controller 不持有可變的實例欄位，所有狀態都存在資料庫或外部快取中。如果有必須持有的共享狀態，用 ConcurrentHashMap、AtomicInteger 等並行安全容器，或用 synchronized / Lock 保護。

4. @Transactional：資料庫層的並行控制

大多數 Spring Boot 的並行問題不是發生在 Java 程式碼中，而是資料庫層級——兩個請求同時修改同一筆資料。

交易隔離級別（Isolation Level）

@Transactional(isolation = Isolation.READ_COMMITTED)  // Spring 預設
public void updateTask(Long id, String title) {
    // ...
}
隔離級別Dirty ReadNon-repeatablePhantom效能
READ_UNCOMMITTED✅ 可能✅✅最快
READ_COMMITTED❌✅✅↑
REPEATABLE_READ❌❌✅↑
SERIALIZABLE❌❌❌最慢

實務建議：READ_COMMITTED 是大多數情況的最佳平衡。面試中要能說出每個級別防止什麼問題。

5. 悲觀鎖 vs 樂觀鎖

這題後端面試中出現率極高。

🔒 悲觀鎖（Pessimistic）

「我假設一定會有人跟我搶，所以先鎖住。」

用 SELECT ... FOR UPDATE 鎖定該列，其他交易必須等待。適合衝突頻繁的場景（例如扣庫存）。

@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id=:id")
Optional<Product> findByIdWithLock(Long id);

📝 樂觀鎖（Optimistic）

「我假設不太會有人跟我搶，如果真的衝突了再處理。」

用 version 欄位檢查：更新時比對 version，如果不符表示被別人改過，拋出 OptimisticLockException。適合衝突少的場景。

@Entity
public class Product {
    @Version  // ← JPA 自動管理版本
    private Long version;
}
維度悲觀鎖樂觀鎖
機制資料庫層級鎖定應用層版本號比對
適用場景高衝突（搶票、限時秒殺）低衝突（一般 CRUD）
效能衝擊等待/阻塞重試（無等待）
Deadlock 風險有無
Spring 實現@Lock(PESSIMISTIC_WRITE)@Version
🔥 面試標準答案 Q: 悲觀鎖和樂觀鎖的差異？
A: 悲觀鎖假設衝突會發生，在讀取時就鎖定資料列（SELECT FOR UPDATE），其他交易必須等待。適合高衝突場景（如秒殺庫存扣減），但有 deadlock 風險。樂觀鎖假設衝突不常發生，只在更新時比對版本號——如果版本號變了表示被別人改過，拋出異常由應用層處理（通常是重試）。適合一般 CRUD，無鎖等待、無 deadlock。Spring Data JPA 用 @Lock 和 @Version 分別實現兩者。

6. Spring 內建的並行陷阱

陷阱 1：@Transactional 的自呼叫失效

@Service
public class OrderService {

    // ❌ 從同一個 class 的其他方法呼叫 @Transactional 方法不會觸發交易！
    public void processOrder() {
        this.saveOrder();  // ← 不是透過代理呼叫，@Transactional 不生效
    }

    @Transactional
    public void saveOrder() { ... }
}

原因：@Transactional 是透過 AOP 代理實現的（T06 學過），類別內部呼叫繞過了代理。解法：把 @Transactional 方法抽到另一個 Service。

陷阱 2：Spring Bean 的 @Scope("prototype") 不是執行緒安全的保證

Prototype 只是讓你每次取得新實例，不保證內部狀態的執行緒安全。如果你在同一個執行緒中多次操作 prototype Bean 的內部狀態，仍然需要自行處理並行。

自我檢驗 🧠

Q1: 為什麼 Spring Boot 的 Service 預設 singleton 需要考慮執行緒安全？

點擊展開答案

因為 Tomcat 使用多執行緒處理請求，所有執行緒共享同一個 singleton Service 實例。如果 Service 有可變的實例欄位（例如 private int count），多個執行緒同時讀寫會造成 race condition。解決方案：① 讓 Service 無狀態（最佳）② 使用並行安全容器（ConcurrentHashMap、AtomicInteger）③ 用 synchronized 保護關鍵區段。

Q2: 悲觀鎖和樂觀鎖的核心差異？各適用什麼場景？

點擊展開答案

悲觀鎖在讀取時就鎖定資料，其他交易等待（SELECT FOR UPDATE）。樂觀鎖只在更新時比對版本號，衝突時拋異常由應用層重試。悲觀鎖適合高衝突場景（秒殺），但有等待和 deadlock 風險。樂觀鎖適合低衝突場景（一般 CRUD），無鎖等待。Spring 用 @Lock(PESSIMISTIC_WRITE) 和 @Version。

Q3: 為什麼 @Transactional 方法從同一個 class 內部呼叫會失效？

點擊展開答案

@Transactional 透過 AOP 代理實現——Spring 為你的 Service 建立一個 proxy，外部呼叫透過 proxy 才會觸發交易攔截器。但 class 內部用 this.method() 呼叫時繞過了 proxy，直接呼叫原始物件的方法，因此攔截器不觸發。解法：將 @Transactional 方法移到另一個 Service，或注入自己的 proxy（不推薦）。

📌 關鍵 Takeaway

  1. Stateless = Thread-safe：Service 不持有可變欄位，是最根本的執行緒安全策略
  2. Spring 多執行緒，FastAPI 單執行緒——這是兩個框架最根本的並行模型差異
  3. HashMap → ConcurrentHashMap：Singleton 中的集合必須用並行安全版本
  4. 悲觀鎖 = 先鎖再讀（SELECT FOR UPDATE）；樂觀鎖 = 更新時比對版本（@Version）
  5. @Transactional 靠 AOP proxy——內部呼叫會繞過，必須跨 class 呼叫
  6. READ_COMMITTED 是大多數場景的最佳隔離級別
