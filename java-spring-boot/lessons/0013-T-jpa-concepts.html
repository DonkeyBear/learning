Spring Data JPA 概念：ORM 與 Repository 模式

你在 P05 中用 ConcurrentHashMap 模擬了一個 Repository。現在要把它換成真正的資料庫——但不用寫一行 SQL。這就是 JPA（Java Persistence API）的威力。

🎯 學完這堂課你要能回答 「JPA、Hibernate、Spring Data JPA 三者的關係？」「@Entity 和 @Table 的作用？」「JpaRepository 如何自動產生查詢？」「衍生查詢方法（derived query）的命名規則是什麼？」

1. 名詞澄清：JPA vs Hibernate vs Spring Data JPA

這三個詞在你 Google 的時候會反覆出現。它們的關係是：

JPA 規格（Specification）——一份文件，定義了 ORM 應該怎麼做。只是一個標準（像 ECMAScript 定義 JS 應該有什麼語法），不是可以執行的程式庫。
全名：Jakarta Persistence API（以前叫 Java Persistence API）
Hibernate 實作（Implementation）——真正做事的 ORM 框架，實作了 JPA 規格。你引入了 spring-boot-starter-data-jpa，底層就是 Hibernate 在運作。就像瀏覽器實作 ECMAScript 標準一樣。
Spring Data JPA Spring 的包裝層——在 Hibernate 之上再加一層，提供 Repository 介面自動實作、衍生查詢方法、分頁排序等便利功能。是你在程式碼中直接使用的東西。 
你的程式碼
  → Spring Data JPA（Repository 自動實作）
    → Hibernate（ORM 引擎，管理 Entity 生命週期）
      → JDBC（Java 資料庫連線）
        → MySQL / PostgreSQL / H2
💡 FastAPI 對照 JPA ≈ SQLAlchemy ORM 的介面定義
Hibernate ≈ SQLAlchemy 的 ORM engine
Spring Data JPA ≈ FastAPI 的 Depends(get_db) + session 管理的自動化版本
如果你用過 SQLAlchemy 的 session.query(User).filter(...)，JPA 的 entityManager.find(...) 是類似概念——但 Spring Data JPA 更進一步，連查詢都可以靠方法名稱自動產生。

2. @Entity：把 Java Class 變成資料表

這是整個 JPA 的起點。一個最簡單的 Entity：

import jakarta.persistence.*;

@Entity                     // ← 宣告這是一個 JPA Entity
@Table(name = "users")      // ← 對應到資料表 users（不寫則預設用 class 名小寫）
public class User {

    @Id                     // ← 主鍵
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // ← 資料庫自動遞增
    private Long id;

    @Column(nullable = false, unique = true, length = 100)
    private String email;

    @Column(nullable = false)
    private String name;

    @Column(name = "created_at")  // ← Java 用 camelCase，資料表用 snake_case
    private LocalDateTime createdAt;

    // JPA 要求一個無參建構子（可以是 protected，不一定要 public）
    protected User() { }

    // 你用的建構子
    public User(String email, String name) {
        this.email = email;
        this.name = name;
        this.createdAt = LocalDateTime.now();
    }

    // getters...
}
⚠️ JPA Entity 三條鐵律 ① 必須有 @Id（主鍵）。② 必須有無參建構子（JPA 從資料庫讀取資料時需要先用無參建構子建立物件，再用 setter/field 注入值）。③ 不要用 record（record 是不可變的，和 JPA 的運作方式衝突——JPA 需要能修改 Entity 的狀態來反映資料庫變更）。

3. JpaRepository：你定義介面，Spring 寫實作

這是 Spring Data JPA 最神奇的部分。你只需要定義一個 interface，不需要寫實作類別：

import org.springframework.data.jpa.repository.JpaRepository;
import java.util.List;
import java.util.Optional;

// JpaRepository<Entity型別, 主鍵型別>
public interface UserRepository extends JpaRepository<User, Long> {

    // 已經自動擁有的方法（不用寫）：
    //   findAll(), findById(Long id), save(User), deleteById(Long id), count()...

    // === 衍生查詢：靠方法名稱自動產生 SQL ===

    // SELECT * FROM users WHERE email = ?
    Optional<User> findByEmail(String email);

    // SELECT * FROM users WHERE name LIKE ? ORDER BY created_at DESC
    List<User> findByNameContainingOrderByCreatedAtDesc(String keyword);

    // SELECT * FROM users WHERE email = ? AND name = ?
    Optional<User> findByEmailAndName(String email, String name);

    // SELECT COUNT(*) FROM users WHERE created_at > ?
    long countByCreatedAtAfter(LocalDateTime date);
}

P05（手寫）JpaRepository（自動）
public class InMemoryTaskRepository implements TaskRepository { // 50+ 行實作 } public interface UserRepository extends JpaRepository<User, Long> { // 一行實作都不用寫 }
🔥 面試必考：衍生查詢命名規則 findBy[屬性][條件][連結詞][排序]
關鍵字對照：
findBy / readBy / queryBy → SELECT
And / Or → 連接條件
Containing → LIKE %?%
After / Before → > / <
OrderBy...Desc/Asc → ORDER BY

4. 關聯映射：Entity 之間的關係

就像 SQL 有 FOREIGN KEY，JPA 有關聯 annotation：

// 一個 User 有多個 Order
@Entity
@Table(name = "users")
public class User {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // mappedBy = "user" 指向 Order.java 中的 user 欄位
    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL)
    private List<Order> orders = new ArrayList<>();
}

// 多個 Order 屬於一個 User
@Entity
@Table(name = "orders")
public class Order {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)   // LAZY = 延遲載入（推薦）
    @JoinColumn(name = "user_id")        // 外鍵欄位名稱
    private User user;
}

⚠️ FetchType：EAGER vs LAZY

  • EAGER（預設 @OneToOne 和 @ManyToOne）：查詢時自動 JOIN 關聯表，一條 SQL 全部載入
  • LAZY（預設 @OneToMany 和 @ManyToMany）：查詢時不載入關聯，等到你真正呼叫 getOrders() 時才發第二條 SQL
  • 實務建議：幾乎永遠用 LAZY。EAGER 容易造成 N+1 查詢問題和意外的大量資料載入
  • @ManyToOne 預設是 EAGER，建議手動改成 fetch = FetchType.LAZY

5. 對比：FastAPI + SQLAlchemy

把兩邊的 ORM 程式碼並排，結構驚人地相似：

Spring Data JPAFastAPI + SQLAlchemy
@Entity class User { @Id Long id; String name; } class User(Base): id: Mapped[int] = mapped_column(primary_key=True) name: Mapped[str]
interface UserRepository extends JpaRepository<User, Long> { List<User> findByName(String n); } def get_users(db: Session, name: str): return db.query(User).filter(User.name == name).all()
userRepo.save(new User("Alice")); db.add(User(name="Alice")) db.commit()

關鍵差異：Spring Data JPA 讓你不用寫 SQL 也不用寫查詢邏輯——方法名稱就等於查詢。這是 Java 生態系特有的「約定大於配置」極致體現。

自我檢驗 🧠

Q1: JPA、Hibernate、Spring Data JPA 三者的關係是什麼？

點擊展開答案

JPA 是規格（一份 API 標準文件），Hibernate 是 JPA 的實作（真正做 ORM 的框架），Spring Data JPA 是 Spring 在 Hibernate 之上的包裝層（提供 Repository 自動產生、衍生查詢等便利功能）。你用 Spring Data JPA 的 interface → 它委派給 Hibernate → Hibernate 產生 SQL → 透過 JDBC 執行。

Q2: Entity 的 @OneToMany 和 @ManyToOne 的 fetch 預設值分別是什麼？為什麼建議改用 LAZY？

點擊展開答案

@OneToMany 和 @ManyToMany 預設 LAZY。@OneToOne 和 @ManyToOne 預設 EAGER。建議幾乎永遠用 LAZY，因為 EAGER 會在不必要的時候 JOIN 關聯表，造成① N+1 查詢問題（查 10 個 User 卻產生了 11 條 SQL）② 載入大量不需要的資料 ③ 序列化時可能造成無限迴圈。所以 @ManyToOne 要手動加上 fetch = FetchType.LAZY。

Q3: Spring Data JPA 的衍生查詢方法，寫出一個查詢「email 包含某字串且建立時間在指定日期之後，依 id 降冪排列的前 5 筆」的方法簽章。

點擊展開答案 
List<User> findTop5ByEmailContainingAndCreatedAtAfterOrderByIdDesc(
    String emailKeyword, LocalDateTime after
);

📌 關鍵 Takeaway

  1. JPA 是規格，Hibernate 是實作，Spring Data JPA 是便利層
  2. @Entity + @Id 是 JPA 的起手式；Entity 不能用 record（必須可變）
  3. JpaRepository 讓你只定義 interface，實作由 Spring 自動產生——對比 P05 的手寫 Repository，這是質的飛躍
  4. 衍生查詢靠方法命名自動產生 SQL（findBy + 屬性 + 條件 + 排序）
  5. @ManyToOne 預設 EAGER，務必手動改成 LAZY
  6. Entity 必須有無參建構子——JPA 需要它來從資料庫結果集重建物件
