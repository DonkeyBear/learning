DI 與 IOC 深度解析：面試必考核心

這是整個課程最重要的理論課。IoC 和 DI 不只是面試萬年考題，更是理解 Spring Boot 一切行為的鑰匙。不理解 IoC 容器，就無法真正理解 Spring Boot。

🎯 學完這堂課你要能回答 「請解釋什麼是 IoC 和 DI？」「Spring 中有哪些注入方式？各有什麼優缺點？」「IoC 容器是怎麼管理 Bean 的？」——這三題是後端面試的入場券。

1. IoC 到底是什麼？用生活例子理解

沒有 IoC 的世界（傳統做法）

你需要一把螺絲起子，所以你自己去五金行買材料、自己打造一把：

// 你控制一切：建立物件、管理依賴
public class Mechanic {
    public void fixCar() {
        var tool = new Screwdriver();    // 你自己 new
        var parts = new SpareParts();    // 你自己 new
        tool.use();
        parts.install();
    }
}

有 IoC 的世界（Spring 的做法）

你只需要說「我需要一把螺絲起子」，工具箱（IoC 容器）會自動遞給你：

// 框架控制一切：你只管宣告需求，框架負責提供
@Component
public class Screwdriver { ... }

@Component
public class SpareParts { ... }

@Component
public class Mechanic {
    private final Screwdriver tool;
    private final SpareParts parts;

    // 你只宣告你需要什麼，框架自動傳入
    public Mechanic(Screwdriver tool, SpareParts parts) {
        this.tool = tool;
        this.parts = parts;
    }
}

控制反轉（IoC）= 把「建立和管理物件」的控制權從你的程式碼轉移到框架。你的程式不再主動「做」事，而是被動「宣告需求」，由框架來滿足。

💡 IoC ≠ DI IoC（控制反轉）是原則——一種設計理念。DI（依賴注入）是實作 IoC 的具體手段。還有其他實作 IoC 的方式（例如 Service Locator 模式），但 DI 是最主流、Spring 採用的方案。面試中問「IoC 和 DI 的關係」就是在考你有沒有搞懂這個層次。

2. 三種依賴注入方式

Spring 提供三種注入方式。面試中常問「你推薦用哪一種？為什麼？」

方式一：建構子注入（Constructor Injection）✅ 推薦

@RestController
public class UserController {
    private final UserService userService;

    // Spring 自動從容器中找 UserService 傳入
    public UserController(UserService userService) {
        this.userService = userService;
    }
}

優點：

  • 欄位可以宣告為 final → 不可變，執行緒安全
  • 強制依賴：沒有傳入就不能建立物件 → 不會忘記設定
  • 純 Java：不依賴 Spring annotation（方便單元測試）
  • Spring 官方推薦

方式二：Setter 注入（Setter Injection）⚠️ 特定場景

@RestController
public class UserController {
    private UserService userService;

    @Autowired   // ← 必須加 @Autowired
    public void setUserService(UserService userService) {
        this.userService = userService;
    }
}

缺點：欄位不能 final，可能被中途修改（mutable）。僅在「依賴是選擇性的」（可有可無）時使用。

方式三：欄位注入（Field Injection）❌ 不推薦

@RestController
public class UserController {
    @Autowired
    private UserService userService;  // ← 直接寫在欄位上
}

為什麼不推薦：

  • 欄位不能 final → 不可變性被破壞
  • 難以單元測試——你需要反射來注入 mock
  • 隱藏了依賴數量——建構子參數太多會提醒你重構
  • 面試中說「我推薦建構子注入」會加分，說「我常用 @Autowired 欄位」會扣分
🔥 面試標準答案 Q: 你推薦哪種注入方式？
A: 建構子注入。原因：① 欄位可用 final 確保不可變性與執行緒安全 ② 強制依賴不為 null ③ 純 Java 不依賴 Spring annotation 容易測試 ④ 當建構子參數過多時自然提醒你這個 class 做了太多事（SRP）。唯一例外是選擇性依賴時用 setter 注入。Spring 4.3+ 後只有一個建構子時甚至不用加 @Autowired。

3. Spring IoC 容器的內部結構

Spring 的 IoC 容器本質上是一個巨大的 Map：

// IoC 容器的概念模型（簡化版）
Map<String, Object> container = new HashMap<>();
// key   = bean 名稱（預設是 class 名的小駝峰）
// value = bean 實例（singleton 預設只會有一個）

// 當你寫 @Service + 建構子注入，Spring 等於做了：
container.put("userService", new UserService(userRepo));
container.put("userController", new UserController(
    (UserService) container.get("userService")
));

BeanFactory vs ApplicationContext

BeanFactoryApplicationContext
定位低階容器介面高階容器介面（繼承 BeanFactory）
Bean 實例化懶加載（用到才建立）預載（啟動時就建立 singleton Bean）
功能只有 DIDI + AOP + 國際化 + 事件發布 + ...
實際使用幾乎不直接用Spring Boot 預設
 
// Spring Boot 啟動時你拿到的就是 ApplicationContext
var ctx = SpringApplication.run(DemoApplication.class, args);
// ctx 的實際型別是 AnnotationConfigApplicationContext（ApplicationContext 的子類）

P04 中你印出了所有的 Bean 名稱——那就是 ApplicationContext 的內容。你看到的 helloController、greetingService 和幾十個 Spring 內建 Bean 都住在那個 Map 裡。

4. FastAPI Depends() vs Spring DI：同一個概念，兩種實現

把兩邊的程式碼並排來看，你會發現它們解決的是同一個問題，只是機制不同：

FastAPISpring Boot
宣告依賴Depends(get_service)建構子參數 + @Service
解析機制函式呼叫鏈（每次請求時解析）IoC 容器預先建立（啟動時就組裝好）
生命週期預設每個請求建立一次預設 singleton（整個應用只有一個）
覆蓋/替換app.dependency_overrides@Primary / @Qualifier / profile
適用場景中小型後端、微服務大型企業應用、複雜依賴圖
 
# FastAPI：每個請求時解析依賴
def get_user_service():
    return UserService()

@app.get("/users")
def get_users(svc: UserService = Depends(get_user_service)):
    return svc.get_all()

// Spring Boot：啟動時就建好整個依賴圖
@Service
public class UserService {
    public List<UserDto> getAll() { ... }
}

@RestController
public class UserController {
    private final UserService svc;
    public UserController(UserService svc) { this.svc = svc; }

    @GetMapping("/users")
    public List<UserDto> getUsers() { return svc.getAll(); }
}
🎯 關鍵差異 FastAPI 的依賴是請求範圍的——每個 HTTP request 都可能建立新的 Service 實例。Spring 的依賴預設是應用範圍的——singleton，整個應用只有一個實例，所有請求共享。這對執行緒安全有重大影響（T08 會深入）。

5. @Qualifier：當同一個介面有多個實作

這是實務中常遇到的情境：

public interface PaymentService {
    void pay(double amount);
}

@Service("creditCard")  // ← 給 Bean 一個名稱
public class CreditCardPaymentService implements PaymentService {
    public void pay(double amount) { /* 信用卡 */ }
}

@Service("paypal")
public class PayPalPaymentService implements PaymentService {
    public void pay(double amount) { /* PayPal */ }
}

如果你的 Controller 需要某個特定實作：

@RestController
public class CheckoutController {
    private final PaymentService paymentService;

    public CheckoutController(
        @Qualifier("creditCard") PaymentService paymentService  // ← 指名要哪個
    ) {
        this.paymentService = paymentService;
    }
}

或者用 @Primary 指定預設值：

@Service
@Primary   // ← 當有多個 PaymentService 時，這個是預設選擇
public class CreditCardPaymentService implements PaymentService { ... }

6. IoC 容器的優缺點（面試加分題）

✅ 優點

  • 鬆耦合：依賴由容器管理，換實作只需改 annotation
  • 易測試：可注入 mock 而不需改程式碼
  • 集中管理：所有物件的生命週期統一由容器控制
  • 減少 boilerplate：不用手寫工廠、singleton 模式

⚠️ 缺點

  • 學習曲線：新手需要時間理解「框架在背後做了什麼」
  • 除錯難度：依賴圖複雜時，追蹤問題需要了解容器狀態
  • 啟動成本：大型應用容器初始化可能需要數十秒
  • 過度使用：把所有東西都變成 Bean 反而增加複雜度

自我檢驗 🧠

Q1: IOC 和 DI 分別是什麼？它們的關係是什麼？

點擊展開答案

IoC（控制反轉）是一種設計原則——把建立和管理物件的控制權從程式碼轉移到框架。你不再自己 new 物件，而是由容器來建立和組裝。DI（依賴注入）是實作 IoC 的具體手段——容器在建立物件時，自動將其依賴的物件「注入」進去。IoC 是理念，DI 是方法。Spring 的 DI 主要透過建構子注入來實現。

Q2: Spring 的三種注入方式各是什麼？為什麼推薦建構子注入？

點擊展開答案

① 建構子注入（推薦）：透過建構子參數，欄位可設 final，強制依賴不為 null，純 Java 易測試。② Setter 注入：透過 setter + @Autowired，適用於選擇性依賴。③ 欄位注入（不推薦）：@Autowired 直接加在欄位上，無法 final、難以測試、隱藏依賴數量。

Q3: 當同一個 interface 有多個 @Service 實作時，Spring 如何決定注入哪一個？

點擊展開答案

Spring 遇到多個同型別的 Bean 時會拋出 NoUniqueBeanDefinitionException。解決方式有三種：① @Primary：標記一個預設實作 ② @Qualifier("beanName")：在注入點指名要哪一個 ③ 使用 @Resource(name="...")（JSR-250 標準）。實務中最常用 @Primary 設定預設 + @Qualifier 處理例外。

📌 面試用一句話總結

「IoC 是把物件建立的控制權交給框架；DI 是框架自動把依賴注入到你宣告的地方。Spring 透過 ApplicationContext（一個巨大的 Map）在啟動時預先建立所有 singleton Bean，並用建構子注入自動組裝依賴圖。」
