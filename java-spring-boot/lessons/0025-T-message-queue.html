Message Queue：非同步訊息與 Spring Boot 整合

目前你的 blog-api 專案中，所有操作都是同步的——使用者註冊後立即回傳、文章建立後馬上寫入資料庫。但真實世界的後端有很多事不需要立即完成：發送 Email、產生報表、更新搜尋索引。Message Queue（MQ）就是用來處理這類場景的核心工具。

🎯 學完這堂課你要能回答 「什麼是 Message Queue？解決什麼問題？」「RabbitMQ 和 Kafka 的差異？」「Spring Boot 如何整合 RabbitMQ？」「何時該用同步 REST、何時該用非同步 MQ？」

1. 為什麼需要 Message Queue？

沒有 MQ 的世界（你的 blog-api 現狀）

// 使用者註冊後的同步處理
@PostMapping("/register")
public ApiResponse<UserResponse> register(@Valid @RequestBody RegisterRequest req) {
    var user = userService.register(req);   // 1. 寫入資料庫
    emailService.sendWelcome(user);          // 2. 發送歡迎信（等 2 秒）
    searchService.indexUser(user);           // 3. 更新搜尋索引（等 1 秒）
    analyticsService.track("signup");        // 4. 追蹤事件（等 0.5 秒）
    return ApiResponse.ok(user);             // 5. 3.5 秒後才回傳給使用者！
}

有 MQ 的世界

@PostMapping("/register")
public ApiResponse<UserResponse> register(@Valid @RequestBody RegisterRequest req) {
    var user = userService.register(req);   // 1. 寫入資料庫
    eventPublisher.publish(new UserRegisteredEvent(user));  // 2. 發送訊息到 Queue
    return ApiResponse.ok(user);             // 3. 立即回傳（< 100ms）
}
// Email、搜尋、分析各自在自己的 consumer 中非同步處理

MQ 解決的核心問題

非同步處理耗時操作不阻塞主流程，使用者立即得到回應
服務解耦Producer 不知道 Consumer 的存在——新增 Consumer 不改 Producer
流量削峰瞬間大量請求先進 Queue 排隊，Consumer 依自身速度消化
可靠性Consumer 掛了？訊息留在 Queue 中，重啟後繼續處理

2. 核心概念：Producer、Queue、Consumer

Producer                    Broker                     Consumer
(你的 App)                 (RabbitMQ)                 (背景服務)

UserService ──→ [Queue: email] ──→ EmailConsumer（發送歡迎信）
  註冊成功  ──→ [Queue: search] ──→ SearchConsumer（更新索引）
            ──→ [Queue: analytics] ──→ AnalyticsConsumer（追蹤事件）
術語解釋對應你的程式碼
Producer發送訊息的一方UserService 或 PostService
Queue存放訊息的地方（FIFO）email-queue、search-queue
Consumer接收並處理訊息的一方EmailConsumer class
ExchangeRabbitMQ 的路由中心決定訊息該送到哪個 Queue
BindingExchange 和 Queue 的連線規則「所有 user.registered 送到 email-queue」
BrokerMQ 伺服器本身RabbitMQ Server

3. RabbitMQ vs Kafka：面試必問題

RabbitMQApache Kafka
核心模型訊息 Queue（FIFO）分散式日誌（Log）
訊息刪除Consumer 確認後刪除保留一段時間（如 7 天），可重複讀取
吞吐量數萬/秒百萬/秒
RoutingExchange + Routing Key（靈活）Topic + Partition（簡單）
適用場景指令型（發 Email、通知）事件型（點擊流、日誌、跨服務通訊）
延遲毫秒級毫秒～秒級
Spring 支援spring-boot-starter-amqpspring-kafka
🔥 面試一句話總結 RabbitMQ 是聰明 broker、笨 consumer（broker 負責路由），適合業務指令的非同步處理。Kafka 是笨 broker、聰明 consumer（consumer 自己追蹤進度），適合大規模事件串流和跨服務資料同步。中小型專案用 RabbitMQ，微服務大資料用 Kafka。

4. Spring Boot 整合 RabbitMQ

4.1 依賴與設定

build.gradle：

dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-amqp'
}

application.properties（本機 Docker RabbitMQ）：

spring.rabbitmq.host=localhost
spring.rabbitmq.port=5672
spring.rabbitmq.username=guest
spring.rabbitmq.password=guest

4.2 宣告 Queue、Exchange、Binding

config/RabbitMQConfig.java：

package com.blogapi.config;

import org.springframework.amqp.core.*;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class RabbitMQConfig {

    // 定義 Queue
    @Bean
    public Queue emailQueue() {
        return QueueBuilder.durable("email-queue").build();
        // durable：RabbitMQ 重啟後 Queue 還在
    }

    // 定義 Exchange（Topic 類型最常用）
    @Bean
    public TopicExchange blogExchange() {
        return new TopicExchange("blog-exchange");
    }

    // 綁定：routing key 符合 "user.registered" 的訊息送到 email-queue
    @Bean
    public Binding emailBinding(Queue emailQueue, TopicExchange blogExchange) {
        return BindingBuilder.bind(emailQueue)
            .to(blogExchange)
            .with("user.registered");
    }
}

4.3 Producer：發送訊息

package com.blogapi.event;

import org.springframework.amqp.rabbit.core.RabbitTemplate;
import org.springframework.stereotype.Component;

@Component
public class EventPublisher {
    private final RabbitTemplate rabbitTemplate;

    public EventPublisher(RabbitTemplate rabbitTemplate) {
        this.rabbitTemplate = rabbitTemplate;
    }

    public void publishUserRegistered(Long userId, String email, String displayName) {
        var event = new UserRegisteredEvent(userId, email, displayName);
        rabbitTemplate.convertAndSend(
            "blog-exchange",      // exchange
            "user.registered",     // routing key
            event                  // 訊息本體（自動 JSON 序列化）
        );
    }
}

// 訊息 DTO（用 record！）
public record UserRegisteredEvent(
    Long userId, String email, String displayName
) { }

4.4 Consumer：接收並處理訊息

package com.blogapi.event;

import org.springframework.amqp.rabbit.annotation.RabbitListener;
import org.springframework.stereotype.Component;

@Component
public class EmailConsumer {

    @RabbitListener(queues = "email-queue")
    public void handleUserRegistered(UserRegisteredEvent event) {
        System.out.println("📧 發送歡迎信給: " + event.email());
        // 實際發送 Email 的邏輯
        // 如果丟出例外，RabbitMQ 會自動重試（預設 3 次）
    }
}
💡 @RabbitListener 背後是 Observer 模式 還記得補充課學的 Observer 模式嗎？@RabbitListener 就是 Observer 在 MQ 世界的實現——Producer 發佈事件到 Queue，Consumer 透過 @RabbitListener 自動接收。和 Spring Events（@EventListener）的差異：MQ 是跨服務、跨 JVM 的；Spring Events 只在同一個 JVM 內。

5. 何時用同步 REST vs 非同步 MQ？

場景用什麼原因
查詢使用者資料同步 REST使用者需要立即看到結果
建立文章同步 REST需要確認寫入成功
發送歡迎信MQ不需立即完成，失敗可重試
產生月報表MQ耗時長（幾分鐘），不該阻塞 HTTP 請求
更新搜尋索引MQ次要操作，不影響使用者體驗
庫存扣減同步 REST必須立即確認（防止超賣）
跨服務通知MQ解耦；發送方不該知道誰在接收
⚠️ 不要用 MQ 替代所有同步呼叫 MQ 引入了最終一致性——訊息可能延遲、可能重複、可能失敗。如果你的場景需要立即確認結果（如扣庫存、確認付款），同步 REST + 資料庫交易才是正確選擇。MQ 適合「做了很好，但不做也不會世界末日」的操作。

6. MQ 的常見陷阱與面試加分題

陷阱 1：重複消費

Consumer 處理成功但 ACK 遺失 → MQ 重送 → 同一訊息被處理兩次。
解法：Consumer 實作冪等性（idempotent）——用 messageId 或業務 ID 檢查是否已處理過。

陷阱 2：訊息遺失

Producer 發送後 Broker 還沒寫入磁碟就掛了。
解法：Publisher Confirm（RabbitMQ 確認寫入後才回 ACK）+ Queue 宣告為 durable + 訊息標記 persistent。

陷阱 3：死信 Queue（DLQ）

訊息重試 N 次後仍然失敗 → 不能無限重試。
解法：設定 x-dead-letter-exchange，失敗 N 次的訊息自動轉到 DLQ，由人工或排程處理。

自我檢驗 🧠

Q1: 使用者註冊後要發送 Email、更新搜尋索引、記錄分析事件——這些用同步 REST 還是 MQ？為什麼？

答案

用 MQ。這三個操作都不需要立即回傳結果給使用者，失敗可以重試，不影響核心業務（註冊本身）。用 MQ 可以：① 讓註冊 API 快速回傳 ② 三者獨立處理，互不影響 ③ 某個 Consumer 掛了不影響其他 Consumer 和主流程。

Q2: RabbitMQ 和 Kafka 的核心差異？各適合什麼場景？

答案

RabbitMQ 是訊息 Queue 模型：訊息處理完即刪除、broker 負責路由、毫秒級延遲。適合業務指令（發 Email、通知）。Kafka 是分散式日誌模型：訊息保留一段時間、consumer 自行追蹤進度、百萬級吞吐。適合大規模事件串流（點擊流、跨服務資料同步）。

Q3: 什麼是死信 Queue（DLQ）？解決什麼問題？

答案

DLQ 是一個特殊的 Queue，用來接收「正常處理多次後仍然失敗」的訊息。解決的問題：訊息不能無限重試（會阻塞後續訊息），也不能直接丟棄（會遺失資料）。DLQ 將失敗訊息隔離，由人工或排程任務單獨處理，確保正常訊息能繼續流動。

📌 關鍵 Takeaway

  1. MQ 解決非同步、解耦、削峰、可靠性——讓慢操作不阻塞快操作
  2. Producer → Exchange → Queue → Consumer：RabbitMQ 的核心資料流
  3. RabbitMQ 適合業務指令，Kafka 適合事件串流
  4. Spring 整合極簡：RabbitTemplate 發送、@RabbitListener 接收
  5. MQ 不是銀彈：需要立即回應的場景用同步 REST + 交易
  6. 必須處理重複消費和訊息遺失：idempotent + publisher confirm + DLQ
