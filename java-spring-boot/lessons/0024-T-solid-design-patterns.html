SOLID 原則與常見 Design Patterns

SOLID 和 Design Patterns 是後端面試的必考題——而且面試官通常會追問「你在 Spring Boot 中哪裡用過這個」。這堂課把每個原則和模式對應到你已經在這門課中寫過的程式碼，讓你能用實例回答，而非背誦定義。

🎯 這堂課的定位 將分散在 24 堂課中的設計原則和模式系統化整理。你已經用過大部分的 pattern——只是當時沒叫出它們的名字。這堂課把名字和程式碼對齊。

Part A：SOLID 原則

S — 單一職責原則（Single Responsibility）

定義：一個 class 應該只有一個改變的理由。

你已經在用的地方：整個課程的三層架構就是 SRP 的體現——Controller 只處理 HTTP、Service 只處理業務邏輯、Repository 只處理資料存取。

// ❌ 違反 SRP：這個 class 同時處理 HTTP、業務、資料存取
@RestController
public class TaskController {
    @Autowired EntityManager em;  // 直接操作資料庫

    @GetMapping("/tasks")
    public List<Task> list() {
        // Controller 不應該知道 EntityManager 的存在
        return em.createQuery("SELECT t FROM Task t").getResultList();
    }
}

// ✅ 符合 SRP：各層各司其職
@RestController  // ← 只負責 HTTP
public class TaskController {
    private final TaskService service;  // ← 業務邏輯委派給 Service
}
@Service        // ← 只負責業務邏輯
public class TaskService {
    private final TaskRepository repo;   // ← 資料存取委派給 Repository
}

// 💡 額外例子：P11 的 PostService 不直接碰 CategoryRepository
// 而是透過 CategoryService.getOrCreate()——每個 Service 只管自己的領域

O — 開閉原則（Open/Closed）

定義：對擴展開放、對修改關閉。新增功能時不應修改既有程式碼。

你已經在用的地方：Interface + @Primary/@Qualifier 切換實作、BeanPostProcessor 擴展點、@ControllerAdvice 新增例外處理。

// ✅ OCP：新增付款方式不需要改 CheckoutService
public interface PaymentService {
    void pay(double amount);
}

@Service("creditCard")
public class CreditCardPayment implements PaymentService { ... }

// 新增 PayPay 付款——只需加一個 class，不改任何既有程式碼！
@Service("paypay")
public class PayPayPayment implements PaymentService { ... }

// CheckoutService 完全不需修改
@Service
public class CheckoutService {
    private final PaymentService payment; // 注入新實作即可
}

// 💡 Spring 本身就是 OCP 的極致：BeanPostProcessor、@ControllerAdvice
// 都是「加新 class 就能擴展行為」而不用改框架原始碼

L — 里氏替換原則（Liskov Substitution）

定義：子類別必須能完全替換父類別，而不破壞程式的正確性。

你已經在用的地方：P02 的 BaseEntity abstract class、P05 的 TaskRepository interface。

// ✅ LSP：InMemoryTaskRepository 和 FileTaskRepository 都能替換 TaskRepository
public interface TaskRepository {
    List<Task> findAll();
    Optional<Task> findById(String id);
}

// 任何使用 TaskRepository 的程式碼不應該知道（也不在乎）
// 底層是 InMemory 還是 File 實作
@Service
public class TaskService {
    private final TaskRepository repo;
    // ↑ 傳入任何符合介面合約的實作，行為都正確
}

// ❌ 違反 LSP：子類別改變了父類別方法的語意
public class BrokenRepository implements TaskRepository {
    public List<Task> findAll() {
        throw new UnsupportedOperationException();  // ← 破壞合約！
    }
}

I — 介面隔離原則（Interface Segregation）

定義：不該強迫 client 依賴它不需要的方法。大介面應拆成多個小介面。

你已經在用的地方：P05 的 TaskRepository 只有 4 個方法（小而專注）、JpaRepository 繼承多層小介面（CrudRepository → PagingAndSortingRepository → JpaRepository）。

// ❌ 違反 ISP：肥介面強迫所有人都要實作不需要的方法
public interface MegaRepository<T> {
    T save(T entity);
    List<T> findAll();
    void sendEmail(T entity);    // ← 只有部分實作需要
    void exportToPdf(T entity);  // ← 只有部分實作需要
    void backupToCloud(T entity);// ← 只有部分實作需要
}

// ✅ 符合 ISP：拆成專注的小介面
public interface CrudRepository<T> { T save(T e); List<T> findAll(); }
public interface EmailNotifier<T> { void sendEmail(T e); }
public interface PdfExporter<T> { void exportToPdf(T e); }

// 需要什麼就 implements 什麼
public class TaskRepo implements CrudRepository<Task>, EmailNotifier<Task> { }
public class LogRepo implements CrudRepository<Log> { }  // 不需要 Email

D — 依賴反轉原則（Dependency Inversion）

定義：高層模組不該依賴低層模組，兩者都應依賴抽象（interface）。

你已經在用的地方：整個 Spring DI 就是 DIP 的實現。Controller 不依賴具體的 Service 實作，只依賴 Service interface（或至少依賴型別而非具體 class）。

// ❌ 違反 DIP：Controller 直接依賴具體實作
@RestController
public class UserController {
    private final UserServiceImpl service;  // 依賴具體 class
    // ↑ 換實作就要改 Controller
}

// ✅ 符合 DIP：Controller 依賴抽象（interface 或型別）
@RestController
public class UserController {
    private final UserService service;  // 依賴 interface
    // ↑ Spring 注入任何實作，Controller 不需修改
}

// 💡 這就是 T05 學的「針對介面寫程式」
// Spring 的 IoC 容器本身就是 DIP 的超級實現

Part B：常見 Design Patterns（在 Spring Boot 中的應用）

1. Singleton（單例模式）

意圖：確保一個 class 只有一個實例，並提供全域存取點。

Spring 中的應用：所有 @Service、@Repository、@Controller 的預設 scope 就是 singleton。你在 P06 學過——Spring 的 ApplicationContext 保證每個 Bean 只有一個實例。

// 傳統手寫 Singleton（你不需要這樣做了）
public class ManualSingleton {
    private static ManualSingleton instance;
    private ManualSingleton() { }
    public static ManualSingleton getInstance() { ... }
}

// Spring 的 Singleton（自動）
@Service  // ← Spring 自動保證只有一個實例，不需要任何手寫程式碼
public class MyService { }

2. Factory Method（工廠方法）

意圖：將物件的建立邏輯封裝起來，client 不用知道具體的建立細節。

Spring 中的應用：BeanFactory、ApplicationContext.getBean()、以及你 P10 中 CategoryService.getOrCreate()。

// Spring 的 BeanFactory 就是一個巨大的工廠
var ctx = SpringApplication.run(DemoApplication.class, args);
var service = ctx.getBean(UserService.class);  // 工廠幫你建立

// 你自己寫的工廠方法（P11）
public Category getOrCreate(String name) {
    return categoryRepo.findByName(name)
        .orElseGet(() -> categoryRepo.save(new Category(name, toSlug(name))));
    // ↑ 找不到就建立——封裝了「如何建立 Category」的邏輯
}

3. Proxy（代理模式）

意圖：提供一個代理物件來控制對原始物件的存取。

Spring 中的應用：AOP（@Transactional、@Async、@Cacheable）——這是你在 T06 和 T08 學過的核心機制。Spring 為你的 Service 建立一個 proxy，在方法呼叫前後插入橫切邏輯。

// 當你寫：
@Service
public class UserService {
    @Transactional
    public void transfer() { ... }
}

// Spring 實際上建立了：
// UserServiceProxy extends UserService {
//     public void transfer() {
//         tx.begin();           ← proxy 加入的交易管理
//         super.transfer();     ← 你的原始邏輯
//         tx.commit();          ← proxy 加入的交易管理
//     }
// }

// 這就是為什麼 T08 自呼叫 @Transactional 會失效——
// this.transfer() 跳過了 proxy

4. Template Method（模板方法）

意圖：定義演算法骨架，將步驟延遲到子類別實作。

Spring 中的應用：JdbcTemplate、RestTemplate、JpaRepository 的實作。

// JdbcTemplate 幫你處理：連線管理、關閉資源、例外轉譯
// 你只需要提供 SQL 和參數
@Autowired JdbcTemplate jdbc;
List<User> users = jdbc.query("SELECT * FROM users", rowMapper);
// ↑ Template 管理了 boilerplate，你只填變化部分

// P02 的 BaseEntity 也是一種 Template Method
public abstract class BaseEntity {
    public final String getEntityInfo() {  // ← final：骨架固定
        return "[" + getTypeLabel() + "] " + id;  // getTypeLabel() 由子類別填
    }
    public abstract String getTypeLabel(); // ← 子類別實作
}

5. Strategy（策略模式）

意圖：定義一系列可互換的演算法，讓 client 可以在執行期選擇。

Spring 中的應用：@Qualifier 切換實作（P05、T05）、Spring Security 的 PasswordEncoder。

// 同一介面，多種策略
public interface PaymentStrategy {
    void pay(double amount);
}

@Service("creditCard")  // 策略 A
public class CreditCardPayment implements PaymentStrategy { ... }

@Service("paypal")      // 策略 B
public class PayPalPayment implements PaymentStrategy { ... }

// 執行期選擇策略
@RestController
public class CheckoutController {
    private final Map<String, PaymentStrategy> strategies;  // Spring 自動注入所有實作

    public CheckoutController(Map<String, PaymentStrategy> strategies) {
        this.strategies = strategies;
    }

    @PostMapping("/checkout")
    public void checkout(@RequestParam String method, @RequestParam double amount) {
        strategies.get(method).pay(amount);  // 執行期選擇策略
    }
}

// 💡 這就是 P05 中 @Qualifier("fileRepo") 的本質——策略模式的 Spring 實現

6. Observer（觀察者模式）

意圖：當一個物件狀態改變時，自動通知所有依賴它的物件。

Spring 中的應用：Spring Events（ApplicationEvent + @EventListener）。

// 定義事件
public record PostPublishedEvent(Long postId, String title) { }

// 發佈者（Publisher）
@Service
public class PostService {
    private final ApplicationEventPublisher publisher;

    public void publish(Long postId) {
        // ... 發佈文章 ...
        publisher.publishEvent(new PostPublishedEvent(postId, post.getTitle()));
        // ↑ 不須知道誰在聽
    }
}

// 觀察者 1：發送通知
@Component
public class NotificationListener {
    @EventListener
    public void onPostPublished(PostPublishedEvent event) {
        System.out.println("🔔 新文章發佈通知：" + event.title());
    }
}

// 觀察者 2：更新搜尋索引（未來新增，完全不影響 PostService）
@Component
public class SearchIndexListener {
    @EventListener
    public void onPostPublished(PostPublishedEvent event) {
        System.out.println("📇 更新搜尋索引：" + event.postId());
    }
}

7. Builder（建造者模式）

意圖：將複雜物件的建構過程分步進行，避免建構子參數過多。

Spring 中的應用：UriComponentsBuilder、Spring Security 的 HttpSecurity 設定鏈、PageRequest.of()。

// Spring Security 的流暢 API 就是 Builder 模式（P10）
http
    .csrf(csrf -> csrf.disable())
    .sessionManagement(sm -> sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
    .authorizeHttpRequests(auth -> auth.requestMatchers("/api/**").permitAll());

// PageRequest（P11）
PageRequest.of(0, 20, Sort.by("createdAt").descending());

// UriComponentsBuilder
var uri = UriComponentsBuilder.fromPath("/api/posts/{id}")
    .queryParam("lang", "zh")
    .buildAndExpand(postId);

8. Decorator（裝飾者模式）

意圖：動態地為物件添加額外職責，而不影響其他物件。

Spring 中的應用：BeanPostProcessor（你在 P06 寫過！）就是 Decorator——它可以在每個 Bean 初始化前後包裹額外行為。

// P06 中你寫的 LifecycleLogger 就是 Decorator 的體現
@Component
public class LifecycleLogger implements BeanPostProcessor {
    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        // 為每個 Bean 裝飾上額外的 logging 行為
        if (beanName.contains("Service")) {
            System.out.println("Bean '" + beanName + "' 已就緒");
        }
        return bean;  // 可以回傳一個包裝過的 proxy！
    }
}

模式速查：面試一句話總結

模式一句話Spring Boot 中的實例
Singleton全應用只有一個實例@Service 的預設 scope
Factory Method封裝物件建立邏輯ApplicationContext.getBean()
Proxy代理控制對原始物件的存取@Transactional AOP 代理
Template Method固定骨架，子類別填細節JdbcTemplate、RestTemplate
Strategy可互換的演算法家族@Qualifier 切換實作
Observer狀態改變時自動通知@EventListener + ApplicationEvent
Builder分步建構複雜物件HttpSecurity 設定鏈

自我檢驗 🧠

Q1: SOLID 中哪一個原則直接對應到 Spring 的 DI 容器？為什麼？

答案

D — 依賴反轉原則（DIP）。高層模組（Controller）不依賴低層模組（具體 Service 實作），兩者都依賴抽象（Service interface）。Spring 的 IoC 容器負責在執行期注入具體實作，讓高層模組完全不需要知道低層的細節。

Q2: @Transactional 背後用了哪個設計模式？為什麼自呼叫會失效？

答案

Proxy 模式。Spring 為你的 Service 建立一個 proxy 物件，在方法前後加入 begin/commit/rollback。外部呼叫透過 proxy → 攔截器觸發。內部 this.method() 繞過 proxy 直接呼叫原始物件 → 攔截器不觸發。

Q3: 用 @Qualifier 切換不同 Repository 實作是用了哪個模式？和 OCP（開閉原則）有何關聯？

答案

Strategy 模式。每個 Repository 實作是一種策略。和 OCP 的關聯：當你需要新的持久化策略（例如從 InMemory 換成 Redis），只需新增一個 @Repository class，不需修改 Service 或 Controller——這正是「對擴展開放、對修改關閉」。
