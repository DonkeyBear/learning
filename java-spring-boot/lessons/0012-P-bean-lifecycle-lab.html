Bean 生命週期觀察實驗

這堂課你會用 log 和斷點親眼觀察 T06 學到的生命週期階段。在 P05 的 demo 專案中加入診斷程式碼，實際看到 Bean 何時被建立、初始化、注入、和銷毀。

🎯 這堂課的目標 用實驗驗證生命週期順序（建構子 → @PostConstruct → 就緒 → @PreDestroy）、觀察 singleton vs prototype 的行為差異、重現「singleton 注入 prototype」的陷阱。

Step 1：建立生命週期觀察用的 Bean

在 demo 專案中建立 lifecycle/LifecycleDemo.java：

package com.example.demo.lifecycle;

import jakarta.annotation.PostConstruct;
import jakarta.annotation.PreDestroy;
import org.springframework.stereotype.Component;

@Component
public class LifecycleDemo {

    public LifecycleDemo() {
        System.out.println("🏗️  [1] LifecycleDemo 建構子執行");
    }

    @PostConstruct
    public void init() {
        System.out.println("🔧 [3] LifecycleDemo @PostConstruct 執行");
    }

    @PreDestroy
    public void cleanup() {
        System.out.println("🧹 LifecycleDemo @PreDestroy 執行");
    }

    // 注意：這個方法由 TaskService 呼叫，我們用它來追蹤注入順序
    public void selfCheck() {
        System.out.println("✅ LifecycleDemo 已經就緒");
    }
}

Step 2：加入 BeanPostProcessor 觀察

建立 lifecycle/LifecycleLogger.java：

package com.example.demo.lifecycle;

import org.springframework.beans.factory.config.BeanPostProcessor;
import org.springframework.stereotype.Component;

@Component
public class LifecycleLogger implements BeanPostProcessor {

    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        if (beanName.contains("lifecycle") || beanName.contains("Lifecycle")) {
            System.out.println("  ↳ [2] BeanPostProcessor: BEFORE init — " + beanName);
        }
        return bean;
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        if (beanName.contains("lifecycle") || beanName.contains("Lifecycle")) {
            System.out.println("  ↳ [4] BeanPostProcessor: AFTER init — " + beanName);
        }
        return bean;
    }
}

Step 3：把 LifecycleDemo 注入到 TaskService

修改 TaskServiceImpl.java，注入 LifecycleDemo 並在 Service 建構子中紀錄順序：

// 在 TaskServiceImpl 中新增：
private final LifecycleDemo lifecycleDemo;

public TaskServiceImpl(TaskRepository taskRepository, LifecycleDemo lifecycleDemo) {
    this.taskRepository = taskRepository;
    this.lifecycleDemo = lifecycleDemo;
    System.out.println("  🏗️ TaskServiceImpl 建構子執行（LifecycleDemo 已被注入）");
    lifecycleDemo.selfCheck();  // 確認依賴已就緒
}

啟動應用，觀察 Console 輸出順序：

🏗️  [1] LifecycleDemo 建構子執行
  ↳ [2] BeanPostProcessor: BEFORE init — lifecycleDemo
🔧 [3] LifecycleDemo @PostConstruct 執行
  ↳ [4] BeanPostProcessor: AFTER init — lifecycleDemo
  🏗️ TaskServiceImpl 建構子執行（LifecycleDemo 已被注入）
✅ LifecycleDemo 已經就緒
🔍 觀察重點 BeanPostProcessor 的 before/after 分別在 @PostConstruct 前後執行。TaskServiceImpl 的建構子在 @PostConstruct 之後才被呼叫——因為 Spring 必須先建好 LifecycleDemo 才能注入它。這證明了 T06 的順序：建構子 → before → @PostConstruct → after → 才能被注入到其他 Bean。

Step 4：實驗 Singleton vs Prototype Scope

建立 lifecycle/ScopeDemo.java 來對比：

package com.example.demo.lifecycle;

import jakarta.annotation.PostConstruct;
import jakarta.annotation.PreDestroy;
import org.springframework.context.annotation.Scope;
import org.springframework.stereotype.Component;

@Component
@Scope("singleton")  // 預設值，明確寫出來做對比
public class ScopeDemo {
    private static int singletonCount = 0;
    private final int instanceId;

    public ScopeDemo() {
        instanceId = ++singletonCount;
        System.out.println("🟢 [Singleton] ScopeDemo #" + instanceId + " 建構");
    }

    @PostConstruct
    public void init() {
        System.out.println("🟢 [Singleton] ScopeDemo #" + instanceId + " @PostConstruct");
    }

    @PreDestroy
    public void cleanup() {
        System.out.println("🟢 [Singleton] ScopeDemo #" + instanceId + " @PreDestroy");
    }
}

package com.example.demo.lifecycle;

import jakarta.annotation.PostConstruct;
import jakarta.annotation.PreDestroy;
import org.springframework.context.annotation.Scope;
import org.springframework.stereotype.Component;

@Component
@Scope("prototype")
public class PrototypeDemo {
    private static int protoCount = 0;
    private final int instanceId;

    public PrototypeDemo() {
        instanceId = ++protoCount;
        System.out.println("🟡 [Prototype] PrototypeDemo #" + instanceId + " 建構");
    }

    @PostConstruct
    public void init() {
        System.out.println("🟡 [Prototype] PrototypeDemo #" + instanceId + " @PostConstruct");
    }

    // ⚠️ prototype Bean 的 @PreDestroy 不會被 Spring 呼叫！
    @PreDestroy
    public void cleanup() {
        System.out.println("🟡 [Prototype] PrototypeDemo #" + instanceId + " @PreDestroy（不會出現！）");
    }
}

Step 5：建立 Controller 來觸發 prototype

singleton Bean 在啟動時就建立一次。prototype Bean 需要主動觸發。建立 controller/ScopeTestController.java：

package com.example.demo.controller;

import com.example.demo.lifecycle.PrototypeDemo;
import org.springframework.beans.factory.ObjectProvider;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@RequestMapping("/api/scope")
public class ScopeTestController {
    private final PrototypeDemo singletonPrototype;   // ← 陷阱！
    private final ObjectProvider<PrototypeDemo> provider;  // ← 正確做法

    public ScopeTestController(
        PrototypeDemo singletonPrototype,
        ObjectProvider<PrototypeDemo> provider
    ) {
        this.singletonPrototype = singletonPrototype;
        this.provider = provider;
    }

    @GetMapping("/trap")
    public String trapTest() {
        // 連續呼叫兩次，會是同一個 instance 還是兩個？
        return "singleton 中注入的 prototype: " + singletonPrototype.hashCode();
    }

    @GetMapping("/correct")
    public String correctTest() {
        // 用 ObjectProvider 每次都取新的
        var p1 = provider.getObject();
        var p2 = provider.getObject();
        return "p1: " + p1.hashCode() + " | p2: " + p2.hashCode()
            + " | 相同? " + (p1 == p2);
    }
}

啟動後，觀察 Console：

  • singleton ScopeDemo 在啟動時就被建立（一行）
  • prototype PrototypeDemo 因為被注入到 Controller（singleton），所以啟動時也被建立一行——但只有一行！

然後用瀏覽器連續訪問兩次 /api/scope/trap，hashCode 相同——證明 prototype 只被建立了一次。

再訪問 /api/scope/correct，兩個 hashCode 不同——ObjectProvider 每次 getObject() 都建立新實例。

🔥 親眼見證：singleton 注入 prototype 陷阱 你在 ScopeTestController 中直接注入的 PrototypeDemo——因為 Controller 是 singleton，只在啟動時建立一次，所以 PrototypeDemo 也只被注入一次。之後每次請求用的都是同一個 instance。這就是 T06 說的面試陷阱題。

解決方案 ObjectProvider<T> 每次呼叫 getObject() 都會從容器取得新的 prototype 實例。

Step 6：觀察 @PreDestroy

優雅關閉 Spring Boot 應用（按 IntelliJ 的紅色 ■ 停止按鈕，或 Ctrl+F2）。觀察 Console 最後幾行輸出：

🧹 LifecycleDemo @PreDestroy 執行
🟢 [Singleton] ScopeDemo #1 @PreDestroy
（PrototypeDemo 的 @PreDestroy 不會出現！）

驗證：prototype Bean 的 @PreDestroy 不會被呼叫。

課後檢查清單

  • 啟動 log 中看到建構子 → BeanPostProcessor:before → @PostConstruct → BeanPostProcessor:after 的正確順序
  • singleton Bean 只在啟動時建立一次
  • prototype Bean 直接注入到 singleton 時也只建立一次（陷阱確認）
  • ObjectProvider.getObject() 每次回傳新的 prototype 實例
  • 優雅關閉時 singleton 的 @PreDestroy 被觸發
  • prototype 的 @PreDestroy 不會被觸發

📌 實驗結論

生命週期順序建構子 → BeanPostProcessor:before → @PostConstruct → BeanPostProcessor:after → 就緒
Singleton 特徵啟動時建立一次、@PreDestroy 正常觸發
Prototype 特徵每次 getObject() 建立新實例、@PreDestroy 不觸發
注入陷阱解法ObjectProvider<T> 或 @Lookup 方法注入
