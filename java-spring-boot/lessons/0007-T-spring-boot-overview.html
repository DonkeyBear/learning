Spring Boot 概論：從 FastAPI 到 Spring Boot 的思維轉換

你寫過 FastAPI 後端。你知道什麼是路由、什麼是 request body、什麼是 middleware。現在讓我們把這些概念映射到 Spring Boot，並理解 Spring Boot 底層最核心的設計理念——IoC 容器。

1. Spring Boot 是什麼？

一句話：Spring Boot 是 Java 生態系中建立後端應用的事實標準框架。它幫你處理了 Java web 開發中 90% 的 boilerplate（伺服器啟動、設定載入、依賴管理），讓你專注寫業務邏輯。

但 Spring Boot 和 FastAPI 有一個根本的差異——不是語法層面，而是程式執行的主控權在誰手上。

⚠️ 最重要的心態轉換 在 FastAPI 中，你建立 app、你註冊路由、你控制啟動。你是程式的主控者。

在 Spring Boot 中，框架啟動一個巨大的 IoC 容器，掃描你的程式碼，自動建立你宣告的物件，並把它們注入到需要的地方。框架是主控者，你只需要「宣告你需要什麼」。

這個反轉稱為 Inversion of Control（IoC，控制反轉）。這是整個 Spring 生態系的基石，也是下一堂課（T05）的主題。

2. 兩個世界的對照：同一個 API，兩種寫法

假設你要做一個「查詢用戶」的 GET endpoint：

FastAPI 寫法（你熟悉的）

# main.py
from fastapi import FastAPI, Depends

app = FastAPI()                    # 你手動建立 app

class UserService:
    def get_user(self, user_id: int):
        return {"id": user_id, "name": "Alice"}

@app.get("/users/{user_id}")       # 你手動註冊路由
def get_user(user_id: int, svc: UserService = Depends()):
    return svc.get_user(user_id)

# 啟動：uvicorn main:app

Spring Boot 寫法（即將學的）

// UserController.java
@RestController                    // ← 告訴 Spring：「這是一個 Controller」
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {  // ← Spring 自動傳入！
        this.userService = userService;
    }

    @GetMapping("/users/{userId}")  // ← 告訴 Spring：「這是一個 GET 路由」
    public UserDto getUser(@PathVariable Long userId) {
        return userService.getUser(userId);
    }
}

// UserService.java
@Service                           // ← 告訴 Spring：「這是一個 Service bean」
public class UserService {
    public UserDto getUser(Long userId) {
        return new UserDto(userId, "Alice");
    }
}

// 啟動：main 方法一行 run()，Spring 自動掃描並組裝一切

核心差異：

  • FastAPI 的 @app.get() 是你自己建立 app 實例並手動加路由
  • Spring Boot 的 @GetMapping 是宣告意圖，框架自動發現並註冊
  • FastAPI 的 Depends() 是你手動宣告依賴
  • Spring Boot 的建構子參數是框架自動解析並注入

3. Spring Boot 的三層架構

Spring Boot 專案幾乎都遵循同一套分層模式。你在 P02 練習的目錄結構就是它的雛形：

src/main/java/com/example/demo/
├── DemoApplication.java          ← Spring Boot 進入點
├── controller/
│   └── UserController.java       ← 接收 HTTP 請求，回傳回應
├── service/
│   ├── UserService.java          ← 介面（合約）
│   └── UserServiceImpl.java      ← 實作（業務邏輯）
├── repository/
│   └── UserRepository.java       ← 資料庫存取（JPA）
├── model/
│   ├── User.java                 ← Entity（對應資料表）
│   └── UserDto.java              ← DTO（傳輸物件）
└── config/
    └── SecurityConfig.java       ← 設定類別

🟢 Controller 層

職責：接收 HTTP 請求、參數驗證、回傳 HTTP 回應。不寫業務邏輯。

對應 FastAPI 的路由函式

Annotation: @RestController
@GetMapping / @PostMapping

🟡 Service 層

職責：業務邏輯、交易管理、協調多個 Repository。不知道 HTTP 的存在。

對應 FastAPI 中被 Depends() 注入的類別

Annotation: @Service

🔵 Repository 層

職責：資料庫 CRUD。Spring Data JPA 會幫你自動產生實作。

對應 FastAPI 的 SQLAlchemy session / ORM

Annotation: @Repository 或繼承 JpaRepository

🟣 Model / DTO

職責：定義資料形狀。Entity 對應資料表，DTO 對外傳輸。

對應 FastAPI 的 Pydantic models

Annotation: @Entity / Java record

🎯 關鍵規則：依賴方向 Controller → Service → Repository（單向！）
Controller 可以呼叫 Service，Service 可以呼叫 Repository。但 Repository 永遠不該知道 Service 的存在，Service 永遠不該知道 Controller 的存在。這確保了業務邏輯獨立於 HTTP 層，可以單獨測試。

4. Annotation 驅動：宣告取代命令

Spring Boot 大量使用 annotation（就是 @ 開頭的東西）來讓框架「發現」你的程式碼。你可以把它們想像成 FastAPI 的 decorator，但作用更深層：

Spring Boot Annotation作用FastAPI 對應
@SpringBootApplication標記啟動類別，觸發自動掃描無（你手動建立 FastAPI()）
@RestController標記 HTTP Controller，自動註冊路由無（你手動 @app.get()）
@GetMapping("/path")註冊 GET 路由@app.get("/path")
@PostMapping("/path")註冊 POST 路由@app.post("/path")
@Service標記 Service，納入 IoC 容器管理無（你手動 new 或 Depends）
@Repository標記資料存取層無（你手動管理 session）
@Autowired要求框架注入依賴Depends()

5. 啟動流程：Spring Boot 在背後做了什麼

當你執行 SpringApplication.run() 時，Spring Boot 依序做以下事情：

  1. 掃描：從 @SpringBootApplication 所在的 package 開始，往下掃描所有 class
  2. 註冊 Bean：找到所有帶 @Component / @Service / @Repository / @Controller 的 class，把它們實例化並放入 IoC 容器
  3. 注入依賴：檢查每個 Bean 的建構子需要什麼參數，從容器中找出對應的 Bean 傳入（這就是 DI——Dependency Injection）
  4. 自動配置：根據 classpath 中的 jar 和你的設定檔（application.properties），自動配置資料庫連線、內嵌伺服器等
  5. 啟動內嵌 Tomcat：Spring Boot 內建一個 Tomcat 伺服器，不需要另外安裝
  6. 就緒：應用程式可以接收 HTTP 請求
💡 對比 FastAPI 的啟動 FastAPI 的啟動：你寫好 Python 檔 → uvicorn main:app → 就緒。簡單直接。
Spring Boot 的啟動：多了一個 IoC 容器組裝的步驟。這一步在開發時感覺多餘，但在大型專案中是 Spring 最大的優勢——你不需要手動管理物件之間複雜的依賴關係，框架幫你處理。
P04 實作時你會親眼看到這個過程。

6. Convention over Configuration

Spring Boot 的哲學是「約定大於配置」——只要你照著慣例放檔案、命名，框架就自動幫你處理好。不需要寫一大串 XML 或 JSON 設定檔。

舉例：

  • 你把 Controller 放在 controller/ package 下 → Spring 自動發現
  • 你把 application.properties 放在 src/main/resources/ → Spring 自動載入
  • 你定義一個 JpaRepository interface → Spring Data 自動產生 SQL

自我檢驗 🧠

Q1: Spring Boot 和 FastAPI 最根本的「主控權」差異是什麼？

點擊展開答案

在 FastAPI 中，你是主控者——你建立 app、註冊路由、管理依賴。在 Spring Boot 中，框架是主控者——它啟動 IoC 容器，掃描你的 class，自動建立並注入物件。你只需要用 annotation 宣告「我需要什麼」。這個反轉就是 IoC（控制反轉）。

Q2: Spring Boot 的三層架構中，各層的 annotation 和職責是什麼？

點擊展開答案

Controller（@RestController）：接收 HTTP 請求、回傳回應，不寫業務邏輯。Service（@Service）：業務邏輯、交易管理，不知道 HTTP 的存在。Repository（@Repository 或 JpaRepository）：資料庫存取，Spring Data 自動產生實作。依賴方向是單向的：Controller → Service → Repository。

Q3: @Service 和 Depends() 的核心差異是什麼？

點擊展開答案

Depends() 是 FastAPI 中手動宣告依賴的方式——你在每個路由函式的參數中明確寫出需要注入什麼。Spring 的 @Service + 建構子注入則是自動解析——你只在建構子中宣告參數型別，框架自動從 IoC 容器中找到對應的 Bean 傳入。Spring 的方式在大型專案中減少了大量重複的依賴管理程式碼。

📌 關鍵 Takeaway

  1. IoC 容器是 Spring 的大腦——它管理所有物件的生命週期和依賴關係
  2. Annotation 是你的宣告——@Service 說「這是一個 Bean」，@GetMapping 說「這是一條路由」
  3. 三層架構是鐵律——Controller → Service → Repository，單向依賴
  4. Spring Boot = Spring + 自動配置 + 內嵌伺服器——讓你一行 run() 就能啟動整個應用
  5. Convention over Configuration——照慣例放，框架幫你處理；不照慣例才需要寫設定
