Part A：Race Condition 的六種解法

1. 問題回顧：race condition 的本質

任何 讀-改-寫（read-modify-write）在沒有同步保護的情況下，都是 race condition 的溫床：

// ❌ 經典 race condition：++ 不是原子操作
public class UnsafeCounter {
    private int count = 0;
    public void increment() { count++; }
    // count++ 實際上是三步：讀取值 → 加 1 → 寫回
    // 兩個 thread 可能同時讀到 5，都寫回 6 → 遺失一次遞增
}

解法 1：synchronized — Java 最基礎的鎖

public class SyncCounter {
    private int count = 0;

    // 方法層級：鎖住整個 this 物件
    public synchronized void increment() {
        count++;
    }

    // 區塊層級：只鎖關鍵區段（更精細、效能更好）
    public void incrementBetter() {
        // ... 可以在這裡做不用鎖的事情 ...
        synchronized (this) {
            count++;
        }
    }

    // 靜態方法層級：鎖住 Class 物件（全 JVM 一把鎖）
    public static synchronized void staticMethod() { }
}

synchronized 的底層原理（面試加分）

每個 Java 物件都有一個 monitor 鎖。synchronized 進入時獲取 monitor（monitorenter 指令）、離開時釋放（monitorexit）。Java 6 之後做了大量優化：偏向鎖 → 輕量級鎖 → 重量級鎖的逐步升級，稱為鎖膨脹（lock inflation）。大部分情況下的 synchronized 效能已經不輸 ReentrantLock。

解法 2：ReentrantLock — 更靈活的鎖

import java.util.concurrent.locks.ReentrantLock;

public class LockCounter {
    private int count = 0;
    private final ReentrantLock lock = new ReentrantLock();

    public void increment() {
        lock.lock();
        try {
            count++;
        } finally {
            lock.unlock();  // ← 必須在 finally 中解鎖！
        }
    }

    // synchronized 沒有的功能：
    public void tryIncrement() {
        if (lock.tryLock()) {  // ← 嘗試獲取鎖，拿不到就放棄（不阻塞）
            try { count++; }
            finally { lock.unlock(); }
        }
    }

    public void safeIncrement() throws InterruptedException {
        // 等待最多 1 秒，拿不到就放棄
        if (lock.tryLock(1, java.util.concurrent.TimeUnit.SECONDS)) {
            try { count++; }
            finally { lock.unlock(); }
        }
    }
}
特性synchronizedReentrantLock
語法關鍵字，Java 內建API，需要 finally 解鎖
tryLock（非阻塞）❌✅
公平鎖❌（非公平）✅ new ReentrantLock(true)
可中斷等待❌✅ lockInterruptibly()
Condition（多條件）❌（只有 wait/notify）✅ lock.newCondition()
🔥 面試題：synchronized vs ReentrantLock 何時用哪個？ 一般場景用 synchronized——語法簡單、不易出錯、JVM 自動優化。需要 tryLock、公平鎖、或 Condition 時用 ReentrantLock。Spring 內部兩者都有使用，但對開發者暴露的 API（如 @Transactional）都是 synchronized-free 的。

解法 3：Atomic 類別 — 無鎖並行（CAS）

import java.util.concurrent.atomic.*;

public class AtomicCounter {
    private final AtomicInteger count = new AtomicInteger(0);
    private final AtomicLong seq = new AtomicLong(0);
    private final AtomicReference<String> lastUser = new AtomicReference<>("none");
    private final AtomicBoolean initialized = new AtomicBoolean(false);

    // 原子遞增（底層用 CAS：Compare-And-Swap）
    public int next() { return count.incrementAndGet(); }

    // CAS 更新：只有目前值是 5 時才改成 10
    public boolean tryUpdate() { return count.compareAndSet(5, 10); }

    // 只初始化一次（類似 singleton）
    public void initOnce() {
        if (initialized.compareAndSet(false, true)) {
            System.out.println("執行一次性初始化");
        }
    }

    // AtomicReference 的 CAS 更新
    public void updateUser(String newUser) {
        lastUser.updateAndGet(current ->
            current.equals("none") ? newUser : current);
    }
}

CAS（Compare-And-Swap）原理

CAS 是 CPU 指令層級的原子操作：比較記憶體中的值是否等於預期值，如果是就寫入新值，如果不是就什麼都不做。這是無鎖並行的基石——沒有鎖的 overhead，沒有 deadlock 風險。缺點：① ABA 問題（用 AtomicStampedReference 解決）② 高競爭下 CAS 自旋次數過多。

解法 4：volatile — 可見性保證（不是原子性！）

public class VolatileExample {
    // volatile 保證：① 寫入立即對所有 thread 可見 ② 禁止指令重排
    private volatile boolean running = true;

    // ✅ volatile 適合的場景：單一 thread 寫、多 thread 讀的 flag
    public void stop() {
        running = false;  // Thread-1 寫入
    }
    public void work() {
        while (running) {  // Thread-2 讀取——保證看到最新值
            // do work
        }
    }

    // ❌ volatile 不夠的場景：複合操作
    // private volatile int count = 0;
    // count++;  ← 仍然是 race condition！volatile 只保證可見性，不保證原子性
}
⚠️ 面試陷阱 volatile 保證可見性（一個 thread 改了，其他 thread 看得到）但不保證原子性（count++ 仍然不安全）。面試官常問「volatile 能解決 race condition 嗎？」→ 不能，只能解決可見性問題。

解法 5：ThreadLocal — 每個 Thread 自己的副本

public class RequestContext {
    // 每個 thread 有自己的 requestId，完全隔離
    private static final ThreadLocal<String> requestId = new ThreadLocal<>();

    public static void setRequestId(String id) { requestId.set(id); }
    public static String getRequestId() { return requestId.get(); }
    public static void clear() { requestId.remove(); }  // ← 必須清理！避免記憶體洩漏
}

// Spring 的 RequestContextHolder 就是用 ThreadLocal
// 你在 Controller 中不需要傳遞 request，但框架知道目前是哪個 request

解法 6：不可變物件（Immutable）— 最根本的解法

// 不可變 = 絕對執行緒安全，因為沒有寫操作
public final class ImmutableUser {  // ← final class
    private final String name;      // ← final fields
    private final int age;

    public ImmutableUser(String name, int age) {
        this.name = name;
        this.age = age;             // 建構後永不改變
    }

    public String getName() { return name; }  // 只有 getter，沒有 setter
}

// Java String、Integer、LocalDateTime 都是不可變的
// record 也是不可變的！
public record UserDto(String name, int age) { }
解法適用場景效能複雜度
synchronized簡單互斥、少量競爭中低
ReentrantLock需要 tryLock、公平鎖中中
Atomic（CAS）單一變數原子操作高低
volatile讀多寫少的 flag最高低
ThreadLocalThread 範圍的隔離資料高中
Immutable不需要改的資料最高最低

Part B：Deadlock 死鎖

Q: 什麼是 Deadlock？

兩個（或更多）Thread 各自持有對方需要的鎖，互相等待，永遠無法繼續執行。

// 🔴 經典死鎖範例：兩個 Thread 交叉獲取兩把鎖
public class DeadlockDemo {
    private final Object lockA = new Object();
    private final Object lockB = new Object();

    public void method1() {
        synchronized (lockA) {                     // Thread-1 拿到 lockA
            sleep(100);
            synchronized (lockB) {                 // Thread-1 等待 lockB...
                System.out.println("method1");     // ← 永遠不會執行
            }
        }
    }

    public void method2() {
        synchronized (lockB) {                     // Thread-2 拿到 lockB
            sleep(100);
            synchronized (lockA) {                 // Thread-2 等待 lockA...
                System.out.println("method2");     // ← 永遠不會執行
            }
        }
    }
    // Thread-1: 持有 lockA，等待 lockB
    // Thread-2: 持有 lockB，等待 lockA
    // → 死鎖！兩者永遠無法繼續
}

死鎖的四個必要條件（面試必背）

1. 互斥（Mutual Exclusion）資源一次只能被一個 thread 持有
2. 持有並等待（Hold and Wait）持有資源的同時等待其他資源
3. 不可搶佔（No Preemption）資源不能被強行奪走，只能自願釋放
4. 循環等待（Circular Wait）形成封閉的資源依賴環
🔥 面試關鍵 打破任一條件即可預防死鎖。實務中最容易打破的是條件 4（循環等待）——只要你規定所有 thread 獲取鎖的順序一致（例如都先拿 lockA 再拿 lockB），就不會形成循環。這就是鎖定順序（Lock Ordering）策略。

死鎖的檢測與預防

預防策略 1：固定鎖定順序

// ✅ 所有人按相同順序獲取鎖
public void safeMethod(Account from, Account to) {
    // 用 id 決定鎖定順序，避免交叉
    var first = from.getId() < to.getId() ? from : to;
    var second = from.getId() < to.getId() ? to : from;

    synchronized (first) {
        synchronized (second) {
            // 轉帳邏輯
        }
    }
}

預防策略 2：tryLock + 逾時 + 放棄

public boolean safeTransfer(Account from, Account to, int amount) {
    var lock1 = from.getLock();
    var lock2 = to.getLock();

    while (true) {
        try {
            if (lock1.tryLock(100, TimeUnit.MILLISECONDS)) {
                try {
                    if (lock2.tryLock(100, TimeUnit.MILLISECONDS)) {
                        try {
                            from.debit(amount);
                            to.credit(amount);
                            return true;
                        } finally { lock2.unlock(); }
                    }
                } finally { lock1.unlock(); }
            }
        } catch (InterruptedException e) { Thread.currentThread().interrupt(); return false; }
        // 沒拿到兩把鎖，短暫等待後重試（避免 livelock）
        sleep(10);
    }
}

Livelock（活鎖）— 死鎖的狡猾表親

面試進階題：兩個 thread 不斷「禮讓」對方，都在動、但都沒進展。

// 活鎖：兩人不斷互相禮讓，誰都過不了門
public void passDoor(Person other) {
    while (true) {
        if (other.isInMyWay()) {
            Thread.yield();  // 「你先過」
            continue;
        }
        break;  // 終於過了
    }
}
// 兩個人同時 yield → 同時回來 → 同時再 yield → 無限循環

Part C：Spring Boot 中的並行工具

@Async — 讓方法非同步執行

@Configuration
@EnableAsync  // ← 啟用非同步
public class AsyncConfig { }

@Service
public class NotificationService {

    @Async  // ← 這個方法會在獨立 thread 中執行
    public void sendWelcomeEmail(String email) {
        // 發送 Email（不阻塞呼叫者）
        System.out.println("📧 發送中... " + Thread.currentThread().getName());
    }
}

// 呼叫時：
notificationService.sendWelcomeEmail("alice@example.com");
// 主 thread 立即繼續執行，不等待 Email 發完
⚠️ @Async 的陷阱 ① 自呼叫無效（和 @Transactional 一樣——繞過 proxy）。② 預設使用 SimpleAsyncTaskExecutor（每次新建 thread，不適合生產環境）——務必自訂 ThreadPoolTaskExecutor。③ 例外不會傳播到呼叫者，必須用 AsyncUncaughtExceptionHandler 處理。

自訂 Thread Pool

@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean("emailExecutor")
    public Executor emailExecutor() {
        var executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(2);        // 核心 thread 數
        executor.setMaxPoolSize(5);         // 最大 thread 數
        executor.setQueueCapacity(100);     // 等待 Queue 容量
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy()  // Queue 滿時由呼叫者 thread 執行
        );
        executor.setThreadNamePrefix("email-");
        executor.initialize();
        return executor;
    }
}

// 使用特定 thread pool：
@Async("emailExecutor")
public void sendEmail(String email) { ... }

並行工具三劍客：CountDownLatch / CyclicBarrier / Semaphore

// CountDownLatch：等待 N 個任務完成（一次性）
var latch = new CountDownLatch(3);
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        doWork();
        latch.countDown();  // 「我完成了」
    }).start();
}
latch.await();  // 等到 3 個 countDown 為止
System.out.println("全部完成");

// Semaphore：限制同時存取數量（連接池的本質）
var semaphore = new Semaphore(3);  // 最多 3 個 thread 同時執行
semaphore.acquire();  // 獲取許可（許可不足時阻塞）
try { doLimitedWork(); }
finally { semaphore.release(); }  // 歸還許可

// CyclicBarrier：等待所有 thread 到達同一點（可重複使用）
var barrier = new CyclicBarrier(3, () ->
    System.out.println("所有人都到了，開始下一階段！"));
// 三個 thread 各自工作到 barrier.await() → 全部到達後觸發 callback → 繼續

Part D：分散式環境中的並行挑戰

🚨 核心問題：為什麼 JVM 層的鎖在分散式環境中失效？ Part A/B/C 學的所有工具（synchronized、ReentrantLock、Atomic、volatile）都是單一 JVM 內部的鎖。當你把 blog-api 部署到 3 台伺服器，前面放一個 Load Balancer，每台伺服器有自己的 JVM、自己的鎖——伺服器 A 的 synchronized 完全管不到伺服器 B 的 thread。這就是分散式並行問題的本質。

D-1. 分散式鎖（Distributed Lock）

核心思想：把鎖從 JVM 記憶體移到一個所有伺服器都能存取的外部儲存上。最常見的實作是 Redis 和 ZooKeeper。

方案一：Redis + Redisson（最常見）

// build.gradle
implementation 'org.redisson:redisson-spring-boot-starter:3.34.0'

// application.properties
spring.redis.host=localhost
spring.redis.port=6379

// 使用 Redisson 分散式鎖
@Service
public class InventoryService {
    private final RedissonClient redisson;

    public InventoryService(RedissonClient redisson) {
        this.redisson = redisson;
    }

    public boolean deductStock(Long productId, int quantity) {
        RLock lock = redisson.getLock("product:lock:" + productId);
        try {
            // 嘗試獲取鎖：最多等 3 秒，鎖定後 10 秒自動釋放（防止 deadlock）
            if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
                var product = productRepo.findByIdWithPessimisticLock(productId)
                    .orElseThrow();
                product.decreaseStock(quantity);
                productRepo.save(product);
                return true;
            }
            return false;  // 拿不到鎖（庫存被搶完或系統繁忙）
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return false;
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
💡 Redisson 的 Watchdog 機制（面試加分） 你有注意到鎖定時間設為 10 秒嗎？如果業務邏輯實際花了 15 秒呢？Redisson 的 Watchdog 會自動續期——每 1/3 鎖定時間自動檢查，如果業務還在執行就自動延長鎖定時間。這解決了「鎖定時間設太短業務中斷、設太長 deadlock 風險高」的兩難。

方案二：ZooKeeper（強一致性場景）

// 使用 Curator（ZooKeeper 的 Java client）
// ZooKeeper 利用 ephemeral sequential node 實現鎖定：
// 1. 每個 client 在 /lock 下建立一個 ephemeral + sequential 節點
// 2. 序號最小的節點獲得鎖
// 3. 沒有拿到鎖的 client watch 前一個節點
// 4. 前一個節點刪除時，下一個自動獲得鎖（公平排隊）

// 優勢：強一致性（CP），不會有 Redis cluster 的腦裂問題
// 劣勢：效能比 Redis 低，運維較複雜
🔥 面試題：Redis vs ZooKeeper 分散式鎖 Redis（Redisson）：AP 模型（高可用），效能高，適合大多數場景。但有極小機率在 Redis sentinel 切換時兩個 client 同時拿到鎖（RedLock 演算法可緩解）。ZooKeeper：CP 模型（強一致），透過 ephemeral node + watch 機制實現公平排隊。適合對一致性要求極高的場景。中小型專案用 Redis Redisson 就夠了。

D-2. 分散式交易（Distributed Transaction）

當一個業務操作需要跨多個服務且必須全部成功或全部失敗：

// 😰 分散式交易場景：下訂單需要三個服務都成功
// OrderService  → 建立訂單（自己的 DB）
// InventoryService → 扣庫存（自己的 DB）
// PaymentService   → 扣款（外部金流）

// 如果扣款失敗，必須 rollback 訂單和庫存——但各自在不同的 DB！
// @Transactional 只管理一個 DataSource，跨服務無能為力

解法：Saga 模式（補償交易）

// Saga：每個步驟有對應的「反向操作」
// 訂單建立        ←→  訂單取消（補償）
// 庫存扣減        ←→  庫存歸還（補償）
// 付款扣款        ←→  退款（補償）

// 實作有兩種：
// 1. Choreography（編排式）：每個服務做完後發事件，下一個服務監聽
// 2. Orchestration（指揮式）：一個 Saga Orchestrator 指揮所有步驟

// Orchestrator 範例（簡化）：
@Service
public class OrderSagaOrchestrator {

    public void execute(CreateOrderRequest req) {
        try {
            var order = orderService.create(req);    // Step 1
            inventoryService.deduct(req.items());     // Step 2（失敗時補償 Step 1）
            paymentService.charge(req.payment());     // Step 3（失敗時補償 Step 1+2）
            orderService.confirm(order.getId());      // Step 4：全部成功
        } catch (Exception e) {
            compensate(req);  // 依序執行所有已成功步驟的補償操作
            throw e;
        }
    }
}
💡 面試加分：2PC vs Saga 2PC（Two-Phase Commit）：資料庫層的分散式交易協議（XA transaction）。強一致性，但效能差、blocking、單點故障風險高——現代微服務幾乎不用。Saga：應用層的分散式交易，透過補償操作實現最終一致性。Spring 生態系的 Saga 實作有 Axon Framework、Camunda、或自行用 MQ + 事件實現。

D-3. 冪等性（Idempotency）

在分散式環境中，同一個請求可能被重複執行——MQ 重送、使用者 double-click、Load Balancer 重試、網路 timeout 後 client 重發。

// ❌ 沒有冪等性保護：同一個請求執行兩次 = 扣兩次款
@PostMapping("/orders")
public ApiResponse<OrderResponse> create(@RequestBody CreateOrderRequest req) {
    return ApiResponse.ok(orderService.create(req));
}

// ✅ 加入冪等性保護：用 idempotency key
@PostMapping("/orders")
public ApiResponse<OrderResponse> create(
    @RequestHeader("Idempotency-Key") String idempotencyKey,  // client 端產生 UUID
    @RequestBody CreateOrderRequest req
) {
    // 檢查是否已經處理過這個 key
    var existing = orderRepo.findByIdempotencyKey(idempotencyKey);
    if (existing.isPresent()) {
        return ApiResponse.ok(existing.get());  // 回傳之前的結果，不重複執行
    }
    var order = orderService.create(req, idempotencyKey);
    return ApiResponse.ok(order);
}

// MQ Consumer 的冪等性：用 messageId 做去重
@RabbitListener(queues = "email-queue")
public void handleEmail(UserRegisteredEvent event) {
    if (processedMessageRepo.existsById(event.messageId())) {
        return;  // 已處理過，跳過
    }
    emailService.send(event.email());
    processedMessageRepo.save(new ProcessedMessage(event.messageId()));
}

D-4. Spring Boot 中的分散式鎖實戰

用 Spring AOP 包裝分散式鎖——讓業務程式碼完全不碰鎖的細節：

// 自訂 annotation
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface DistributedLock {
    String key();       // Redis key 的 SpEL 表達式，如 "'product:' + #productId"
    long waitTime() default 3;
    long leaseTime() default 10;
}

// AOP aspect
@Aspect
@Component
public class DistributedLockAspect {
    private final RedissonClient redisson;

    @Around("@annotation(lock)")
    public Object around(ProceedingJoinPoint joinPoint, DistributedLock lock) throws Throwable {
        String key = /* 解析 SpEL */;
        RLock rLock = redisson.getLock(key);
        if (rLock.tryLock(lock.waitTime(), lock.leaseTime(), TimeUnit.SECONDS)) {
            try { return joinPoint.proceed(); }
            finally {
                if (rLock.isHeldByCurrentThread()) rLock.unlock();
            }
        }
        throw new RuntimeException("無法獲取分散式鎖: " + key);
    }
}

// 使用——業務程式碼完全不知道鎖的存在！
@Service
public class InventoryService {
    @DistributedLock(key = "'product:' + #productId")
    public boolean deduct(Long productId, int qty) {
        // 純業務邏輯，鎖由 AOP 處理
    }
}

自我檢驗 🧠

Q1: synchronized 和 ReentrantLock 各有什麼對方沒有的功能？

答案

synchronized：語法簡單、JVM 自動解鎖（不會忘記）、偏向鎖優化。ReentrantLock：tryLock（非阻塞嘗試）、公平鎖（先到先得）、可中斷等待（lockInterruptibly）、Condition（多個等待條件）。一般場景用 synchronized，需要這些進階功能時用 ReentrantLock。

Q2: Deadlock 的四個必要條件是什麼？最實務上如何預防？

答案

① 互斥 ② 持有並等待 ③ 不可搶佔 ④ 循環等待。最實務的預防是打破條件 ④——固定鎖定順序（Lock Ordering）：所有 thread 按相同順序獲取鎖。例如轉帳時依帳號 ID 決定先鎖哪個帳戶。也可以用 tryLock + 逾時來打破條件 ②。

Q3: volatile 能解決 count++ 的 race condition 嗎？為什麼？

答案

不能。volatile 只保證可見性（一個 thread 改了後其他 thread 能看到），不保證原子性。count++ 是讀-改-寫三步驟，兩個 thread 仍然可以同時讀到相同值、各自加 1、寫回相同結果（遺失一次遞增）。volatile 適合單一 thread 寫入、多 thread 讀取的 flag 場景。

Q4: 為什麼 synchronized 在分散式環境（多台伺服器）中無法防止 race condition？用什麼替代？

答案

synchronized 只能鎖住同一個 JVM 內的 thread。當應用部署到多台伺服器時，每台有自己的 JVM 和獨立的鎖——伺服器 A 的 synchronized 管不到伺服器 B。替代方案：分散式鎖——把鎖移到所有伺服器共享的外部儲存。最常見是用 Redis + Redisson（高效能、Watchdog 自動續期），對強一致性要求高的場景用 ZooKeeper（ephemeral node + 公平排隊）。

Q5: Saga 模式和 2PC 有什麼不同？現代微服務為什麼偏好 Saga？

答案

2PC 是資料庫層的強一致性協議——coordinator 要求所有參與者 prepare，全部 OK 才 commit。缺點：blocking（等待最慢的參與者）、單點故障（coordinator 掛了全部卡住）、效能差。現代微服務幾乎不用。Saga 是應用層的最終一致性方案——每個步驟有對應的補償操作，失敗時反向執行已完成的步驟。非 blocking、無單點瓶頸、適合長時間跨服務業務流程。
