Bean 生命週期：從建立到銷毀的全流程

你在 P04 和 P05 中用 @Service、@Repository 建立了許多 Bean。但 Spring 是何時建立它們的？建立過程中經過哪些階段？你可以介入哪些階段來執行自訂邏輯？這堂課回答這些問題。

🎯 學完這堂課你要能回答 「請描述 Spring Bean 的生命週期」「@PostConstruct 和 @PreDestroy 何時被呼叫？」「singleton 和 prototype scope 的差異？」「BeanPostProcessor 是什麼？」——全是面試高頻題。

1. Bean 生命週期全景圖

一個 Spring Bean 從無到有再到消失，會經過以下階段：

1. 實例化（Instantiation）
   └→ new 出物件（呼叫建構子）
2. 填充屬性（Populate Properties）
   └→ 注入 @Autowired 的依賴、@Value 的值
3. BeanPostProcessor::postProcessBeforeInitialization
   └→ Spring 內部 hook（@PostConstruct 在此觸發）
4. 初始化（Initialization）
   └→ @PostConstruct 方法被呼叫
5. BeanPostProcessor::postProcessAfterInitialization
   └→ Spring 內部 hook（AOP proxy 在此建立！）
6. 就緒（Ready）
   └→ Bean 可以被使用了（注入到其他 Bean、處理請求）
7. 銷毀（Destruction）
   └→ 容器關閉時，@PreDestroy 方法被呼叫
🔥 面試關鍵點：AOP 代理的建立時機 Spring AOP（例如 @Transactional）的代理（proxy）是在 postProcessAfterInitialization 階段建立的。這表示：如果你在 @PostConstruct 中呼叫了一個標記 @Transactional 的方法，交易可能不會生效——因為代理還沒建立。這是面試中常見的陷阱題。

2. @PostConstruct 與 @PreDestroy

這是最常用的兩個生命週期 hook：

@Service
public class CacheService {
    private final Map<String, Object> cache = new HashMap<>();

    // 在所有依賴注入完成後、Bean 就緒前被呼叫
    @PostConstruct
    public void init() {
        System.out.println("📦 CacheService 初始化：預熱快取...");
        cache.put("app.status", "starting");
        // 適合做的事情：
        //   - 初始化連線池
        //   - 預載資料到快取
        //   - 驗證必要的設定值存在
        //   - 啟動背景排程
    }

    // 在容器關閉、Bean 銷毀前被呼叫
    @PreDestroy
    public void cleanup() {
        System.out.println("🧹 CacheService 清理：釋放資源...");
        cache.clear();
        // 適合做的事情：
        //   - 關閉資料庫連線
        //   - 釋放檔案控制代碼
        //   - 停止背景執行緒
        //   - 發送關閉通知
    }

    public void put(String key, Object value) {
        cache.put(key, value);
    }
}
⚠️ @PreDestroy 的限制 @PreDestroy 只在「正常關閉」時觸發（例如 ctx.close() 或 kill -15）。如果是 kill -9 強制終止，不會觸發。另外 prototype scope 的 Bean 不會觸發 @PreDestroy——Spring 不追蹤 prototype Bean 的完整生命週期。

對比：InitializingBean / DisposableBean 介面（舊式寫法）

// 舊式寫法——不推薦，因為和 Spring API 耦合
@Service
public class OldStyleService implements InitializingBean, DisposableBean {

    @Override
    public void afterPropertiesSet() {  // = @PostConstruct
        // 初始化邏輯
    }

    @Override
    public void destroy() {             // = @PreDestroy
        // 清理邏輯
    }
}

面試中建議說：偏好 @PostConstruct/@PreDestroy（JSR-250 標準註解），因為不和 Spring 框架耦合。

3. Bean Scope：控制生命週期的長短

@Scope 決定了 Spring 何時建立新的 Bean 實例。預設是 singleton。

Scope行為生命週期使用場景
singleton 整個容器只有一個實例 隨容器啟動建立，容器關閉時銷毀 預設。Service、Repository、Controller
prototype 每次注入/查詢都建立新實例 Spring 建立後就不再追蹤（@PreDestroy 不會被呼叫！） 每次請求需要獨立狀態的 Bean
request 每個 HTTP 請求一個實例 請求結束時銷毀 Web 應用中請求範圍的資料
session 每個 HTTP session 一個實例 Session 過期時銷毀 購物車、使用者偏好
application 整個 ServletContext 一個實例 隨應用關閉 與 singleton 類似但作用域不同 
@Service
@Scope("singleton")  // 預設值，通常不需要寫
public class DefaultService { }

@Service
@Scope("prototype")
public class PrototypeService {
    // 每次注入都會得到一個全新的實例
}
🔥 面試陷阱題：singleton 注入 prototype 如果你把一個 prototype Bean 注入到 singleton Bean 中，prototype Bean 仍然只會被建立一次。因為 singleton Bean 只會被建立一次、注入也只發生一次。解決方案：① 注入 ObjectProvider<T> ② 注入 ApplicationContext 手動 getBean ③ 使用 @Lookup 方法注入。

4. BeanPostProcessor：Spring 的「中介層」

BeanPostProcessor 是 Spring 內部最強大的擴展點。它可以在每個 Bean 初始化前後插入自訂邏輯：

@Component
public class CustomBeanPostProcessor implements BeanPostProcessor {

    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        // 在 @PostConstruct 之前被呼叫
        if (beanName.contains("Service")) {
            System.out.println("🔧 初始化前: " + beanName);
        }
        return bean;  // 可以回傳代理物件！
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        // 在 @PostConstruct 之後被呼叫
        // Spring AOP 就是在這裡建立 proxy 的！
        if (beanName.contains("Service")) {
            System.out.println("🔧 初始化後: " + beanName);
        }
        return bean;
    }
}

你極少需要自己寫 BeanPostProcessor——但你需要知道它的存在，因為 Spring 的許多核心功能（AOP、@Transactional、@Async）都是透過 BeanPostProcessor 實現的。

5. 實務中的生命週期應用場景

自我檢驗 🧠

Q1: 從建立到銷毀，Bean 生命週期包含哪些主要階段？

點擊展開答案

實例化（建構子）→ 填充屬性（注入依賴、@Value）→ BeanPostProcessor: before → 初始化（@PostConstruct）→ BeanPostProcessor: after（AOP proxy 在此建立）→ 就緒（可使用）→ 銷毀（@PreDestroy）。面試時至少要說出：建構子 → 注入 → @PostConstruct → 就緒 → @PreDestroy。

Q2: singleton Bean 注入 prototype Bean 會有什麼問題？如何解決？

點擊展開答案

prototype Bean 被注入到 singleton Bean 中時，只會在 singleton 建立時被注入一次——之後每次使用都是同一個實例，失去 prototype 的意義。解決方案：① 注入 ObjectProvider<T> 並每次呼叫 getObject() ② 使用 @Lookup 抽象方法 ③ 注入 ApplicationContext 手動 getBean（最不推薦）。

Q3: 為什麼在 @PostConstruct 中呼叫 @Transactional 方法可能不會生效？

點擊展開答案

因為 @Transactional 是透過 AOP 代理實現的，而 AOP 代理是在 postProcessAfterInitialization 階段建立的——這個階段在 @PostConstruct 之後。@PostConstruct 執行時，代理還沒建立，因此 @Transactional 的攔截器不會被觸發。解法：將需要交易的初始化邏輯移到 ApplicationRunner 或 @EventListener(ApplicationReadyEvent.class) 中。

📌 關鍵 Takeaway

  1. 生命週期順序：建構子 → 注入 → @PostConstruct → （AOP proxy）→ 就緒 → @PreDestroy
  2. @PostConstruct = 依賴注入完成後執行一次，適合初始化邏輯
  3. @PreDestroy = 容器關閉前執行，適合釋放資源（prototype 不會觸發）
  4. Singleton 是預設——大多數 Bean 只需要一個實例；prototype 每次都是新的
  5. AOP 代理在 @PostConstruct 之後才建立——初始化邏輯中不能用 @Transactional
  6. BeanPostProcessor 是 Spring 的萬能中介層，AOP、@Async 等核心功能都靠它
