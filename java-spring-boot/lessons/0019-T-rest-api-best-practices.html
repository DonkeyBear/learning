REST API 設計最佳實踐

你已經寫過 FastAPI 後端，對 REST 的基本概念不陌生。這堂課聚焦在 Spring Boot 生態中特有的設計慣例——全域例外處理、回應格式統一、分頁標準、Bean Validation——這些都是你在整合專案中會立刻用到的技能。

🎯 這堂課的目標 學會 Spring Boot 中打造「專業級 API」的關鍵技術：@ControllerAdvice 全域例外處理、ResponseEntity 回應控制、@Valid 請求驗證、Spring 分頁標準。這些也是面試中「你有沒有寫過生產環境 API」的判斷標準。

1. HTTP 狀態碼：說正確的語言

FastAPI 和 Spring Boot 對 HTTP 狀態碼的支援都很完善。重點不是「能不能設」，而是「有沒有養成習慣設對」：

2. 統一回應格式（Response Envelope）

實務上大多數 API 不會直接回傳裸 Entity，而是包一層固定的外殼：

// ❌ 裸回傳——每個 endpoint 格式可能不同
{"id": 1, "title": "Task 1"}

// ✅ 統一回傳格式——前端可以寫通用的錯誤處理
{
  "success": true,
  "data": {"id": 1, "title": "Task 1"},
  "timestamp": "2026-07-05T10:30:00"
}

// 錯誤時也長一樣
{
  "success": false,
  "error": {"code": "NOT_FOUND", "message": "找不到任務: 99"},
  "timestamp": "2026-07-05T10:30:00"
}

// 通用回應 wrapper（用 record 定義）
public record ApiResponse<T>(
    boolean success,
    T data,
    Object error,
    LocalDateTime timestamp
) {
    public static <T> ApiResponse<T> ok(T data) {
        return new ApiResponse<>(true, data, null, LocalDateTime.now());
    }

    public static ApiResponse<Void> error(String code, String message) {
        return new ApiResponse<>(
            false, null,
            Map.of("code", code, "message", message),
            LocalDateTime.now()
        );
    }
}

3. 分頁（Pagination）：Spring 的標準做法

Spring Data 內建分頁支援，不需要自己算 offset/limit：

// Controller
@GetMapping
public Page<Task> getAll(
    @RequestParam(defaultValue = "0") int page,
    @RequestParam(defaultValue = "20") int size,
    @RequestParam(defaultValue = "createdAt") String sort
) {
    return taskService.getAllPaged(
        PageRequest.of(page, size, Sort.by(sort).descending())
    );
}

// Service
public Page<Task> getAllPaged(Pageable pageable) {
    return taskRepo.findAll(pageable);
}

Spring 的 Page<T> 回傳 JSON 自動包含分頁 meta：

{
  "content": [ /* 20 筆 Task */ ],
  "pageable": { "page": 0, "size": 20 },
  "totalElements": 156,
  "totalPages": 8,
  "first": true,
  "last": false
}
💡 FastAPI 對照 FastAPI 的分頁通常手動處理 query params（skip/limit）或在 route 中查詢。Spring Data 的 Pageable + Page 把分頁標準化了——只要你的 Repository 繼承 JpaRepository，接收 Pageable 參數就能自動分頁。

4. 全域例外處理（@ControllerAdvice）

你在 P05 的 Controller 中手動寫了 try-catch 或 ResponseStatusException。更好的做法是將例外處理集中到一個 class：

@RestControllerAdvice  // = @ControllerAdvice + @ResponseBody
public class GlobalExceptionHandler {

    // 資源不存在
    @ExceptionHandler(NoSuchElementException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ApiResponse<Void> handleNotFound(NoSuchElementException ex) {
        return ApiResponse.error("NOT_FOUND", ex.getMessage());
    }

    // 參數驗證失敗（@Valid）
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ApiResponse<Void> handleValidation(MethodArgumentNotValidException ex) {
        var errors = ex.getBindingResult().getFieldErrors().stream()
            .map(e -> e.getField() + ": " + e.getDefaultMessage())
            .toList();
        return ApiResponse.error("VALIDATION_FAILED", String.join("; ", errors));
    }

    // 請求參數不合法
    @ExceptionHandler(IllegalArgumentException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ApiResponse<Void> handleBadRequest(IllegalArgumentException ex) {
        return ApiResponse.error("BAD_REQUEST", ex.getMessage());
    }

    // 其他未預期的例外
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ApiResponse<Void> handleUnknown(Exception ex) {
        log.error("Unexpected error", ex);
        return ApiResponse.error("INTERNAL_ERROR", "伺服器內部錯誤");
    }
}
🔥 全域例外處理的好處 ① Controller 不再有 try-catch 雜訊，只寫快樂路徑 ② 錯誤格式全 API 統一 ③ 新增例外型別只需在一個地方加 handler ④ 面試中展示這種架構意識會加分——代表你寫過正式環境的 API。

5. 請求驗證（Bean Validation）

Spring Boot 內建 Jakarta Bean Validation（@NotNull、@Size、@Email 等），和 Pydantic 的 validator 是對應概念：

// DTO 上宣告驗證規則
public record TaskRequest(
    @NotBlank(message = "標題不能為空")
    @Size(max = 200, message = "標題最多 200 字")
    String title,

    @Size(max = 50, message = "分類最多 50 字")
    String category
) { }

// Controller 中加 @Valid 觸發驗證
@PostMapping
@ResponseStatus(HttpStatus.CREATED)
public ApiResponse<Task> create(@Valid @RequestBody TaskRequest request) {
    // 如果 request 驗證失敗，MethodArgumentNotValidException 會被 @ControllerAdvice 捕捉
    return ApiResponse.ok(taskService.createTask(...));
}
驗證Spring BootFastAPI / Pydantic
必填@NotNull / @NotBlankfield: str（預設必填）
長度@Size(min=1, max=200)Field(max_length=200)
正數@PositiveField(gt=0)
Email@EmailEmailStr
自訂寫 custom validator@validator decorator

6. API 版本控制策略

實務上 API 一定會改版。三種常見做法：

// 方式 1：URL 前綴（最常見）
@RestController
@RequestMapping("/api/v1/tasks")

// 方式 2：Header（URL 乾淨，但除錯較難）
@RestController
@RequestMapping("/api/tasks")
// 客戶端帶 header: API-Version: 1

// 方式 3：Query param（最直覺但 REST 純淨派不喜歡）
// /api/tasks?version=1

面試建議：說「團隊偏好 URL 前綴，因為最透明、最易除錯」。

自我檢驗 🧠

Q1: @ControllerAdvice 解決了什麼問題？為什麼比在 Controller 中寫 try-catch 更好？

點擊展開答案

@ControllerAdvice 將所有 Controller 的例外處理集中到一個 class。好處：① Controller 只寫快樂路徑，沒有 try-catch 雜訊 ② 錯誤回應格式全 API 統一 ③ 新增例外型別只需加一個 @ExceptionHandler 方法，不需改每個 Controller ④ 符合 DRY 和單一職責原則。對比：FastAPI 中透過註冊全域 exception_handler 達到類似效果。

Q2: DELETE endpoint 成功後應該回傳什麼 HTTP 狀態碼和 body？

點擊展開答案

204 No Content，body 為空。DELETE 成功後沒有內容需要回傳（資源已刪除），204 明確告訴客戶端「請求成功，但沒有 response body」。200 OK 也可以，但 204 語意更精確。Spring Boot 中用 @ResponseStatus(HttpStatus.NO_CONTENT) 且方法回傳 void。

Q3: Spring Data 的分頁機制和手寫 LIMIT/OFFSET 相比有何優勢？

點擊展開答案

Spring Data 的 Pageable 自動處理：① 查總筆數（COUNT query）② 計算總頁數 ③ 判斷是否第一頁/最後一頁 ④ 排序。這些在手寫 LIMIT/OFFSET 時都要自己處理。Page<T> 的回傳 JSON 已包含所有分頁 meta（totalElements、totalPages、first、last），前端可以直接用這些資訊渲染分頁元件。

📌 專業 API 的標配清單

  1. 統一回應格式：ApiResponse<T> wrapper（success + data + timestamp）
  2. 全域例外處理：@ControllerAdvice 集中管理所有錯誤回應
  3. 正確的 HTTP 狀態碼：201 Created、204 No Content、400/404/500 語意正確
  4. Bean Validation：@Valid + Jakarta annotations 自動驗證請求參數
  5. Spring 分頁標準：Pageable + Page，不要手寫 offset/limit
