把「做一個關卡」變成一份可以執行的 Backlog

Lesson 0001 · 主題：Work Breakdown · 預估時間：15–20 分鐘

🎯 與你的 Mission 的關聯——這是從「想到哪做到哪」到「有節奏地推進」的第一步。沒有 breakdown，就沒有估算、沒有排程、沒有辦法讓別人參與。

問題：為什麼「做關卡」是一個糟糕的任務

你的核心框架已經完成了。剩下的工作是「做關卡設計、填充內容」。讓我猜猜你的待辦清單（如果有的話）長什麼樣子：

🗒️ 做 Level 1
🗒️ 做 Level 2
🗒️ 做 DLC 關卡

如果你看到這份清單時感受到的是一股模糊的焦慮，而不是「好，我知道下一步是什麼」——恭喜，你已經親身體驗了遊戲開發中最常見的陷阱：任務粒度過大。

一個叫做「做 Level 1」的任務，它真正的問題是：

  • 無法估算——2 天還是 2 個月？完全不知道。
  • 無法開始——坐下來之後要先做什麼？設計謎題？畫場景？寫腳本？
  • 無法追蹤進度——做到一半是什麼意思？50% 是完成了什麼？
  • 無法交接——如果之後有人幫你做關卡設計，你怎麼告訴他該做什麼？

核心概念：兩種 Board，兩種視角

單人開發者最實用的管理模式是雙板系統（Two-Board System）——讓「玩家的大腦」和「開發者的大腦」分開運作：

📋 Board 1：願景板 Vision Board

從玩家視角描述你想要的最終體驗。粒度大約是 feature 層級。

只在每個工作週期結束時才回來調整。

範例項目：
• 玩家能找到隱藏在書架後的鑰匙
• 玩家能組合道具來解開謎題
• 遊戲有兩個免費關卡 + DLC

⚙️ Board 2：執行板 Execution Board

從開發者視角寫下具體要執行的工作。任務必須小到可以估算。

每天工作時只看這塊板子。

範例項目：
• 繪製書架道具的點擊區域
• 實作鑰匙拾取後顯示在物品欄
• 撰寫書架謎題的提示文字

關鍵規則：工作時只看執行板。願景板是你「做完一個週期後」才回來檢視的東西。兩塊板子同時看的結果就是大腦同時當產品經理和工程師——然後兩邊都做不好。

Task Sizing：2 小時到 3 天的黃金法則

來自 Calgary 大學遊戲開發管理課程的經驗法則：

🏔️ 太大：超過 3 天
→
拆！
→
✅ 剛好：2h–3d
→
🧩 太小：少於 2h
→
合併

如果你的任務超過 3 天才能完成，代表它應該被繼續拆解。如果一個任務不到 2 小時就能做完，考慮把它跟相關任務合併——管理 overhead 不值得。

密室逃脫遊戲的 Breakdown 模板

對於你的密室逃脫遊戲，一個關卡可以沿著這五個維度來拆：

維度 包含什麼 範例任務
🔍 謎題鏈 謎題邏輯、觸發條件、提示系統 「實作書架鑰匙謎題：點擊正確順序→鑰匙出現」
🎨 場景素材 背景、道具美術、UI 元素 「繪製書架的三種狀態：正常／移動後／空」
💻 互動腳本 點擊回應、道具組合、過場動畫 「實作道具組合：鑰匙+鎖→門打開的過場」
🔊 音效／氛圍 背景音樂、音效回饋、環境音 「添加謎題解開時的音效回饋」
🧪 測試 通關測試、邊界情況、提示覆蓋 「測試 Level 1 完整通關流程，記錄所有卡關點」
💡 非程式工作的估算——很容易只估算「寫程式」的時間，忘了美術、音效、測試、整合也需要時間。Calgary 課程特別強調：估算時必須納入 non-code content。

動手練習

✏️ 拆解一個謎題

假設你要為密室逃脫遊戲製作這樣一個謎題：

「玩家需要在書房中找到三張散落的紙條，拼出一個四位數密碼，用這個密碼打開牆上的保險箱，取得裡面的鑰匙。」

請在下方嘗試把這個謎題拆成 4–8 個具體任務，每個任務應該落在 2 小時到 3 天的範圍內：

參考 breakdown（一種可能）：
  1. 設計紙條內容：寫出三張紙條的文字，確保拼起來能推導出密碼（2–4h）
  2. 場景配置：在書房場景中放置三張紙條的可點擊熱區（1–2h）
  3. 紙條 UI：實作點擊紙條後顯示放大檢視、已收集紙條的列表（3–6h）
  4. 保險箱互動：實作四位數字輸入面板、密碼驗證邏輯（4–8h）
  5. 保險箱美術：繪製保險箱關閉／開啟兩種狀態的素材（3–5h）
  6. 鑰匙拾取：保險箱打開後出現鑰匙、拾取後加入物品欄（1–3h）
  7. 提示系統：加入卡關提示（「或許我該找找有沒有其他紙條...」）（2–4h）
  8. 整合測試：完整跑一次這個謎題，確認所有邊界情況正常（1–2h）

小測驗：哪些任務粒度正確？

Q1. 以下哪個任務的粒度最適合放進執行板？

Q2. 「繪製 Level 1 所有場景素材」這個任務的問題是什麼？

Q3. 根據本課內容，執行板和願景板的正確使用方式是？

下一步：你的第一個 Backlog

現在輪到你自己的專案了。選擇一個關卡中的一個謎題，用上面學到的五維度模板，拆成 5–10 個具體任務。寫在紙上、Obsidian 裡、或任何你順手的工具中。

如果你還沒有管理工具，以下是最受 Godot 社群推薦的選項（從最簡單到最強大）：

  • Trello——如果你想要零摩擦開始，拖放卡片即可
  • Obsidian + Kanban 插件——如果你已經在用 Obsidian 做筆記，這是自然延伸
  • Codecks——如果你想要專為遊戲開發設計的結構（卡牌遊戲隱喻）
  • GitHub Issues / Projects——如果你想把程式碼和任務管理放在同一個地方
⚠️ 工具選擇的陷阱——不要在工具選擇上花超過 30 分鐘。用 Trello 開始，如果一個月後覺得不夠用再換。選擇工具本身不是工作。
